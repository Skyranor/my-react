# 🔵 Модуль 2: React Core & Architecture (Вопросы 21-40)

Базовые концепции React на Senior-уровне (в банках или BigTech) никогда не спрашивают "по верхам". От вас больше не ждут ответа "Пропсы нельзя менять, а Стейт можно". Проверяется то, как вы понимаете **ссылочную идентичность (Reference Equality)**, **фазы Commit/Render (React Fiber)**, **батчинг (Batching)** и **архитектурные антипаттерны**.

Все 20 ответов переписаны с акцентом на "подкапотную" физику процессов и Enterprise-опыт.

---

### ❓ 21. Что такое JSX? Что изменилось в новом JSX Transform (React v17+)?
*   **Официальный ответ:**
    > JSX — это синтаксический сахар, описывающий объекты Virtual DOM. До React 17 любой JSX компилировался в вызов функции `React.createElement`, из-за чего в каждом файле был обязателен `import React`. С 17 версии Babel/SWC автоматически подставляет `import { jsx as _jsx } from 'react/jsx-runtime'`.
*   **🗣️ Pitch (Senior):**
    > *"Новый `jsx-runtime` — это не просто эстетика и удаление импортов. Архитектурно старый `createElement` динамически парсил аргументы `children` внутри браузера клиента (Runtime). Новый компилятор статичен: он жестко формирует объект пропсов еще на этапе сборки в Webpack/Vite. Это дало нам бесплатное (Zero-Config) уменьшение бандла и микро-прирост производительности движка V8, так как он лучше оптимизирует статичные формы объектов. Плюс, это отвязало саму генерацию UI-дерева от глобального объекта React."*

### ❓ 22. Props vs State: Разница под капотом. Почему мутация State ломает React?
*   **Официальный ответ:**
    > Props (Data Down) — неизменяемые(Immutable) аргументы функции-компонента. State — локальная память компонента. React сверяет изменения стейта через алгоритм поверхностного сравнения `Object.is()`. 
*   **🗣️ Pitch (Senior):**
    > *"Классическая джуновская бага — мутировать вложенный массив стейта `state.users.push(user)` и вызвать стейт-апдейтер `setState(state)`. UI не перерисуется. React работает на парадигме Иммутабельности. Во время фазы Render, React сверяет прошлый стейт с новым через обычный референсный `===`. Если указатель (адрес в памяти компьютера) объекта остался прежним, React Fiber 'решает', что изменений нет (Bailout) и мгновенно обрывает рендер-цикл ради экономии CPU. Поэтому мы обязаны всегда создавать физически новый объект/массив в памяти через Spread `[...users, user]` или `structuredClone`."*

### ❓ 23. Controlled vs Uncontrolled Components. Что выбрать для формы из 100 полей?
*   **Официальный ответ:**
    > *   **Controlled:** Значение инпута жестко привязано к React State (`value={state}`). Любое нажатие клавиши вызывает ререндер компонента.
    > *   **Uncontrolled:** Инпут хранит значение сам в себе (нативный DOM). Мы читаем его только в момент сабмита через `useRef` или `FormData`.
*   **🗣️ Pitch (Senior):**
    > *"Если у вас в админке сложная банковская форма на 100 полей (Анкета заемщика), подход 'Controlled' убьет отзывчивость страницы. Печать одного символа будет дергать ререндер всего огромного родительского компонента 60 раз в секунду (Input Lag). Мой архитектурный стандарт для Enterprise-форм: использование библиотеки `React Hook Form`. Под капотом она строит форму на **Uncontrolled** инпутах через `refs`, собирая валидацию без ререндеров React-дерева. Controlled UI мы оставляем только для зависимых полей (например, Маска паспорта реагирует на выбор Гражданства)."*

### ❓ 24. Что такое Error Boundaries? Почему они не ловят ошибки из `try/catch` или `setTimeout`?
*   **Официальный ответ:**
    > Компоненты-предохранители. Ловят падения JavaScript во всем дочернем дереве во время фазы Рендеринга и показывают Fallback UI (например, "Что-то пошло не так"). Реализуются ТОЛЬКО через Классовые компоненты (`static getDerivedStateFromError`).
*   **🗣️ Pitch (Senior):**
    > *"Error Boundary — это защитный купол, но он ловит только ошибки жизненного цикла React (Рендер, useEffect). Если HTTP-запрос в `setTimeout` или кнопке `onClick` кидает `500 Error` — предохранитель ее не увидит (т.к. асинхронщина выпадает из Call Stack'а реакта в Event Loop браузера). Чтобы пробросить асинхронную ошибку из Catch-блока в Error Boundary, я использую микро-хак: локальный стейт `const [, setError] = useState()`. В блоке `catch(err) { setError(() => { throw err }) }`. Мутация стейта функцией с throw мгновенно триггерит рендер-фазу, и Error Boundary успешно перехватывает упавший промис, защищая Приложение от 'Белого Экрана Смерти'."*

### ❓ 25. SyntheticEvent в React: Делегирование событий (React 17 Changes).
*   **Официальный ответ:**
    > React оборачивает нативные события браузера (`Event`) в кросс-браузерный `SyntheticEvent`. В целях экономии памяти, React использует **Event Delegation** — он не вешает слушателей на каждую из 1000 ваших кнопок, а вешает 1 глобальный слушатель на родителя.
*   **🗣️ Pitch (Senior):**
    > *"До React 17 делегирование было привязано к корневому `document`. Это ломало архитектуру Микрофронтендов: если на странице 'жили' React 16 и другая версия React (или jQuery), их события перехватывались и глушили `e.stopPropagation()` друг друга на уровне документа. В React 17+ слушатели физически перевесили с `document` на `div#root` (Корневой контейнер монтирования). Это изолировало 'Вселенную' реакта от остальной веб-страницы, позволив нам безопасно рендерить несколько разных React-приложений внутри одного энтерпрайз-портала без конфликта кликов."*

### ❓ 26. `forwardRef` и паттерн Оберток (Декораторы).
*   **Официальный ответ:**
    > Функциональные компоненты не могут получать атрибут `ref` напрямую в пропсы. Обертка `React.forwardRef((props, ref) => ...)` позволяет прокинуть ссылку сквозь ваш кастомный компонент прямо к нативному внутреннему HTML-тегу.
*   **🗣️ Pitch (Senior):**
    > *"Каждая дизайн-система внутри банка начинается с `forwardRef`. Когда вы пишите кастомную `<Button>`, вы ОБЯЗАНЫ прокидывать реф внутрь нативного тега `<button ref={ref}>`. Если этого не сделать, внешние библиотеки анимаций (Framer Motion) или библиотеки тултипов (Tippy.js), которые полагаются на `getBoundingClientRect()` элемента через реф, просто сломаются и не смогут узнать физические координаты вашей кнопки на экране."*

### ❓ 27. React Portals: Выход из DOM-матрицы.
*   **Официальный ответ:**
    > `ReactDOM.createPortal(child, domNode)` позволяет отрендерить React-компонент в любой другой узел браузерного DOM (например, прямо в `<body>`), при этом сохраняя логическую привязку компонента к родительскому React-дереву (Контекст и События продолжают работать).
*   **🗣️ Pitch (Senior):**
    > *"Без Порталов разработать UI Kit невозможно. Если вы вставляете `Dropdown` меню глубоко в карточку товара, а у родительского списка стоит `overflow: hidden` или `z-index: 1`, выпадающее меню жестко 'обрежется' границами карточки. Мой стандарт: Модальные окна, Тултипы, Попапы всегда рендерятся через Portals в выделенный `div#modal-root` на самом дне `<body>`. Они физически лежат поверх вообще всего сайта, но логически я могу легко передавать в них данные и ловить `onClick` через React Context из их инициатора."*

### ❓ 28. `React.memo` (Shallow Compare) vs `useMemo`. Когда мемоизация убивает аппки?
*   **Официальный ответ:**
    > `React.memo` оборачивает компонент. Если новые Пропсы поверхностно равны старым (`prev.a === next.a`) — компонент не ререндерится. `useMemo` — кэширует результат выполнения математической функции между рендерами.
*   **🗣️ Pitch (Senior):**
    > *"Слепое оборачивание всех компонентов в `React.memo` (Premature Optimization) — это катастрофа. Shallow-сравнение старых и новых пропсов стоит процессорного времени. Если родители компонента постоянно прокидывают в него новые объекты ссылок (`<Profile user={{ id: 1 }} onClick={() => log()} />`), то `React.memo` каждую миллисекунду будет делать дорогую проверку пропсов, понимать, что объект `user` и лямбда `onClick` по ссылке изменились, и ВСЕ РАВНО делать `Render`. Мы сжигаем CPU дважды: сначала на сверке пропсов, потом на самом рендеринге. Мемоизация оправдана только на тяжелых SVG/DataGrid графиках, где пропсы железобетонно стабильны (стабилизированы через `useCallback`)."*

### ❓ 29. Зачем нужен ключ `key` в списках? Что будет при `key={index}`?
*   **Официальный ответ:**
    > Ключ помогает алгоритму React Reconciliation (Fiber) идентифицировать элементы при изменениях (добавление, удаление, сортировка массива).
*   **🗣️ Pitch (Senior):**
    > *"Вставка `key={index}` — это бомба замедленного действия. Представьте список To-Do. Первая задача — 'Купить хлеб' (индекс 0). Вы удалили её. Вторая задача 'Выгулять собаку' перескакивает наверх и получает индекс 0. React видит, что компонент с ключом '0' остался жив. Вместо того, чтобы удалить физическую DOM-ноду, React просто мутирует (переписывает) её текст. НО! Если внутри этой ноды был 'Неконтролируемый инпут' (галочка чекбокса), она останется нажатой! Состояние старой задачи просочится в новую. В Enterprise мы используем исключительно строгие Database UUID (`uuidv4()`) для списков."*

### ❓ 30. Разница между `useEffect` и `useLayoutEffect`.
*   **Официальный ответ:**
    > *   `useEffect`: Асинхронен. Запускается **ПОСЛЕ** того, как браузер физически отрисовал пиксели на мониторе (Paint phase).
    > *   `useLayoutEffect`: Синхронен. Запускается **ДО** отрисовки (между мутацией DOM и стадией Paint). Блокирует рендеринг браузера.
*   **🗣️ Pitch (Senior):**
    > *"В 99% случаев мы используем `useEffect` (API-вызовы, аналитика), чтобы не задерживать First Contentful Paint. Но если наш эффект меряет геометрию DOM (`element.getBoundingClientRect()`) и мгновенно меняет Стейт (например, авто-высота текстового поля TextArea), то визуально появится Мерцание (Flicker). Браузер успеет нарисовать старый размер, потом отработает хук, и размер прыгнет. Я меняю его на `useLayoutEffect`: браузер ставит Паузу на конвейер C++, ждет, пока React всё пересчитает, и выплевывает на экран сразу финальный кадр без морганий."*

### ❓ 31. Хуки `useCallback` + Closure Stale State (Устаревшее замыкание).
*   **Официальный ответ:**
    > `useCallback` кэширует Ссылку на функцию. Часто применяется для стабилизации пропсов, передаваемых детям внутри `React.memo`.
*   **🗣️ Pitch (Senior):**
    > *"С `useCallback` связан самый жестокий баг React - 'Stale Closures'. Джун пишет хук таймера, кэширует коллбэк `useCallback(() => console.log(counter), [])` и забывает прокинуть 'counter' в массив зависимостей (Deps array). Хук запоминает самую первую версию переменной (где counter=0) навсегда. React Linter ругается не просто так. Чтобы пробить это без ре-кэширования функции, я использую паттерн 'Latest Ref': записываю мутабельное значение в `ref.current = counter;` во время рендера и читаю `ref.current` прямо внутри кэшированной функции."*

### ❓ 32. Зачем нужен `useRef` кроме референсов на DOM?
*   **Официальный ответ:**
    > `useRef` создает изменяемый объект, значение которого `.current` сохраняется между всеми рендерами. В отличие от `useState`, мутация рефа **не вызывает** перерисовку UI.
*   **🗣️ Pitch (Senior):**
    > *"Я использую `useRef` как 'Карман' для системной памяти компонента. Худшее архитектурное решение — класть в React State переменные, которые не участвуют в отрисовке JSX (например, `setInterval ID` или Previous Data Point). Мутация технического ID-шника заставит дерево перерендериться вхолостую. Любые флаги бизнес-логики (isMounted, abortController, scrollPosition) всегда летят в Ref. Он работает с нулевой нагрузкой на React Fiber Tree."*

### ❓ 33. Фазы рендера в React 18: Render Phase vs Commit Phase.
*   **Официальный ответ:**
    > Рендеринг разбит на два этапа. **Render Phase:** React вызывает ваши функции компонентов, строит новое дерево Virtual DOM и ищет разницу (Diffing). Эта фаза прерываема (Concurrent Mode). **Commit Phase:** Детерминированная фаза. React берет найденую разницу и мутирует физический браузерный DOM (appendChild/removeChild). Блокирует UI.
*   **🗣️ Pitch (Senior):**
    > *"Понимание этой развилки отличает Синьора. Главный закон: тело функционального компонента (Render phase) обязано быть абсолютно 'Чистым' (Pure). Там запрещены мутации глобального стора или DOM. В React 18 с фичами `useTransition` движок может прерывать Render Phase посередине, отбрасывать полуготовое дерево и начинать рендер заново, если влетел высокоприоритетный клик юзера. Если вы мутируете внешнюю среду в теле функции — ваш стейт разобьется вдребезги (Tearing Effect)."*

### ❓ 34. Batching в React (Батчинг стейтов кучей). Как работает в v18?
*   **Официальный ответ:**
    > Батчинг — это группировка серии вызовов `setState` в ОДИН рендер. В React 17 батчинг работал только внутри хуков React. Запросы в `setTimeout` или `fetch.then` не батчились и давали 5 отдельных рендеров. 
*   **🗣️ Pitch (Senior):**
    > *"В React 18 внедрили Automatic Batching. Теперь не важно, где мы делаем `setItems(); setLoading(); setError();` — внутри таймаута, AJAX вызова или нативного слушателя окна. React подождет окончания Microtask Queue браузера, сольет все 3 функции и сделает 1 единственный точечный удар по DOM'у (Commit Phase). Если мне нужно насильно пробить этот батчинг (очень редко, например замер физической высоты между двумя апдейтами стейта), я использую `ReactDOM.flushSync()`. Он заставляет реакт бросить всё и обновить синхронно экран прямо здесь и сейчас."*

### ❓ 35. Контекст (`Context API`) vs `Redux`/`Zustand`. Проблема излишних рендеров.
*   **Официальный ответ:**
    > Context API предназначен для прокидывания глобальных данных (Тема, Язык), решая ловушку `Prop Drilling`.
*   **🗣️ Pitch (Senior):**
    > *"Использовать кастомный Context API (`useReducer` + `Provider`) в качестве глобального менеджера состояний для Enterprise UI (Магазина) — грубейшая ошибка архитектуры. Когда объект в `value={<Provider value={{ cart, user }} />}` меняется (юзер сменил имя), React жестко и бескомпромиссно заставит перерендериться АБСОЛЮТНО ВСЕХ подписчиков (Consumers) хука `useContext` в дереве, даже если компонент 'Корзина' ждет только поле `cart`. Библиотеки типа Redux или Zustand строятся на паттерне Pub/Sub и подписках 'На селекторы'. Они дергают микро-рендер строго в тех нодах, данные которых изменились (O(1) updates). Контекст оставляем только для Theme/Auth, которые редко мутируют."*

### ❓ 36. Hydration Mismatch: Боль Server Side Rendering (SSR/Next.js).
*   **Официальный ответ:**
    > Окно консоли ругается: 'Text content did not match'. Это происходит, когда HTML, срендеренный в строке на сервере (Node.js), не совпадает с первым виртуальным деревом, которое React строит в браузере.
*   **🗣️ Pitch (Senior):**
    > *"Самый страшный баг гидратации — это использование `new Date().toLocaleString()` или `window.innerWidth` прямо в теле компонента. Сервер Node.js срендерит время сервера США, а клиентский React срендерит часовой пояс телефона Москвы. Деревья столкнутся лбами. В React 18 это приведет к полному сбрасыванию (Unmount/Fallback) ветки компонентов до уровня `<Suspense>`. Чтобы это вылечить архитектурно, мы инкапсулируем зависящий от бразуера рендер в двухфазный проход: `const [isMounted, setMounted] = useState(false); useEffect(() => setMounted(true),[])`. Рендерим пустоту, пока не сработает первый useEffect-цикл на клиенте."*

### ❓ 37. React Fragments (`<> ... </>`). Зачем они нужны?
*   **Официальный ответ:**
    > Позволяют группировать список дочерних JSX-элементов без добавления в реальный DOM лишних мусорных контейнеров (например `<div className="wrapper">`).
*   **🗣️ Pitch (Senior):**
    > *"Фрагменты спасают нас от болезни 'div-soup'. Если мы рендерим сложные CSS Grid или Flexbox 'подсетки', любая лишняя HTML-оболочка (wrapper) гарантированно уничтожит (сломает) архитектуру колонок Grid. Мы используем пустые '<>' везде, чтобы не ломать верстку. Единственный нюанс: на короткую запись `<>` нельзя повесить `key`. Если я мапплю массив фрагментов, я вынужден писать полный синтаксис `<React.Fragment key={id}>`."*

### ❓ 38. Strict Mode (`<React.StrictMode>`). Почему компоненты рендерятся дважды?
*   **Официальный ответ:**
    > Инструмент диагностики. В режиме разработчика (Только в DEV среде!) он специально запускает Mount -> Unmount -> Mount цикл (и вызывает функцию рендера два раза подряд) компонента.
*   **🗣️ Pitch (Senior):**
    > *"Junior-разработчики в панике бегут удалять Strict Mode из Next.js, думая, что он 'Убьёт перфоманс в продакшене'. Это колоссальная ошибка. React вызывает вашу функцию два раза специально, чтобы проверить её 'Пуризм' (Отсутствие мутаций внешнего мира). Попытка `const data = []; function App() { data.push(1) }` в Strict Mode даст в дев-среде `[1, 1]`, мгновенно 'дав по рукам' разработчику: Обнаружен side-effect! Также он проверяет корректность очистки ваших подписок в 'Cleanup' блоке `useEffect`. В продакшен-сборку это двойное поведение физически не компилируется."*

### ❓ 39. HOC'и (High-Order Components) vs Хуки. HOC мертв?
*   **Официальный ответ:**
    > HOC — функция, которая брала Компонент в аргумент и возвращала новый Умный Компонент (паттерн Обертка/Декоратор). В 90% случаев их заменили Custom Hooks (Кастомные хуки).
*   **🗣️ Pitch (Senior):**
    > *"Раньше в Redux мы писали спагетти-оболочки `connect(withRouter(withAuth(MyComponent)))`. Это создавало гротескный вложенный ад (Wrapper Hell) в React DevTools (десятки пустых нод в инспекторе). Хуки убили HOC'и для Data-fetching логики. Вы просто вызываете `useAuth()` прямо в теле. Но HOC-паттерн до сих пор незаменим там, где необходимо 'Перехватить' (Intercept) стадию рендеринга самого компонента извне. Например обертка `React.memo(Component)` или `withSentryGuard(Component)`. Выкидывать HOC'и рано, они переместились с уровня логики на Инфраструктурный уровень (Обертки лейаутов, Error Catching)."*

### ❓ 40. Функция очистки в `useEffect` (Cleanup function). Memory Leaks.
*   **Официальный ответ:**
    > Хук может вернуть функцию очистки. React запустит её ДВЕЖДЫ: 1) ПЕРЕД самым запуском следующего рендер-хука (если изменились зависимости). 2) В момент Смерти компонента (Unmount).
*   **🗣️ Pitch (Senior):**
*   **🗣️ Pitch (Senior):**
    > *"Если в банковском Дашборде график запускает подписку на WebSocket (Tickers) и юзер переключает табы — без Cleanup мы получим 10 плодящихся коннекшенов(Connection Leak), забивших RAM браузера и убивших бэкенд. Вторая проблема — 'Race Conditions' (Гонка запросов) в поиске. Если `fetch(query1)` долгий, а `fetch(query2)` быстрый — первый перепишет Стейт экрана. Мой личный стандарт: любой `fetch` внутри эффекта обязан пробрасывать нативный браузерный `AbortController`. В функции очистки мы делаем `controller.abort()`. При каждом изменении инпута или уничтожении компонента старый тяжелый Request физически прервется в Network Tab (отменив мутацию UI)."*

## 🟡 РАЗДЕЛ 2.1: База (Middle Level Core)

### ❓ 41. Что такое Virtual DOM? Как он оптимизирует рендеринг?
*   **Официальный ответ:**
    > Разумно легковесная копия/слепок настоящего DOM-дерева в оперативной памяти браузера. Мутация реального DOM чудовищно медленна. VDOM позволяет изменять UI "в уме", находить разницу (Diffing) и накатываться на реальный DOM одним точечным апдейтом.
*   **🗣️ Pitch (Middle):**
    > *"React решает задачу 'Не трогать лишнее'. Он строит старое Дерево и новое Дерево. Путём эвристического алгоритма O(N) он сравнивает их по узлам и ключам. То, что изменилось, патчится в Нативный Браузерный DOM. Мы получаем декларативный подход (описываем конечный вид) вместо мучительного иммутабельного создания `document.createElement()`."*

### ❓ 42. Unidirectional Data Flow. Зачем нужен подъем стейта (Lifting State Up)?
*   **Официальный ответ:**
    > Данные в React текут строго сверху вниз (от родителя к ребенку) через `props`. Стейт должен жить на минимально общем уровне для всех компонентов, которым он нужен. Способ ребенку поменять родителя — вызов прокинутой callback-функции `onUpdate`.
*   **🗣️ Pitch (Middle):**
    > *"Один из главных архитектурных принципов. Если у нас есть Инпут ввода и Таблица, которая фильтруется по слову из инпута — стейт `{ query }` не может жить внутри Инпута. Они братья. Стейт 'поднимается' в их общего Родителя (Container), он отдает `query` в Таблицу, а `setQuery` функцию прокидывает в Инпут."*

### ❓ 43. Prop Drilling (Сверление пропсов). Как его решать?
*   **Официальный ответ:**
    > Когда мы прокидываем один и тот же `prop` через 5 уровней вложенности транзитных компонентов, которым он сам не нужен, только ради доставки 'на самое дно'.
*   **🗣️ Pitch (Middle):**
    > *"Проп дриллинг убивает читаемость кода. Я решаю это тремя путями: 1) Composition (Композиция) — передача `<Wrapper>{children}</Wrapper>`, чтобы прокинуть уже готовый срендеренный компонент. 2) Использую `Context API` для глобальных данных вроде Темы. 3) Применяю State Manager-ы (Zustand/Redux), вытаскивая данные из единого Store прямо на нужном уровне."*

### ❓ 44. Жизненный цикл классового компонента (аналоги в Хуках).
*   **Официальный ответ:**
    > *   `componentDidMount` -> `useEffect(fn, [])`.
    > *   `componentDidUpdate` -> `useEffect(fn, [deps])`.
    > *   `componentWillUnmount` -> Функция очистки (return/cleanup) в `useEffect`.
*   **🗣️ Pitch (Middle):**
    > *"Переход на хуки стер жестко заданный жизненный цикл. Мы перестали думать в терминах 'Mounting / Updating', мы мыслим механизмом 'Синхронизации' логики со Стейтом. Эффект запускается тогда, когда сменилась хоть одна зависимость из его массива, а не потому что компонент обновился целиком."*

### ❓ 45. Синтетические события (SyntheticEvent) React. Зачем?
*   **Официальный ответ:**
    > Кроссбраузерная обертка над нативным Event'ом. Нивелирует разницу между версиями IE/Chrome/Safari.
*   **🗣️ Pitch (Middle):**
    > *"React использует Event Delegation. Клик по любой Button не вешается как `button.addEventListener`. Слушатель висит глобально на `#root`. Разница от нативных событий еще и в Пулинге (до v17) — раньше реактовский объект события полностью 'опустошался' (null) после вызова коллбека для экономии памяти, и отложенный `setTimeout` не мог прочитать `e.target`."*
