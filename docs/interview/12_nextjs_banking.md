# 🏦 Модуль 12: Next.js в Финтехе — BFF, Безопасность и Дизайн Больших Систем

В банковской сфере **Next.js** (особенно **App Router**) используется не просто как «рендерилка страниц», а как **BFF (Backend For Frontend)** и уровень оркестрации микросервисов. 

Если вы расскажете об этих паттернах на собеседовании, интервьюер поймет, что вы мыслите на уровне **финансовой безопасности и производительности**.

---

## 🏗️ 1. Паттерн BFF (Backend For Frontend)

### ❓ В: "Зачем банку нужен Next.js как BFF между фронтом и микросервисами?"
*   **Официальный подход:**
    На клиенте API-запросы напрямую к 5 микросервисам создают огромный оверхед (Latency) и уязвимы. Next.js агрегирует эти запросы на сервере Node.js.
*   **🗣️ Pitch:**
    > *"В финтехе мы используем Серверный уровень Next.js (Server Components / API Routes) как **BFF-слой**. 
    > **Почему это безопасно:** Секретные API-ключи банковских микросервисов хранятся строго в `.env` на сервере и **никогда не утекают в браузер**. 
    > **Почему это быстро:** Next.js делает запросы к бэкенду внутри внутренней защищенной сети банка (Intranet), где задержки минимальны, и отдает клиенту один легкий готовый JSON или HTML."*

## 🔒 2. Middleware Security & Auth

### ❓ В: "Как организовать проверку сессии (Auth) в Next.js без лагов при переходе между страницами?"
*   **Официальный подход:** Использование `Middleware.js` на границе Edge-сервера.
*   **🗣️ Pitch:**
    > *"Для критических страниц (например, «Перевод денег») недопустимо даже на секунду показать закрытый контент. Я использую файл `middleware.js`. 
    > Он перехватывает запрос **до** того, как Next.js начнет рендерить страницу. Middleware проверяет наличие HttpOnly JWT-токена. Если кука протухла — мы делаем `NextResponse.redirect('/login')` моментально. Это защищает от утечек данных при дефолтной гидратации."*

---

## 🌊 3. Streaming & Suspense (Предохранение от зависания)

### ❓ В: "Что делать, если один из банковских микросервисов (например, Кредиты) отвечает 5 секунд? Страница зависнет?"
*   **Официальный подход:** Использование **Streaming (Потоковый рендер)** через `<Suspense>` в App Router.
*   **🗣️ Pitch:**
    > *"Нет, страница не зависнет. Благодаря **Streaming (серверная отправка чанками)**, Next.js моментально отдаст браузеру Шапку и Скелетон страницы. 
    > Тяжелый компонент `<CreditsList />` мы оборачиваем в `<Suspense fallback={<Skeleton />}>`. Как только бэкенд отдаст данные через 5 секунд, Next.js «дошлет» этот кусок HTML по тому же HTTP-каналу асинхронно. Пользователь видит живой интерфейс, а не зависший экран."*

---

## 📦 4. Server Components (RSC) vs Client Components

### ❓ В: "В банке всё динамическое. Зачем вообще нужны Server Components (RSC), которые нельзя изменять стейтом?"
*   **Официальный подход:** RSC убирают JS-бандл для статических или readonly частей.
*   **🗣️ Pitch:**
    > *"Даже в банке есть тяжелый статический контент: Публичные Оферты, Условия Банковского Обслуживания, Футеры, Меню. 
    > Рендеринг их как **Server Components** позволяет не скачивать парсеры Markdown или тяжелые библиотеки форматирования текста в браузер пользователя. Они рендерятся в байты на сервере. Это кардинально уменьшает **Bundle Size** приложения на мобильных устройствах."*

---

## 🌐 5. Масштаб: Архитектура Multi-Zones

### ❓ В: "Как 3 разные команды могут одновременно разрабатывать 3 независимых Next.js приложения для одного домена банка?"
*   **Официальный подход:** Паттерн **Multi-Zones** (объединение приложений через конфигурацию реверс-прокси или `next.config.js`).
*   **🗣️ Pitch:**
    > *"Когда банк большой, пилить один монолитный Next.js — это боль сборки. Мы используем **Multi-Zones**. 
    > Под капотом приложение `A` отвечает за `/cards`, приложение `B` — за `/credits`. Маршрутизация настраивается на уровне `next.config.js` (`rewrites`) или через внешку nignx / AWS CloudFront. Каждая команда деплоит свой Next.js независимо, сохраняя единый интерфейс для клиента."*

---

### 🔥 Убойный аргумент для Интервьюера (Про Next.js 14+):
Упомяните использование **Server Actions** не для отправки форм, а для **Mutation Security**:
> *"С появлением `Server Actions` мутация данных запечатана декларативно. Мы вызываем функцию из JS, а Next.js сам делает POST запрос. В финтехе это отлично мапится на безопасность, так как под капотом защищено CSRF-токенами из коробки."*
