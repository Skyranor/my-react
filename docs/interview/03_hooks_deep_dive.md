# 🟢 Модуль 3: React Hooks Deep Dive (Вопросы 41-60)

Этот блок проверяет глубинное понимание того, как хуки живут внутри React Fiber. Инженеров Senior-уровня не спрашивают "как создать стейт". Вас спросят про архитектурные боли: **Утечки памяти (Memory Leaks)**, **Tearing в Concurrent Mode**, **Stale Closures** и устройство хуков под капотом (Связные списки).

Ответы переформатированы так, чтобы продемонстрировать ваш опыт проектирования масштабируемых, высокопроизводительных приложений без морганий интерфейса (Flicker-free UI).

---

### ❓ 41. Как устроен `useState` под капотом? Почему у хуков нет Имен/ID?
*   **Официальный ответ:**
    > Хуки не являются магией. Под капотом стейты компонента хранятся в виде **Связного списка (Linked List)** в свойстве `memoizedState` вашей Fiber-ноды. React находит нужный стейт ИСКЛЮЧИТЕЛЬНО по порядку (индексу) вызова хука сверху вниз в теле функции.
*   **🗣️ Pitch (Senior):**
    > *"Когда вы делаете 3 вызова `useState`, React строит список `Hook1 -> Hook2 -> Hook3`. При следующем рендере (апдейте), хуки считывают свои значения строго по этому указателю в памяти. Именно поэтому у хуков нет своих уникальных ID (ключей), и именно отсюда растет фундаментальное правило 'Rules of Hooks': нельзя вызывать хуки в циклах или `if`-условиях, иначе указатель связного списка безвозвратно сместится и сломает всю привязку данных к переменным."*

### ❓ 42. Stale Closures (Устаревшие замыкания). Почему `setInterval` в хуках ломается?
*   **Официальный ответ:**
    > JavaScript замыкает (сохраняет) лексическое окружение в момент создания функции. Если `useEffect` с `setInterval` не имеет нужных зависимостей, он "навсегда" запомнит самый первый Scope функции, где `count === 0`.
*   **🗣️ Pitch (Senior):**
    > *"Stale Closures — это самая частая причина багов с таймерами и Event Listeners в React. Если я пытаюсь прочитать стейт внутри `window.addEventListener` внутри `useEffect([])`, я всегда получу начальное состояние из первого рендера. У меня два архитектурных пути: 1) Добавить стейт в Dependency Array, но тогда `useEffect` будет пересоздавать Listeners на каждый чих (убивая CPU). 2) Использовать 'Latest Ref Паттерн'. Я синхронизирую стейт в реф (`ref.current = state`) на стадии рендера, и читаю мутабельный `ref.current` из хука. Замыкание пробито без ре-рендеров."*

### ❓ 43. Memory Leaks в `useEffect`. Как `AbortController` спасает приложение?
*   **Официальный ответ:**
    > Функция `cleanup` (возвращаемая из хука) запускается перед демонтированием (Unmount) компонента или перед следующим запуском этого же эффекта. Забытая очистка порождает Memory Leak.
*   **🗣️ Pitch (Senior):**
    > *"Гонка сетевых запросов (Race Conditions) — классика сложных Web-дашбордов. Пользователь быстро кликает на 'Таб 1', потом на 'Таб 2'. Улетает два `fetch`. Ответ 'Таба 1' из-за лагов сети приходит ПОЗЖЕ и безвозвратно затирает стейт экрана (мы на втором табе, а видим данные первого). Мой железобетонный стандарт: ВСЕ асинхронные вызовы в эффектах обязаны прокидывать `new AbortController().signal`. В функции `return () => controller.abort()` мы 물리чески убиваем старый HTTP-запрос в Network-клиенте браузера при переходе на следующий таб."*

### ❓ 44. Разница фаз: `useEffect` vs `useLayoutEffect`. Когда использовать второй?
*   **Официальный ответ:**
    > *   `useEffect`: Асинхронен. Запускается **после** фазы Paint (браузер уже нарисовал пиксели на экране).
    > *   `useLayoutEffect`: Синхронен. Срабатывает **после** мутации DOM, но **до** фазы Paint (блокирует отрисовку экрана браузером).
*   **🗣️ Pitch (Senior):**
    > *"В 95% случаев мы ставим `useEffect`, чтобы не блокировать UI нативным кодом. НО, если я строю UI-библиотеку (например Тултип), и мне нужно математически сдвинуть его влево, если он упирается в край экрана. В `useEffect` пользователь сначала увидит тултип справа (за краем), а через миллисекунду тултип 'Прыгнет' (Flicker) влево. `useLayoutEffect` решает эту фундаментальную боль Layout Thrashing: мы замеряем DOM и меняем стейт 'под капотом'. Браузер подождет нас, и выплеснет на экран сразу идеальный финальный кадр без морганий."*

### ❓ 45. `useMemo` vs `useCallback`. Паттерны злоупотребления (Premature Optimization).
*   **Официальный ответ:**
    > `useMemo` кэширует значение (любого типа). `useCallback` кэширует саму функцию (синтаксический сахар над `useMemo(() => fn)`).
*   **🗣️ Pitch (Senior):**
    > *"Многие джуны оборачивают КАЖДУЮ функцию в `useCallback`, думая, что делают React быстрее. В реальности они делают его в 2 раза медленнее. Кэширование использует CPU (сравнение объектов в массиве `deps`) и нехило забивает RAM движка V8. `useCallback` имеет смысл архитектурно ТОЛЬКО в двух случаях: 1) Вы передаете эту лямбду как пропс глубоко вниз в компонент, обернутый в `React.memo()`. Без `useCallback` ссылка (`ref`) на функцию каждый рендер будет новой, и `memo` у детей будет вхолостую срываться. 2) Функция физически передается в Dependency Array чужого `useEffect`."*

### ❓ 46. Ловушка `deps` массива (Dependency Array). Сравнение объектов.
*   **Официальный ответ:**
    > React сравнивает элементы массива `deps` алгоритмом нестрогого равенства `Object.is` (Reference Equality / поверхностное сравнение).
*   **🗣️ Pitch (Senior):**
    > *"Если вы прокинете инлайновый объект `useMemo(fn, [{ id: 1 }])`, ваш хук будет перезапускаться вхолостую на КАЖДОМ рендере. Ссылка на объект `{id: 1}` создается заново каждый вызов функции. Чтобы сломать эту боль глубокого кэширования конфигураций, нам часто приходится использовать кастомные хуки типа `useDeepCompareEffect` (где под капотом `lodash.isEqual` и `useRef` для сохранения 'стабилизированной' ссылки) или вообще спускаться на примитивы (string/number) в `deps`: `[config.id, config.name]`."*

### ❓ 47. State Machine. Почему `useReducer` безопаснее `useState` для сложной логики?
*   **Официальный ответ:**
    > `useReducer` централизует мутации (Action -> Reducer -> New State). На самом деле, `useState` внутри сурсов React'а — это просто урезанная обертка над `useReducer`.
*   **🗣️ Pitch (Senior):**
    > *"Когда бизнес-логика компонента разрастается: юзер нажимает кнопку, мы должны сделать `setLoading(true)`, `setError(null)`, а потом `setData(res)` и `setLoading(false)`. Разрозненные `useState` порождают 'Гонку состояний' и неконсистентность UI. `useReducer` привносит паттерн State Machine (Конечный Автомат). Компонент перестает решать 'Как изменить данные', он становится глупым и просто шлет Намерения: `dispatch({ type: 'FETCH_SUCCESS' })`. А Редьюсер математически строго и предсказуемо меняет 3 флага разом."*

### ❓ 48. Ловушка `useContext`. Архитектурный ад излишних рендеров (Render Hell).
*   **Официальный ответ:**
    > Хук `useContext` подписывает Fiber-узел на Провайдер. Любое изменение `value={obj}` у Провайдера мгновенно заставляет все компоненты-подписчики (Consumers) выполнить Render-фазу, игнорируя любые `React.memo`.
*   **🗣️ Pitch (Senior):**
    > *"Использовать нативный React Context как глобальный State Manager (`Redux`-заменитель) — это смерть перфоманса. Если в корне приложения висит `Provider value={{ user, cart, theme }}`, и мы меняем название темы (Theme), то `Header` (который читает Theme) и `CartList` (который читает Cart, но подписан на этот же гигантский провайдер) рендерятся одновременно!! Мой архитектурный паттерн с Контекстом: 1) Дробление (Split) контекстов: `ThemeContext`, `AuthContext` раздельно. 2) Базирование на Библиотеках (Zustand), которые под капотом используют `useSyncExternalStore` и точечную 'Подписку на Селекторы', обновляя ТОЛЬКО те ноды, которым физически нужны новые данные."*

### ❓ 49. Ленивая инициализация в `useState`. Зачем?
*   **Официальный ответ:**
    > Если передать функцию `useState(() => getHeavyData())`, React вызовет ее строго ОДИН раз при фазе Mount. В отличие от `useState(getHeavyData())`, которая будет "холостить" вычисления на каждом рендере.
*   **🗣️ Pitch (Senior):**
    > *"Синтаксическая мелочь, которая выкачивает RAM и CPU в тяжелых компонентах. Если `getHeavyData()` читает и парсит Мегабайт из `localStorage` (синхронное I/O событие `JSON.parse`), то при каждом нажатии клавиши в инпуте вашего компонента, React будет заново парсить Мегабайт JSON из диска компьютера в оперативу! И мгновенно выбрасывать его (так как Initial State берется только единожды). Ленивое коллбэк-объявление `() =>` уберегает от этого системного overhead'а навсегда."*

### ❓ 50. Отличие Глобальной Вне-компонентной переменной `let timer` от `useRef`.
*   **Официальный ответ:**
    > Глобальная `let timer` делит (Share) одно и то же значение между ВСЕМИ запущенными инстансами данного React-компонента на странице. А `useRef` физически изолирован (Scoped) внутри конкретной Fiber-ноды.
*   **🗣️ Pitch (Senior):**
    > *"Рядовая катастрофа миграции с Vanilla JS: разработчик вынес `let clickId = 0` за пределы компонента `export const Card = () => {}`. Пока эта Карточка одна на экране — багов нет. Но как только мы маппим список из 10 Карточек, они все начинают 'гадить' в одну и ту же общую оперативную память (`closure` модуля Webpack), перетирая таймеры друг друга! `useRef` — это архитектурно безопасный карман, привязанный к Жизненному Циклу (Instance) строго ЕДИНСТВЕННОЙ ноды в Virtual DOM."*

## 🔥 React 18: Архитектура Конкурентного Рендеринга (Concurrent Features)

### ❓ 51. Concurrent Mode: `useTransition` и отмена рендеринга.
*   **Официальный ответ:**
    > Хук помечает апдейт стейта как "Вторичный/Низкоприоритетный" (Transition). React 18 получает право поставить рендеринг тяжелого дерева на "Паузу" (Yielding to Main Thread) в угоду срочным событиям (Ввод текста).
*   **🗣️ Pitch (Senior):**
    > *"До React 18, если юзер вбивал 'A' в Инпут Поиска, и таблица из 10к строк начинала рендер(фильтрацию), инпут намертво зависал. Движок JS блокировал ввод следующих букв 'p', 'p', 'l', 'e', пока таблица не закончит Sync Rendering. Обернув апдейт таблицы в `startTransition(() => setTableData(res))`, я приказываю React: 'Отрисовка таблицы вторична'. Как только юзер нажимает 'p', Реакт выбрасывает наполовину готовую таблицу в мусор (`DISCARD`), мгновенно рендерит букву 'p' в инпут (Urgent Update) и начинает рендер таблицы с нуля в фоне. UI перестает фризить."*

### ❓ 52. `useDeferredValue` vs Debounce/Throttle.
*   **Официальный ответ:**
    > `useDeferredValue(text)` создает "Задержанную" копию переменной. Если `text` меняется быстро, React не прерывает работу, чтобы обновить дочерние элементы с этой переменной.
*   **🗣️ Pitch (Senior):**
    > *"На первый взгляд это похоже на `useDebounce`, но `Debounce` жестко зашит на таймауты (`300ms`), что искусственно тормозит юзеров с мощными Макбуками (M3). `useDeferredValue` интегрирован прямо в C++ ядро Браузера и React Fiber Tree. Если компьютер быстрый, React обновит 'Задержанное' значение за 5мс. Если это старый Android — он сбросит лишние кадры (Drop Frames) и обновит значение только когда процессор реально освободится. Оптимизация динамически адаптируется под железо клиента."*

### ❓ 53. Tearing (Разрывы Стейта) и спасительный `useSyncExternalStore`.
*   **Официальный ответ:**
    > В Concurrent Mode фаза Render может быть прервана и растянута во времени. Если в этот момент внешнее хранилище (Redux) обновится, часть дерева отрендерится с 'Синим' цветом, а часть с 'Красным'. Это называется Tearing.
*   **🗣️ Pitch (Senior):**
    > *"Лютеший страх создателей Zustand/Redux при переходе на React 18. Так как React больше не делает рендеры синхронными, чтение внешнего стейта напрямую может дать 'разорванный' (неконсистентный) кадр. Хук `useSyncExternalStore(subscribe, getSnapshot)` был добавлен специально для авторов Store-библиотек. Он жестко гарантирует (Force Sync), что пока React дерево рендерится, оно будет использовать один и тот же 'Железобетонный слепок' внешних данных, защищаясь от Tearing."*

### ❓ 54. Хук `useInsertionEffect`. Спец-API для CSS-in-JS.
*   **Официальный ответ:**
    > Исполняется РАНЬШЕ всех. До мутации DOM и до `useLayoutEffect`. Не имеет доступа к `refs`.
*   **🗣️ Pitch (Senior):**
    > *"Этот хук вообще не нужен продуктовым программистам. React Core Team добавила его исключительно для разработчиков библиотек `Styled-Components` и `Emotion`. CSS-in-JS либам критически важно вставить сгенерированные стили `<style>` напрямую в `<head>` документа строго ДО того, как `useLayoutEffect` внутри компонентов потребителей начнет вычислять физические размеры элементов (через `getBoundingClientRect()`), иначе высота будет замерена до применения стилей. Узкоспециализированная архитектурная вставка."*

### ❓ 55. Различия `useId` от `Math.random()` или `uuid`.
*   **Официальный ответ:**
    > Генерирует стабильный уникальный ID (например `:r1:`), который одинаково вычисляется как при Server-Side Rendering (Node.js), так и при Hydration (Браузер).
*   **🗣️ Pitch (Senior):**
    > *"Генерация ID для HTML аттрибутов `aria-labelledby` или `<label htmlFor>` — это база Accessibility. Если мы используем `uuid()` внутри компонента, сервер срендерит HTML `<input id="123">`. Клиентский JS при попытке активации срендерит `<input id="456">`. React сходит с ума, крича в консоль `Hydration Mismatch Error: UI doesn't match Server`. Приложение сбросит виртуальное дерево и перерендерит страницу, убивая FCP и SEO. Архитектура `useId` выстраивает ID на основе топологии (позиции) компонента в дереве, что обеспечивает их идеальное 100% математическое совпадение на бэке и фронте."*

### ❓ 56. Custom Hooks (Кастомные хуки). Передают ли хуки стейт?
*   **Официальный ответ:**
    > Кастомный хук (`useAuth`) НЕ разделяет свое состояние между разными компонентами. Он шарит (Share) только логику (Stateful Logic). У каждого компонента, вызвавшего хук, формируется свой абсолютно независимый кусок `memoizedState` в Fiber дереве.
*   **🗣️ Pitch (Senior):**
    > *"Главное заблуждение: 'Давайте напишем хук `useCart()`, чтобы делиться корзиной'. Это не сработает. Два компонента Header и Sidebar, вызвав `useCart`, получат ДВЕ разные пустые корзины. Кастомные хуки нужны для композиции и сокрытия сложности (Separation of Concerns). Если вам нужно ШАРИТЬ именно Стейт между инстансами, внутри абстракции кастомного хука мы всё равно обязаны использовать Context API, Redux подписку (`useSyncExternalStore`), или глобальный Zustand Store."*

### ❓ 57. В React 18 `useEffect` вызывается дважды при Mount (Strict Mode). Зачем эта боль?
*   **Официальный ответ:**
    > В DEV режиме React 18 искусственно прогоняет цикл `Mount -> Unmount -> Mount`. Это сделано специально, чтобы выловить баги неубитых эффектов (Memory Leaks).
*   **🗣️ Pitch (Senior):**
    > *"React-команда готовит нас к фичам 'Offscreen / Suspense'. В будущем React сможет убирать скрытые табы ('About') из DOM в оперативную память (Keep-Alive), а потом возвращать их без ре-рендеринга. Если вы забыли написать логику отписки `socket.off('message')` внутри Cleanup-функции `useEffect`, двойной Mount в Strict Mode немедленно удвоит ваши WebSocket сессии в дев-запуске. Это жестоко, но это моментально заставляет инженеров писать Идеально чистые, само-уничтожающиеся эффекты, готовые к Архитектуре Enterprise-масштаба."*

### ❓ 58. Separation of Concerns в useEffect. Почему нельзя все пихать в один Effect?
*   **Официальный ответ:**
    > Хуки должны разделяться по логическому смыслу, а не по жизненному циклу компонента.
*   **🗣️ Pitch (Senior):**
    > *"В мире Классов (Class Components) у нас был метод-мусорка `componentDidUpdate`, куда инженеры скидывали и проверку ID продукта, и логику скролла, и аналитику. Один гигантский `if/else` ад. Хуки позволяют дробить логику на микро-скоупы: `useEffect(fetchData, [id])` и `useEffect(logScroll, [scrollY])`. Это позволяет каждому эффекту синхронизироваться ИСКЛЮЧИТЕЛЬНО по своему массиву зависимостей(deps), и изолированно возвращать собственный Garbage Collection (Очистку). Это радикально упрощает DX и тестирование."*

### ❓ 59. Event Hander vs `useEffect` (Проблема синхронизации).
*   **Официальный ответ:**
    > Данные должны обновляться ИЗ обработчика событий (`onClick`), а не ИЗ реакции на изменение стейта внутри Эффекта.
*   **🗣️ Pitch (Senior):**
    > *"В официальной доке React.dev есть четкий закон: Эффекты — это способ синхронизировать React-дерево с вне-реактовскими системами (API, DOM, WebSockets). Эффекты НЕ созданы для реагирования на действия пользователя! Паттерн: юзер кликнул -> `setPage(page+1)` -> `useEffect` ловит `[page]` и делает `fetch()` — это 'Водопадный' антипаттерн (Waterfall), ломающий Suspense архитектуру. Вызывайте `fetch` СИНХРОННО прямо внутри `handleClick`, тогда данные начнут лететь мгновенно, а не через 1 микротасочный цикл (Microtask) ожидания перерендера React-дерева."*

### ❓ 60. `useEffectEvent` (RFC хук). Зачем он разрабатывается?
*   **Официальный ответ:**
    > Новый хук (экспериментальный) изолирует "не-реактивные" переменные от "реактивных" внутри `useEffect`, избавляя от глупого спама лишних линтер-варнингов.
*   **🗣️ Pitch (Senior):**
    > *"Часто в Дашборде мы слушаем вебсокет, и при новом сообщении шлем пуш-уведомление `showToast(msg, theme)`. Хук `useEffect` требует положить `theme` внутрь deps-массива. НО это ломает логику: если юзер сменит тему (Dark/Light), сработает `useEffect` и 'Левое' (старое) сообщение выскочит заново! Мы НЕ хотим реагировать на `theme`, нам нужна только ее 'Последняя' версия для цвета пуша. Использование `useEffectEvent` кэширует код-логику так, что мы можем вызывать его внутри эффекта 'молча', без необходимости просовывать ломающие зависимости в `deps`."*

## 🟡 РАЗДЕЛ 3.1: База Хуков (Middle Level Core)

### ❓ 61. Правила Хуков (Rules of Hooks).
*   **Официальный ответ:**
    > 1. Вызывать только на верхнем уровне (Top level). Никаких циклов, if-блоков или вложенных функций. 
    > 2. Вызывать только из React-компонентов или из кастомных хуков.
*   **🗣️ Pitch (Middle):**
    > *"Эти строгие правила — следствие того, как хуки живут под капотом. Индекс в массиве Fiber Tree — это единственный способ для хука восстановить свой стейт между рендерами. Если хук спрятать в `if()`, порядок собьется, что приведет к фатальной ошибке чтения переменных. Помогают держать код чистым правила линтера `eslint-plugin-react-hooks`."*

### ❓ 62. Массив зависимостей (Dependency Array). Что будет, если не передать?
*   **Официальный ответ:**
    > *   `useEffect(fn)` : Эффект будет запускаться после **каждого** рендера компонента.
    > *   `useEffect(fn, [])`: Эффект запустится ровно 1 раз (эквивалент Mount фазы).
    > *   `useEffect(fn, [id])`: Эффект пере-запустится только если переменная `id` поменяла значение (`Object.is`).
*   **🗣️ Pitch (Middle):**
    > *"Не передать массив зависимостей = бесконечный цикл (Infinite Loop), если внутри эффекта вызвать `setState`. Компонент рендерится -> эффект вызывается -> дергает стейт -> компонент рендерится -> цикл."*

### ❓ 63. Кастомные हुки (Custom Hooks). Зачем они нужны?
*   **Официальный ответ:**
    > JS функция, чье имя начинается с "use", внутри которой инкапсулируется логика других хуков (`useState`, `useEffect`).
*   **🗣️ Pitch (Middle):**
    > *"Это способ вынести 'Умное' поведение из Компонента. Если мне нужно слушать ширину экрана, я не буду копировать `useEffect(window.addEventListener)` в 10 Button-компонентов. Я напишу `export const useWindowWidth = ()`. Компонент просто вызовет `const width = useWindowWidth();` и станет в десятки раз чище. Главное помнить: стейт внутри хука не шарится между компонентами, у каждого свой клонированный изолированный кусок памяти."*

### ❓ 64. `useState(callback)` — Ленивая инициализация.
*   **Официальный ответ:**
    > Позволяет отложить дорогие синхронные калькуляции дефолтного значения стейта. Коллбэк отработает исключительно 1 раз.
*   **🗣️ Pitch (Middle):**
    > *"Если `initialState` высчитывается из `JSON.parse(localStorage.getItem('theme'))` — прямой вызов функции убьет процессор. `localStorage` медленный. `JSON.parse` тоже. Мы передаем в стейт `() => readStorage()` и избегаем лагов при последующих ререндрерах компонента."*

### ❓ 65. Почему `setCount(count + 1)` не всегда работает корректно внутри слушателей?
*   **Официальный ответ:**
    > Стейт-батчинг (особенно в асинхронных коллбеках). Три вызова `setCount(count + 1)` подряд изменят стейт только на +1. Нужно использовать функциональный апдейтер `setCount(prev => prev + 1)`.
*   **🗣️ Pitch (Middle):**
    > *"Функциональный set-коллбэк дает нам гарантию 100% свежего, последнего предыдущего стейта прямо из Fiber Queue реакта. Если мы используем просто замыкание, компилятор видит старое значение переменной `count`. Я всегда использую `prev =>` для счетчиков, тогглов (переключателей) `setOpen(open => !open)` и модификации списков (push в массив)."*
