## 🐣 0. Базовые понятия (Для тех, кто только зашел)

Прежде чем лезть в дебри, запомни 3 главных правила Next.js в 2026 году:
1.  **Server Components (RSC) по умолчанию:** Весь твой код сначала пытается выполниться на сервере. Это круто, потому что клиенту не летит лишний JavaScript.
2.  **Клиент только там, где 'движ':** Если нужна кнопка, `onClick` или `useState` — только тогда пишем `'use client'`.
3.  **App Router — это структура:** Папки — это твои пути (URL), а файлы внутри (page, layout) — это твои страницы.

## 🚀 1. Стратегии Рендеринга (Rendering 2026)

### ❓ 1. Что такое Partial Prerendering (PPR)?
*   **Простыми словами:** Представь, что ты заказываешь пиццу. Тебе мгновенно приносят пустую коробку и салфетки (Статическая оболочка), а саму горячую пиццу кладут в коробку через 2 минуты (Динамическая часть). Юзер сразу видит структуру сайта, не дожидаясь данных из базы.
*   **Официальный ответ:**
    > **PPR** объединяет статическую оболочку (Static Shell) и динамические области (Dynamic Holes) в одном HTTP-ответе.
*   **Пример с комментариями:**
    ```ts
    // next.config.ts
    const nextConfig = {
      // Это 'рубильник', который включает магию PPR в Next.js 16
      cacheComponents: true, 
    };

    // app/dashboard/page.tsx
    export default function Dashboard() {
      return (
        <main>
          {/* Этот заголовок будет в бандле как чистый HTML, юзер увидит его за 0ms */}
          <h1>Личный кабинет Альфа-Банка</h1>
          
          {/* Все, что в Suspense — это 'динамическая дырка' */}
          <Suspense fallback={<p>Загрузка баланса...</p>}>
            {/* Этот компонент пойдет в базу, он не блокирует загрузку страницы */}
            <DynamicUserBalance /> 
          </Suspense>
        </main>
      )
    }
    ```
*   **🗣️ Pitch:**
    > *"PPR — это лучший UX в 2026 году. Мы убираем белый экран ожидания, давая пользователю мгновенный статический контент, а тяжелые данные догружаем стримингом."*

## 📦 2. Data Fetching & Кэширование (v15+)

### ❓ 2. Как работает кэширование `fetch`? (Почему все поменялось?)
*   **Почему это важно:** Раньше (в v13/14) Next.js кэшировал всё подряд. Это приводило к тому, что юзер видел старый баланс карты. В v15 разработчики сказали: "Хватит! Безопасность важнее".
*   **Официальный ответ:**
    > В v15+ `fetch` **НЕ кэшируется по умолчанию** (`no-store`).
*   **Пример с разбором:**
    ```ts
    // Ситуация А: Данные меняются каждую секунду (Баланс, Курс валют)
    // Ничего не пишем — Next.js сам сделает запрос на каждый рендер.
    const freshData = await fetch('https://api.bank.com/balance');

    // Ситуация Б: Данные почти никогда не меняются (Список городов, FAQ)
    // Добавляем force-cache, чтобы не бить по API зря.
    const staticData = await fetch('https://api.bank.com/cities', { 
      cache: 'force-cache' 
    });
    ```
*   **🗣️ Pitch:**
    > *"В v15 мы ушли от 'неявного' кэширования к явному. Это делает работу с финансовыми данными в Альфа-Банке безопаснее: лучше один лишний запрос, чем показать пользователю чужой или старый баланс."*

## 🏗️ 3. Server Actions & формы (React 19)

### ❓ 3. Зачем использовать Server Actions вместо обычных API?
*   **Аналогия:** Обычный API — это когда ты звонишь курьеру, диктуешь адрес, а потом ждешь. Server Action — это когда ты просто нажимаешь кнопку "Хочу пиццу", и система сама знает, кто ты, где ты и что в корзине (интеграция с сервером бесшовная).
*   **Пример (Разбор по строчкам):**
    ```tsx
    // actions.ts
    'use server'; // Магическая строка: этот код выполнится ТОЛЬКО на сервере

    export async function payInvoice(prevState: any, formData: FormData) {
      const id = formData.get('id'); // Достаем данные из формы напрямую
      // Тут можно безопасно делать запросы к БД, API ключи не утекут на фронт!
      return { success: true };
    }

    // InvoiceForm.tsx
    // useActionState — хук из React 19 для управления состоянием формы
    const [state, action] = useActionState(payInvoice, null);
    
    // useFormStatus — знает, идет ли сейчас отправка (pending)
    const { pending } = useFormStatus();

    return (
      <form action={action}>
        <input name="id" />
        <button disabled={pending}>Оплатить</button>
      </form>
    );
    ```

## 🖼️ 4. Продвинутый Роутинг

### ❓ 4. Parallel & Intercepting Routes — в чем фишка?
*   **Кейс для банка:** Ты листаешь ленту трат. Нажимаешь на одну — открывается красивое окошко (модалка). Но если ты нажмешь "Обновить" или скопируешь ссылку — это будет отдельная страница.
*   **Зачем это нужно:** Тебе не нужно писать сложную логику открытия модалок в `useState`. Роутер Next.js сам понимает: если юзер пришел по ссылке — покажи страницу, если нажал внутри сайта — покажи модалку.

## 🛡️ 5. Оптимизация и SEO (2026)

### ❓ 5. Зачем нам `generateMetadata`?
*   **Для новичка:** Если ты скидываешь ссылку в Telegram, там появляется красивая карточка с картинкой и описанием. Чтобы эта карточка была правильной для каждой страницы, мы используем эту функцию.
*   **Пример:**
    ```ts
    export async function generateMetadata({ params }) {
      const product = await getProduct(params.id);
      return {
        title: product.name, // Например: "Кредитная карта Альфа-Тревел"
        openGraph: { images: [product.img] }
      };
    }
    ```
