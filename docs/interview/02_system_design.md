# 🏛️ Frontend System Design (Senior / Product Engineer)

Этот документ содержит архитектурные вопросы, паттерны проектирования масштабных систем и задачи на System Design интервью. Упор сделан на Fintech (безопасность, отказоустойчивость) и AI-приложения (Streaming, LLM-интеграция).

## 🟢 РАЗДЕЛ 1: Архитектура и микрофронтенды

### ❓ 1. Как бы вы спроектировали архитектуру большого банковского веб-приложения?
*   **Официальный ответ:**
    > Использование монорепозитория (NX/Turborepo) с разделением на Host-приложение (Shell) и независимые фичи-микрофронтенды (Вклады, Карты, Кредиты) через Webpack Module Federation или Rsbuild. Разделение бизнес-логики через FSD (Feature-Sliced Design). Интеграция с бэкендом через паттерн BFF (Backend-For-Frontend).
*   **🗣️ Pitch (Senior):**
    > *"Моя цель как архитектора — независимость команд (Developer Experience) и отказоустойчивость (Fault Tolerance). Каждая продуктовая команда должна иметь свой пайплайн релиза фичи 'Кредитов'. Я выберу Module Federation. Для защиты Host-контейнера от падения дочерних модулей, каждый Remote-компонент оборачивается в строгий Error Boundary. Слой данных (API) абстрагируется через BFF (на Nest.js), который агрегирует вызовы к COBOL/Java бэкенду банка, отдавая на React чистые DTO (Data Transfer Objects), избавляя фронт от тяжелой трансформации."*

### ❓ 2. Module Federation: Как решается проблема дублирования зависимостей (React, UI Kit)?
*   **Официальный ответ:**
    > В конфигурации Webpack/Vite указывается секция `shared`, где библиотеки (React, Zustand, UI-Kit) помечаются как `singleton: true`. Host-приложение загружает их в window один раз, а Remote-приложения используют уже загруженный инстанс.
*   **🗣️ Pitch (Senior):**
    > *"Если версии React в Host (18.2) и Remote (18.3) разъезжаются, сборщик смотрит на поле `requiredVersion` (обычно из package.json). Если стоит Strict mode, приложение упадет ('Unsatisfied version'). Поэтому мы используем Eager loading (немедленную загрузку) для вендоров и строгую синхронизацию версий Core-пакетов через единый `package.json` в корне монорепы. Разные версии UI-kit могут жить параллельно, но React и Zustand обязаны быть синглтонами, иначе контексты (React Context) 'разорвутся'. "*

### ❓ 3. Backend-Driven UI (Server-Driven UI): Как спроектировать динамические экраны?
*   **Официальный ответ:**
    > UI строится на основе контракта (JSON) от сервера. JSON содержит массив компонентов (`type: 'Banner', props: { text: 'Sale' }`). Фронтенд работает как интерпретатор (Renderer), который маппит JSON-узлы на реальные React-компоненты.
*   **🗣️ Pitch (Senior):**
    > *"В Сбере мы использовали BDUI для продуктов, где критически важен Time-to-Market (новые условия вкладов). Самая большая сложность тут — аналитика и действия. Для этого мы реализовали Action Builder. Вместе с UI-деревом бэкенд присылал `onClick: { type: 'ROUTE', payload: '/deposit' }` или `onClick: { type: 'API_CALL', endpoint: '/submit' }`. Фронт прокидывал эти экшены в единый Event Manager. Это позволило нам менять бизнес-флоу вкладов без релиза мобильного приложения и веб-версии, просто меняя JSON на бэке."*

### ❓ 4. State Management в больших приложениях: Zustand, Redux Toolkit, или Apollo/Query?
*   **Официальный ответ:**
    > Разделение 'Server State' (данные с бэка) и 'Client State' (UI-состояние, открытые модалки, темы). Для сервера используется React Query / RTK Query / Apollo (авто-кэширование, дедупликация, ревалидация). Для клиента — Zustand или React Context (для простых вещей).
*   **🗣️ Pitch (Senior):**
    > *"Хранить данные API в Redux/Zustand сегодня — антипаттерн, приводящий к тысячам строк бойлерплейта и проблемам инвалидации кэша. Мой стандарт: TanStack React Query для серверного стейта (с Stale-while-revalidate паттернами). Zustand я использую как тонкий слой для локального стейта (настройки фильтров, стейт Sidebar-а) и для синхронизации с URLSearchParams. В AI-приложениях я использую Zustand для хранения истории текущего диалога, оптимизируя рендеры через атомарные селекторы `useStore(state => state.messages)`."*

### ❓ 5. Проектирование UI Kit (Дизайн-системы). NPM пакет или монорепа? Versioning?
*   **Официальный ответ:**
    > UI Kit разрабатывается изолированно в монорепе, тестируется через Storybook + Chromatic (Visual Regression). Публикуется как приватный NPM-пакет или шарится через внутренние пути (workspace:*).
*   **🗣️ Pitch (Senior):**
    > *"Разработка UI Kit — это разработка Open-Source продукта для внутренних клиентов (разработчиков). Главное здесь — API интерфейсов. Мы используем паттерн Polymorphic Components (`<Button as="a" href="...">`), строгий Semantic Versioning (мажорная версия брейкает API) и паттерн 'Headless UI' (как в Radix UI) для сложных компонентов (Select, Combobox), где логика (accessibility, фокусы, стейт) инкапсулирована в хук `useSelect`, а рендер (дерево DOM) полностью настраивается продуктовой командой, если им не подходит дефолтный."*

## 🟢 РАЗДЕЛ 2: Безопасность (Security в Fintech)

### ❓ 6. Как спроектировать JWT-авторизацию, абсолютно защищенную от кражи (XSS/CSRF)?
*   **Официальный ответ:**
    > *   Access Token в памяти (`Closure` или локальная переменная), Refresh Token в `httpOnly Secure SameSite=Strict` Cookie.
    > *   Либо оба токена в `httpOnly Cookie` + Anti-CSRF защита (или Relying on SameSite).
*   **🗣️ Pitch (Senior):**
    > *"В финтехе localStorage/sessionStorage запрещены для токенов — любой XSS скрипт их украдет. Идеальная схема: Backend авторизует юзера и ставит 2 куки. Первая — Access (до 15 минут) как `httpOnly`, вторая — Refresh (месяц) как `httpOnly + path=/api/refresh`. Фронт с API общается в режиме `withCredentials: true`. Так как у нас микросервисы (разные сабдомены), мы устанавливаем атрибуты `Domain=.bank.com` и `SameSite=Lax`. Фронт вообще не знает о существовании токенов, браузер сам прикрепляет их к запросам, убивая поверхность атаки на XSS."*

### ❓ 7. Content Security Policy (CSP). Как это работает и почему это больно?
*   **Официальный ответ:**
    > CSP — это HTTP заголовок (или мета-тег), диктующий браузеру, откуда можно грузить скрипты, картинки или открывать WebSocket. Блокирует XSS и Data Exfiltration.
*   **🗣️ Pitch (Senior):**
    > *"Внедрять CSP на легаси-проекте без простоя — это искусство. Мы используем заголовок `Content-Security-Policy-Report-Only` в паре с эндпоинтом сборщика логов (Sentry). Около месяца мы собираем метрики, выявляя все сторонние метрики и скрипты, которые блокировал бы CSP. Затем переводим в жесткий режим (`default-src 'self'`). Для защиты от Injected Scripts (eval, inline JS), мы используем `nonce-соли` (криптографически случайные строки), генерируемые сервером на каждый рендер, которые прописываются в разрешенные теги `<script nonce="123">`. Это уничтожает 99% XSS векторов."*

### ❓ 8. Защита от Man-in-the-Middle (MITM) на фронтенде
*   **Официальный ответ:**
    > HTTPS/TLS решает большинство проблем. Использование HSTS (HTTP Strict Transport Security) и Certificate Pinning в мобильных приложениях.
*   **🗣️ Pitch (Senior):**
    > *"Для критичных банковских онлайн-транзакций (переводы средств) мы реализуем E2EE (End-to-End Encryption) прямо на уровне JS. Мы просим пользователя ввести PIN-код (или используем WebCrypto API для подписи). Фронтенд сам шифрует JSON payload открытым ключом банка (RSA/ECC) перед отправкой поверх обычного HTTPS. Даже если корпоративный Proxy/VPN или балансировщик нагрузок банка расшифрует HTTPS, они увидят лишь криптографический мусор. Расшифровать запрос сможет только HSM-модуль (аппаратный криптографический сейф) в самом защищенном периметре бэкенда."*

## 🟢 РАЗДЕЛ 3: AI Architecture (LLM, RAG, Generative UI)

### ❓ 9. Спроектируйте архитектуру клиента для общения с несколькими LLM (GPT-4, Claude)
*   **Официальный ответ:**
    > Использование слоя абстракции. Фронт не должен запрашивать OpenAI Store напрямик (утечка API ключей в браузере недопустима). Запросы идут на свой Backend/API Gateway, который проксирует их к провайдеру и отдает обратно SSE stream.
*   **🗣️ Pitch (Senior):**
    > *"Архитектура состоит из 3 слоев: UI (React), State (Zustand) и Network (SSE Adapter). Для мультимодальности Frontend шлет統一 (Unified) DTO объект на наш Gateway. Backend (на Go/Node) реализует 'Adapter Pattern', трансформируя наш стандартизированный запрос в формат OpenAI, Anthropic или Gemini. Назад Gateway всегда возвращает стандартизированный поток SSE. Это делает слой React 'Smart Chat' полностью агностичным к провайдеру LLM, позволяя переключать тяжелые модели на лету для A/B тестирования стоимости инференса."*

### ❓ 10. Как вы реализуете Generative UI (Component Rendering from LLM response)?
*   **Официальный ответ:**
    > Использование LLM Function Calling (например Tool Use). Модель отдает JSON (например, `{ tool: 'WeatherWidget', city: 'Moscow' }`). Фронтенд парсит этот фрагмент и рендерит React Component с этими пропсами.
*   **🗣️ Pitch (Senior):**
    > *"Проблема в том, что LLM отдает валидный полный JSON только в самом конце генерации ответа. Ждать 10 секунд для рендера виджета — ужасный UX. Я использую технику 'Streaming Partial JSON Parsing' (инструменты типа Vercel AI SDK или безопасные regex-парсеры), которые умеют валидировать куски (чанкование) JSON на лету, типа `{ "tool": "WeatherW` ... В связке с React Suspense, мы показываем Скелетон UI виджета сразу, как только модель начала генерировать его имя, и заполняем данные внутри него прямо "на лету" (Streaming props), что дает мгновенный интерактивный фидбек пользователю."*

### ❓ 11. Как вы будете решать проблему ограничения контекста (Context Window limit) на клиенте AI-ассистента?
*   **Официальный ответ:**
    > Хранить всю историю локально (Zustand + IndexedDB). Для отправки применять стратегии сжатия: обрезать старую историю, суммировать (Summarization) прошлый диалог в 1 абзац.
*   **🗣️ Pitch (Senior):**
    > *"Фронтенд — это тонкий клиент. Гонять историю в 5 мегабайт текста туда-сюда по мобильной сети на каждый чих — это смерть (и сброс 'Prefix Caching' кэша на сервере LLM). Архитектура требует серверного стейта истории (Sessions State). Frontend хранит в local storage только ID текущей Session. При отправке сообщения фронт шлет только `{ sessionId: "...", new_text: "..." }`. Бэкенд вытаскивает историю из Redis, склеивает ее, делает RAG (поиск документов) и оркестрирует запрос к LLM. На фронте скроллинг чата вверх реализуется обычной пагинацией или курсорами, без загрузки всей истории в RAM браузера."*

### ❓ 12. Local-first AI: как перенести работу LLM прямо в браузер пользователя (Offline mode)?
*   **Официальный ответ:**
    > Использование WebGPU и проектов вроде WebLLM или Transformers.js. Модель (~1-4 GB) скачивается в Cache Storage браузера и инференсится ресурсами 3D-видеокарты клиента без серверов.
*   **🗣️ Pitch (Senior):**
    > *"Потрясающая технология для Data Privacy (медицинские данные) и жесткой экономии серверных костов. Основная продуктовая боль — загрузка: качать 2 GB на 3G сети нельзя. Применяется архитектура 'Cloud Fallback': в первые секунды все запросы мгновенно летят на облачный (наш) LLM API, а в фоне (чере Service Worker API Background Fetch) скачиваются уже скомпилированные и квантованные веса (Int4/Int8) легкой модели вроде `Llama-3-8B`. Как только EventBus рапортует 100% загрузки, стейт-менеджер переключает транспортный слой с `CloudAdapter` на `WebGPU Adapter`. Юзер даже не замечает, как интерфейс переходит в offline-first."*

## 🟢 РАЗДЕЛ 4: Data Management & Performance 

### ❓ 13. System Design: Лента новостей с бесконечным скроллом на React (Infinite Scroll)
*   **Официальный ответ:**
    > Компоненты: `React Query` (`useInfiniteQuery` для курсорной пагинации дата-фетчинга), `IntersectionObserver` API (для детекта конца списка), React Virtualized / Virtuoso (для рендеринга только видимых эл-тов).
*   **🗣️ Pitch (Senior):**
    > *"Главный капкан Infinite Scroll — утечка памяти и сотен тысяч DOM-узлов. Если в DOM больше 2000 тяжелых компонентов-постов (с видео и лайками), браузер повиснет. Я обязую использовать `Virtuoso` (Windowing Data Structure), где физически в DOM всегда находится лишь ~15-20 элементов (видимая зона + пре-буфер). Вторая проблема: юзер перешел в 'Детали поста' и сделал свайп назад (Back navigation). Vanilla JS убьет позицию скролла. Используя `React Query` для кэширования агрегированного массива страниц и хуки сохранения скролла (через sessionStorage + `scrollRestoration: manual`), мы моментально восстанавливаем отрендеренный стейт без Layout Shifts (CLS скачков)."*

### ❓ 14. Как реализовать Collaborative Editing (Совместное редактирование документа как в Figma/Google Docs)?
*   **Официальный ответ:**
    > Использовать Operational Transformation (OT) или современный алгоритм CRDT (Conflict-free Replicated Data Types). На фронте применяется `Yjs` или `Automerge` поверх WebSockets или WebRTC протоколов для peer-to-peer.
*   **🗣️ Pitch (Senior):**
    > *"Я выберу CRDT поверх библиотеки `Yjs`, интегрированную с Prosemirror (или Lexical) редактором через их коннекторы. CRDT гарантирует математическое сведение конфликтов к единому состоянию (eventual consistency) без нужды в центральном сервере-'правдолюбе', решающем кто прав. Для масштабирования сети (когда редакторов файла 100+), mesh-сеть WebRTC(p2p) умрет из-за экспоненциального количества коннектов. Поэтому используется звезда через WebSocket: все браузеры шлют байтовые инкрементальные Yjs-diff(ы) изменения (а не весь стейт HTML) на легковесный Node.js Relay сервер (или Redis PubSub), который делает Broadcast (рассылку) остальным."*

### ❓ 15. Offline-First Архитектура для сложного Enterprise Мобильного Web-приложения (PWA)
*   **Официальный ответ:**
    > Service Workers (Cache Storage для Application Shell и статики), IndexedDB (для данных/стейта, хранящих гигабайты). Архитектурные библиотеки вроде RxDB или WatermelonDB для реактивной работы поверх локальной SQLite/IndexedDB базы.
*   **🗣️ Pitch (Senior):**
    > *"В мобильном Enterprise (например, выдача кредитов) мы используем `WatermelonDB` (или кастомный фасад над idb). Краеугольный камень — архитектура 'Optimistic UI' с фоновой синхронизацией: когда кредитный инспектор нажимает 'Сохранить изменения', мы моментально пишем в локальную IndexedDB статус `sync_status: 'pending'` и рисуем зеленую галочку. В фоне 'Sync Manager' шлет POST к API. Если мобильного интернета нет (инспектор в подвале) — запрос ложится в `BackgroundSync Queue API` (очередь Service Worker-а). Когда юзер выйдет на улицу в 4G, SW проснется от ОС, вытолкнет очередь на сервер, бэкенд отдаст `200 OK`, и SW пошлет `BroadcastChannel` сообщение во вкладку React, чтобы обновить статус на `sync_status: 'done'`. Решаются потери данных на 100%."*

### ❓ 16. Как оптимизировать Core Web Vitals (LCP, FID/INP, CLS) для e-commerce портала?
*   **Официальный ответ:**
    > **LCP** (Крупная отрисовка): Убрать lazy loading с главного баннера, Image Optimization (AVIF), CDN + SSR с `preload`.
    > **INP** (Interaction to Next Paint, замена FID для отзывчивости): Web Workers для JSON-парсинга, React `Suspense`, удаление/откладывание 3rd-party аналитик (GTM) в `requestIdleCallback`.
    > **CLS** (Смещение макета): Распорки для картинок/реклам (css aspect-ratio).
*   **🗣️ Pitch (Senior):**
    > *"Архитектура производительности (Performance Budget) начинается до написания кода — с аналитики EdgeCDN. Я настраиваю Next.js так, чтобы HTML стримился кусками. Картинка-LCP загружается в первую очередь (`<link rel="preload">` в head). Для драматического улучшения метрики INP (которая меряет общую "задумчивость" страницы в миллисекундах) мы переносим логику визуальных эффектов (фильтры списков, тяжелые селекты с тысячами опций) в хук `useTransition` React 18+. Это разбивает длинные Task-и (Long Tasks > 50ms) на чанки (Yielding to Main Thread). UI продолжает мгновенно отвечать на скролл/клики юзера, оставляя браузер "живым" во время рендера тяжеловесной структуры DOM."*

## 🟢 РАЗДЕЛ 5: CI/CD, Тестирование и Observability Инфраструктура

### ❓ 17. Спроектируйте стратегию Trunk-Based Development релизного цикла Frontend(Zero Downtime)
*   **Официальный ответ:**
    > Команда комитит в 1 ветку (main). Процесс: Linting -> Unit Test -> TS Check (tsc --noEmit) -> Build (Webpack/Vite) -> E2E Smoke Tests (Playwright) -> Deploy (загрузка Assets на S3 / CDN). Прием 'Feature Flags' позволяет прятать сырой функционал.
*   **🗣️ Pitch (Senior):**
    > *"Мы живем без Git-Flow и недельных Merge Conflicts проблем, используя Trunk-Based ветвление. Но как не сломать Прод? Мощные **Feature Toggles (Флаги фичей).** Новый, недописанный до конца UI профиля деплоится в production-бандл каждый день, но он завернут в React Guard-компонент, который открыт только для QA-инженеров и разработчиков (по Internal Cookie или IP). При деплое, наши статика JS/CSS асинхронно заливаются на S3 бакет под Edge CDN Cloudflare-а, а замена `index.html` делает атомарный (zero-downtime) релиз. Если всплыл баг уровня P0 (Production Zero) — откатиться(Rollback) к предыдущей версии занимает 2 секунды путем возврата старого `index.html`."*

### ❓ 18. Стратегия качества кода: "Testing Trophy" Кента Си Дотса vs "Тестовая пирамида"? Как делить?
*   **Официальный ответ:**
    > Пирамида (база — много unit, сверху — чуть e2e) устарела для фронта. Testing Trophy гласит: упор (середина кубка) делается на **Интеграционные тесты** (React Testing Library + MSW), где компонент тестируется вместе со своими хуками и мокнутой сетью, проверяя поведение (Behavior), а не реализацию (Implementation details).
*   **🗣️ Pitch (Senior):**
    > *"Хрупкие Unit-тесты на глупые React-компоненты (тесты на то, какой CSS-класс навесился) — это мертворожденный легаси-код, ломающийся при малейшем рефакторинге дизайна. Я пропагандирую Testing Trophy. Мы обильно (Unit) тестируем лишь чистые 'JS-функции' (стейт-редьюсеры, парсеры данных из LLM, DateUtils). Для UI слоя 70% покрытия делаем через RTL-интеграцию, мокая сложный API-ответ через MSW (Mock Service Worker прямо в NodeJS среде Jest/Vitest). E2E-тесты в браузере (Playwright/Cypress) пишем исключительно на критичные (Critical Path) 'Happy Paths' для бизнеса (цельное флоу 'Зашел-Оформил заявку-Увидел Успех'), так как тысячи e2e тестов флакуют (Flaky Tests) и будут бежать целый час на CI Runner-ах (что блокирует Velocity)."*

### ❓ 19. Дизайн Observability и Error Handling (Поиск плавающих багов)
*   **Официальный ответ:**
    > Глобальный перехват ошибок: `window.onerror`, React `ErrorBoundary` (верхнего уровня для страницы, и локальные для виджетов), отлов отклоненных промисов `unhandledrejection`. Все телеметрии пишутся в Sentry, бизнес-метрики в DataDog/Amplitude.
*   **🗣️ Pitch (Senior):**
    > *"Обычный Sentry показывает сухой JS стектрейс, но никогда не дает контекста 'Дошел ли до этого юзер и что он пошагово кликал'. Я строю Observability-архитектуру на паттерне Redux/Zustand Middleware (или Event Bus Logger): 100% экшенов юзера (клики по роутам, API запросы с HTTP-кодами, смены стейта фильтров) атомарно пишутся в Ring Buffer на 20-30 элементов в локальной RAM-памяти браузера (Breadcrumbs trail). Если происходит фатальный краш, верхний ErrorBoundary достает эту историю и лепит ее в тэги к ошибке, вдобавок мы делаем через `html2canvas` (или rrweb API) мини-скриншот последней секунды DOM-дерева или запись сессии. Sentry-сниппет становится настолько информативным, что разработчики чинят UI-баги без какого-либо привлечения QA за часы, а не дни."*

### ❓ 20. Как оптимизировать большой монолитного Enterprise-React приложения (Bundle Size Budget)?
*   **Официальный ответ:**
    > Техники: Route-based Code Splitting (`React.lazy`), Component Splitting (для тяжелых модалок), Tree Shaking архитектура (замена тяжелых библиотек вроде `lodash` на `lodash-es` с ES-модулями), сплиттинг 'Вендоров' (React/Polyfills в отдельный хэшированный бандл), Brotli-сжатие на Nginx.
*   **🗣️ Pitch (Senior):**
    > *"Декомпозиция Bundle Size должна быть на уровне CI/CD автоматики. Я интегрирую плагин `webpack-bundle-analyzer` (или аналоги для Vite) и настраиваю в GitHub Actions шаг 'Bundle Size Limit/Budget check'. Если Pull Request разработчика увеличивает размер `main.js` больше чем на 10KB (limit violation) — Action падает и билд красится в красный (Merge Blocked). Главные враги перфоманса (пожиратели килобайт) — это устаревшие Date-библиотеки с локалями (меняем тяжелый `moment.js` на встроенный `Intl` браузера), и Иконки-Пакеты, которые импортируют все классы разом (нужно настраивать babel-plugin-import). Более того, сложные и тяжеловесные UI-участки (например PDF-вьюер, Excel Editor, или тяжелые библиотеки графиков `Chart.js`, `Three.js`) всегда выносятся в асинхронный/динамический импорт и подгружаются в браузер с 500ms задержкой ТОЛЬКО по клику юзером, очищая Initial TTI (Time to Interactive) главного окна продукта."*

## 🟢 РАЗДЕЛ 6: Next.js Architecture 

### ❓ 21. RSC (React Server Components - App Router) vs Pages Router (SSR/SSG). Какую стратегию рендеринга вы выберете для высоконагруженного финтех-портала?
*   **Официальный ответ:**
    > App Router использует React Server Components (RSC) по умолчанию. Страницы, Layout-ы генерируются на сервере и грузятся в браузер как легковесные 'DOM Tree JSON', вообще без клиентского JS (если нет Event Handlers-ов). Для сложной клиентской интерактивности (анимации, стейт React-хуков) используется директива `"use client"` сверху файла.
*   **🗣️ Pitch (Senior):**
    > *"Будучи архитектором, я строго разделяю понятия 'Что' и 'Где'. Директива `"use client"` — это не шаг назад к SPA-лего (как думают джуны), это физическая граница (Network Boundary) между Node.js сервером и браузерным JS движком. Моя архитектурная стратегия в Next.js 14+: Роутинг, вложенные Layouts, и фетчинг тяжелых SEO/Public данных осуществляется СТРОГО в RSC-компонентах (на Backend Edge серверах). Они 'передают' сырые, обработанные JSON/отрендеренные DTO-данные пропсами в клиентские "листья" (виджеты): `<ClientWidget initial={data} />`. Гидратация (Hydration - перебор DOM деревом для навешивания слушателей JS) происходит **ТОЛЬКО** для этого маленького виджета интерактивной Финансовой Формы, экономя десятки Мегабайт React-бандла браузера. Для динамичных биржевых сводок я агрессивно применяю Partial Prerendering (PPR) — каркас сайта статически генерируется в CI(SSG) и кэшируется, а пустые 'дырки-заглушки' (`Suspense fallbacks`) стримятся (Chunked Transfer) динамическими котировками в Real-time."*

### ❓ 22. Паттерн Server Actions (Мутации) в Next.js. Плюсы, минусы и вектор взлома (Security)?
*   **Официальный ответ:**
    > Это TS-асинхронные функции (RPC паттерн), которые вызываются напрямик из клиентского OnClick обработчика/компонента, но физически выполняются СТРОГО на сервере (за счет директивы `"use server"`). Под капотом Next.js инкапсулирует и делает невидимый POST-Fetch HTTP запрос. Полностью исключает нужду в написании CRUD Boilerplate API-роутов. Нативно работают для HTML HTML Form (`action={submitFormAction}`).
*   **🗣️ Pitch (Senior):**
    > *"Server Actions ломают парадигму REST, невероятно ускоряя TTM (Time-to-Market) фич. Тем не менее, архитектурно - это публичный, открытый в интернет API-эндпоинт (POST Endpoint)! Если Server Action наивно принимает `formData: FormData` на вход от React-компонента, любой хакер (обойдя фронтенд) может отправить туда вредоносный POST-запрос с Postman/curl утилиты. Поэтому ВЕРШИНОЙ и первой строкой каждого моего Server Action-а является полная 'Server-Side Security' защита. Я требую три слоя: проверка JWT/Cookie сессии (`cookies().get(...)`), строгая проверка RBAC авторизации ролей юзера, валидация DTO Payload схемы через Zod `z.parse(data)`. Только убедившись, что это валидный пользователь с Role: Admin, из правильной организации со строго типизированным JSON-телом, Server Action получает право сделать UPDATE (мутацию) запись в SQL базе данных."*

## 🟢 РАЗДЕЛ 7: State & Reactivity at Scale (Масштабирование стейта)

### ❓ 23. Signals vs Reactivity (Solid.js / Preact vs React) — в чем архитектурная разница?
*   **Официальный ответ:**
    > React VDOM сравнивает два дерева (Reconciliation) на каждый `setState`, спускаясь сверху вниз (O(N)). Signals создают точечный направленный граф зависимостей. При изменении Сигнала обновляется *только* конкретный DOM-узел (например текст внутри `<span>`), минуя любой diffing и жизненный цикл компонента.
*   **🗣️ Pitch (Senior):**
    > *"Архитектурно Signals решают фундаментальную проблему React — непредсказуемый рендеринг 'по цепочке вниз'. В финтехе, когда тикеры акций мигают 50 раз в секунду, React начинает 'кипеть' от вызовов `Object.is()` и пересчета VDOM. Я внедрял стейт-менеджеры на базе Signals (напр. Jotai или Preact/Signals внутри React-приложения). Они позволяют напрямую биндить атомы стейта к тексту (Bypass rendering phase), опуская React-рендеринг до O(1). Мой компонент `StockTicker` может ни разу не перерендериться инструментами React, но в браузере цифра внутри него будет летать синхронно с вебсокетом."*

### ❓ 24. Паттерн Event Sourcing во фронтенде (на базе Redux)
*   **Официальный ответ:**
    > Стейт не хранится "как есть". Стейт — это производная вычисления массива всех событий (Events), произошедших с начала времен (как банковская выписка). Redux реализует урезанный Event Sourcing через экшены и редьюсер.
*   **🗣️ Pitch (Senior):**
    > *"Я применял полноценный архитектурный CQRS / Event Sourcing для сложных многостраничных форм (Wizard) оформления ипотеки. Вместо хранения `{ name: 'Ivan', income: 100 }`, наша локальная IndexedDB писала массив транзакций: `[ {type: 'SET_NAME', payload: 'Ivan'}, {type: 'SET_INCOME', payload: 100} ]`. Это дало мощный Time-Travel Debugging прямо в проде. Если форма ломалась, юзер отправлял нам JSON с массивом экшенов, и мы пошагово 'проигрывали' его путь. А также это позволило реализовать идеальный undo/redo (отмену шагов), просто удаляя последний экшен из массива и пересобирая стейт через `reduce`."*

### ❓ 25. Immutable vs Mutable State (Redux vs MobX/Valtio) — когда что выбрать?
*   **Официальный ответ:**
    > *   `Immutable` (Redux) создает новые ссылки массивов. Гарантирует предиктивность, чистые функции, идеальную работу `React.memo()`. Дорого по памяти для Huge Data.
    > *   `Mutable` (MobX, Valtio) мутирует исходный объект `state.user.name = 'Ivan'` под капотом через ES6 Proxy. Работает в 10 раз быстрее с гигантскими таблицами.
*   **🗣️ Pitch (Senior):**
    > *"React задуман как State Machine, реагирующая на смену Pointer Reference (иммутабильность). Однако, в Enterprise-приложениях с тяжеловесными 'Деревьями-таблицами' (на 5к строк дерева) клонирование иммутабельных объектов (`...state, array: [...state.array]...`) заставляет Garbage Collector браузера 'задыхаться' (GC pauses). Для таких изолированных кусков (Heavy Data Grids) я проксирую архитектуру через `Valtio` или внедряю мутабельный подход State Management-а (где React выступает лишь как слушатель-Snapshot, а не источник правды), спасаясь от диких фризов Main Thread-а."*

### ❓ 26. Проектирование Data Fetching слоя: Как избежать Waterfall Requests (Каскадных запросов)?
*   **Официальный ответ:**
    > Waterfall — это когда запрос B не начнется, пока не ответит запрос A, из-за вложенности компонентов. Решения: загружать все параллельно через `Promise.all()`, выносить API вызовы (Preload) на высший уровень Router-а (Loader pattern).
*   **🗣️ Pitch (Senior):**
    > *"Waterfall убивает конверсионую воронку (Funnel). Джуны часто пишут: `UserComponent` фетчит профиль юзера, и после рендера внутри него `PostsComponent` начинает фетчить посты. Юзер ждет 2 секунды, чтобы только НАЧАТЬ второй запрос. Для устранения Waterfall я применяю **Render-as-you-fetch** паттерн. Вся логика фетчинга проецируется на слой 'Router Loaders' (как в React Router v6.4+). Router, ДО начала рендера первого div-а страницы, параллельно запускает все 5 необходимых к странице API вызовов. Мы экономим 2-3 секунды TTI, а верстка рендерится сразу с Suspense-подгрузкой."*

## 🟢 РАЗДЕЛ 8: Инфраструктура, Сборка и Developer Experience (DX)

### ❓ 27. Архитектура Monorepo (Turborepo vs NX) — Как радикально ускорить CI Pipeline?
*   **Официальный ответ:**
    > Монорепа хороша тем, что код (напр Frontend + Backend) всегда в актуальном стейте и нет боли npm-publish'ей. Инструменты вроде `Turborepo` / `NX` хэшируют Build/Test таски и запускают их.
*   **🗣️ Pitch (Senior):**
    > *"Проектируя монорепозиторий в банке (Сбер), я выбрал инструмент NX (или Turborepo) для постройки мощного графа зависимостей (Dependency Graph). Ключевое архитектурное решение тут — CI/CD. Когда разработчик коммитит в пакет `shared-ui/Button`, CI сервер НЕ должен бежать автотесты на приложение 'Ипотеки', если оно не использует эту кнопку. NX вычисляет `unaffected` (незатронутые ветки) пакеты на базе AST-дерева импортов, пропуская их сборку. Это сократило среднее время нашего Pipeline-а слива PR (Pull Request) с 45 минут до 3-5."*

### ❓ 28. Migration Strategy (Vite vs Webpack vs Rsbuild/Rspack) на крупном Enterprise-легаси
*   **Официальный ответ:**
    > Уход от медленного Webpack (на NodeJS) к более быстрым сборщикам на Rust/Go (esbuild/Rspack/SWC) для ускорения старта dev-сервера и HMR (Hot Module Replacement).
*   **🗣️ Pitch (Senior):**
    > *"Миграция банковского легаси Webpack-монстра на Vite — это огромный бизнес-риск, так как ломается совместимость `require()`, контекстов и webpack-плагинов. Vite в среде dev-mode (esbuild ESM) сильно отличается от prod-mode (Rollup), что вскрывает страшные баги 'undefined imports' только на проде. Мой выбор для мягкой миграции — это `Rsbuild / Rspack` (от создателей ByteDance). Это Drop-in replacement (прямая замена) для Webpack-а: он написан на Rust, поддерживает 95% API-плагинов Webpack и loaders, но собирает проект в 10-15 раз быстрее. Наш переезд занял 3 дня, а HMR отклик ускорился с 12 секунд до миллисекунд. DX (Developer Experience) вырос колоссально."*

### ❓ 29. Архитектура Design Tokens (Tokens Studio, W3C Design Tokens Format)
*   **Официальный ответ:**
    > UI Kit не должен использовать захардкоженные HEX-цвета (`#fff`). Используются семантические токены `color-background-primary`. Дизайн-токены генерируются в Figma и через CI экспортируются в JSON, который конвертируется в CSS Variables/Sass/Android XML.
*   **🗣️ Pitch (Senior):**
    > *"Идеальная дизайн-система агностична к продуктовым фреймворкам. Мы настроили D2C пайплайн (Design to Code): дизайнеры в Figma (через плагин Tokens Studio) меняют радиус скруглений и нажимают 'Push'. Figma дергает Webhook нашего CI (GitHub Actions). Runner берет сырой JSON со стилями, прогоняет через `Amazon StyleDictionary` и автоматом публикует новый npm-пакет `theme-tokens@2.1.0`. Фронтенд-приложения на React и iOS/Android-приложения при следующем билде просто подтягивают новые CSS Variables и XML. Разработчики ВООБЩЕ не участвуют в процессе сдвига пикселей."*

### ❓ 30. Что такое BFF (Backend for Frontend) паттерн с точки зрения GraphQL Federation?
*   **Официальный ответ:**
    > BFF агрегирует данные специфично под нужные клиенту поля (мобильный BFF ужимает картинки, веб BFF возвращает фуллсайз). GraphQL Federation (например Apollo) позволяет склеить множество маленьких GraphQL-микросервисов в один "Supergraph", к которому фронтенд делает единственный запрос.
*   **🗣️ Pitch (Senior):**
    > *"В финтехе у нас было 20+ Data-микросервисов (Профиль, Баланс, Акции, AML-лимиты). React клиент (с дохлого мобильного процессора) не должен сам оркестрировать эти 20 REST API-запросов и склеивать их в useEffect'ах. Мы построили 'Supergraph Mesh Architecture'. Мы поднимаем NodeJS (BFF) шлюз с `Apollo Federation`. Он парсит 1 входящий GraphQL запрос от React, сам распараллеливает вызовы во внутренние REST-микросервисы (используя DataLoaders для борьбы с N+1), и отдает агрегированный JSON. Это переносит тяжеловесную 'Бизнес-оркестрацию' со слабых 3G-мобильников на наши датацентры банка, где Network пинги между серверами базы данных всего < 1ms."*

## 🟢 РАЗДЕЛ 9: LLM Orchestration & Architecture (Generative Engineering)

### ❓ 31. System Design: Архитектура AI Support Agent (Чат-бот с RAG архитектурой)
*   **Официальный ответ:**
    > Чат-бот на базе базы корпоративных знаний. Юзер задает вопрос ('Как закрыть карту'). Сначала делается Vector Search (Embedding) по документам Confluence/компании. Самые близкие абзацы (куски) вшиваются в prompt для LLM, чтобы она опиралась на факты.
*   **🗣️ Pitch (Senior):**
    > *"Проектируя AI Support Agent-а, главное не скормить юзеру 'Галлюцинации'. Архитектура Frontend RAG'а: Frontend шлет запрос юзера. Бэкенд на базе LlamaIndex делает векторный поиск (Pinecone), находит 5 кусков банковской документации по 'закрытию карт', и добавляет их в System Prompt. НО для фронтенда я требую паттерн **'Citations'** (Ссылки-пруфы). Backend возвращает Streaming JSON: сначала летит кусок массива `[{docId: 'doc_1', title: 'Условия дебетовых...'}]` (Это References), а затем сам текст `Для з...а...к...р...ы...тия...`. UI парсит References и в тексте потокового ответа LLM отрисовывает кликабельные сноски `[1]`, показывая юзеру источник. Без пруфов внедрять AI в Banking UI — безответственно."*

### ❓ 32. Оркестрация системного контекста (System Prompt Versioning) на стороне UI
*   **Официальный ответ:**
    > Системные промпты определяют поведение AI (например: 'Ты финансовый ассистент'). Они должны быть версионированы и обновляться без деплоя(перезапуска) frontend-приложения.
*   **🗣️ Pitch (Senior):**
    > *"Системные Role-Промпты — это такой же код, и их нельзя хардкодить в `const prompt = '...'` на клиенте. Frontend при инициализации диалога запрашивает с бэкенда актуальный `promptConfig_v4.json` (который управляется продакт-менеджерами через CMS, например Sanity). В этом JSON зашиты ограничения, Style Guide общения (Tone of Voice), параметры Temperature и Stop-слова LLM. Frontend склеивает этот Remote Context вместе с локальным Runtime Context (напр. 'Баланс текущего юзера: $1000') и отправляет в Inference API. Это развязывает цикл деплоя (Release Cycle) разработчиков UI от настройки характера AI моделей."*

### ❓ 33. Интеграция AI Function Calling: Sandboxing (Концепция 'Human-in-the-loop')
*   **Официальный ответ:**

*   **🗣️ Pitch (Senior):**

### ❓ 41. Blue-Green Deploy vs Canary Releasing во Frontend среде (Traffic Routing Analytics)
*   **Официальный ответ:**
    > *   `Blue-Green`: Одним щелчком переключаем 100% трафика юзеров на новую версию B. Если есть фатальный баг — переключатель на 100% возвращает старой А.
    > *   `Canary` (Канареечный): Аккуратно пускаем 5% юзеров на новую версию, чекаем метрики/Ошибки. Если ок — повышаем до 20%, 50%, 100%.
*   **🗣️ Pitch (Senior):**

Самая сложная и важная суть — 'Session Affinity' (прилипаемости юзера). Если мы кинули юзера Петра в 5%-пул (Версия В с новым крутым Header-ом), он ОБЯЗАН получать эту версию 'В' при любой перезагрузке страницы, вплоть до конца раската. Наш CD-пайплайн внимательно следит за DORA метриками (или Sentry Alerts): если в первые часы 5% трафика выбивает скачок API/Console/Chunk Exceptions — Canary-релиз автоматически стопается системой и переходит в Rollback-state спасая нам 95% клиентской выручки."*

### ❓ 44. Memory Leaks (Утечки памяти) в SPA приложениях (React): Как искать и архитектурно предотвращать их?
*   **Официальный ответ:**
    > Утечки памяти (Memory Leaks) = не отписанные кастомные Event Listeners на window/document, забытые `setInterval()` без clear, или ссылки в гигантские Closure/Global массивы и WeakMaps. Ищем через Chrome DevTools -> Memory tab -> Heap Snapshot (смотрим счетчик и графу 'Retained Size').
*   **🗣️ Pitch (Senior):**
    > *"Самый страшный баг и коварная утечка в React 18+ — это 'Detached DOM Elements'. DOM Element остается 'оторван', но Garbage Collector браузера его не удаляет годами. ## 🟢 РАЗДЕЛ 13: Edge Computing и WebAssembly (Wasm) интеграции

### ❓ 46. Как WebAssembly (Wasm) интегрируется в современную Frontend Архитектуру? Вектор применения в UX и AI?
*   **Официальный ответ:**
    > WebAssembly (Wasm) — компактный бинарный формат инструкций, выполняемый на виртуальной машине браузера со скоростью, близкой к C/Rust/C++ (Native. Он не заменяет JS инфраструктуру, он проксирует тяжелые расчеты. Работает с JS в связке (Interop), выполняя тяжелую математику, обработку аудио/видео(FFmpeg), сжатие картинок или PGP-шифрование БД.
*   **🗣️ Pitch (Senior):**
    > *"JavaScript — крутой протокол, но он строго однопоточен (Single Thread) и катастрофически медленный (из-за JIT/Garbage Collector) для Big Data и векторной алгебры. В банке мы использовали WebAssembly-модуль (скомпилированный из C++ OpenCV кода) для сверх-быстрой и надежной видеообработки и биометрической проверки (Face Liveness Detection в браузере при онбординге кредита до сетевого вызова API). 
    Вторая гигантская сфера — Generative AI архитектура (AI Agents/RAG). Wasm — это сердце. Инференс (запуск тяжелой нейросети 'Внутри' фронтенда, проекты вроде Transformers.js) под капотом компилирует Python/PyTorch тензоры в формат ONNX(через Wasm-биндинги), обеспечивая невероятный перфоманс в браузере без использования V8-JS Engine. Самая прогрессивная коммуникация между JS и Wasm потоками сегодня (во избежании CPU-bottleneck-а) проходит через общую `SharedArrayBuffer` память (Zero-copy), полностью обходя дорогостоящий JSON.stringify() сериалайзинг передачи Data-тикеттов."*

### ❓ 47. Edge Infrastructure (Cloudflare Workers, Next.js Edge Middleware). Что это меняет в слое Frontend Architecture?
*   **Официальный ответ:**
    > **Edge Servers** (Периферийные серверы) раскиданы по всему земному шару. В отличие от тяжелых Region Nodes (напр, Сервер в Калифорнии), Edge сервера обрабатывают запрос на Node/Ближайшей вышке связи до юзера (до 10ms-30ms ping). Выполнение JS "Edge-функций" идет на голом V8 Isolate (без полноценной 'тяжелой' Node.js среды), обеспечивая старт контейнера за фантастические < 1ms.
*   **🗣️ Pitch (Senior):**
    > *"В современных 'High-Scale' Server-Driven продуктах, классическая архитектурная стрела смещается от 'Монолита (Браузер)' к 'Edge-близости'. Я агрессивно использую Vercel/Cloudflare Edge Middleware как глобальный шлюз (Network Gateway), ПРЕДВАРИЯ React-приложение. 
    Что я выношу на Edge: 1. Аутентификация JWT (проверка срока действия токена в куки, отсеивая 401 запросы у края интернета, не будя наши бэкенды); 2. A/B Testing Routing & Feature Flags (проверка кук и редирект на /b-variant на Edge-е); 3. Bot Protection/Rate Limiter и 4. Localization Detection (на лету парсинг `Accept-Language` заголовка юзера). — Все это вычисляется на серверах Cloudflare в 10ms от юзера, ПЕРЕД тем, как вызов 100% пройдет в США и дойдет до наших Node/Java/React Server Components серверов базы данных(150ms). Это колоссально разгружает наш Node-Backend, делает Scale (масштабирование) почти бесплатным, и полностью предотвращает мигание UI (Content Layout Flash) от A/B тестов в браузере (нагружая CDN, а не браузер)."*

## 🟢 РАЗДЕЛ 14: Advanced Typescript Types Mapping & System Design Architecture 

### ❓ 48. Engineering Types: Как типизировать сложный State Dispatcher (или Event Bus архитектуру, Websockets), чтобы полностью исключить 'any' payload-ы? 
*   **Официальный ответ:**
    > Использование паттерна 'Размеченных объединений' (Discriminated Unions) в TS. Мы создаем единый глобальный/модульный объединяющий тип (union), где есть константный ключ 'Type', по которому мы сужаем Payload. `type BaseAction = { type: 'A', payload: string } | { type: 'B', payload: number }`. 
*   **🗣️ Pitch (Senior):**
    > *"Любой масштабный Event Bus (или Redux-Store, или WebSocket Listeners) превращается в тыкву из-за кастинга типов `as unknown as TypeX`. Моя архитектура строится на паттерне TS Discriminated Union вкупе с 'Exhaustive Check' (Сквозная проверка компилятора на исчерпаемость вариантов switch-a).
    Мы прописываем один общий Union Type для поля-дискриминатора: `type`. Когда мы пишем `switch(action.type)` в коде редьюсера, Typescript 5.0 автоматически сужает (Type Narrowing Mechanism) тип переменной `payload` 'прямо в этой ветке'. НО архитектурный маст-хэв: финальный `default` case-fallback ВСЕГДА оборачивается в хелпер-функцию `assertNever(action)`, которая внутри себя выглядит (просто) как: `function assertNever(x: never): never { throw new Error('Unreachable code') }`. 
    Этот трюк заставит TS выбросить Compile Error (Красные логи на этапе сборки), если разработчик 'по фану' добавит новое событие `type: EVENT_C` в общий список Event-ов, *но забудет* написать физический обработчик Case в Switch-программе! Высочайший пилотаж TS-архитектуры и Typesafe Event Systems, убирающий человеческий фактор."*

### ❓ 49. Опасность паттернов: 'Branded Types' (Nominal Typing) vs 'Structural Typing' (Структурная типизация) в TypeScript системах
*   **Официальный ответ:**
    > TypeScript основан на концепции Structural Typing (Structurals — Если оно 'крякает' как утка, то это утка). Два интерфейса, у которых есть поле `id: string`, абсолютно одинаковы для Typescript-а, даже если их имена интерфейсов `UserID` и `PostID` разные! 
    Принудительные Branded Types (Номинальная типизация) — это хак с уникальными полями, заставляющий TS различать их как Java/C# классы.
*   **🗣️ Pitch (Senior):**
    > *"В сложной Banking CRM системе, пусковая путаница(human-error) при передаче `fromUserId: string` в параметр, который ждет `toAccountId: string` приведет к переводу 10.000 Евро на чужой левый счет (Огромный Security Business Bug). Натуральная Структурная типизация (Type-matching) TS спокойно *пропустит* вызов функции `transferMoney(accountId, userId)`, так как они под капотом оба являются просто примитивами типом `string`. 
    Я применяю индустриальный Nominal Pattern (Branded Types) для таких API: `export type UserId = string & { readonly __brand: unique symbol }`.
    Инжектируя этот фантомный 'символьный брэнд' (фантомный только для TS компилятора, после сборки это 100% сырая String-a, нулевой рантайм-кост), мы заставляем TS бить разработчика по рукам (IDE Error), если ты наивно попробуешь вставить 'string', 'number' или 'PostID' из Инпута в API сервисный вызов, который ждет строго `AccountId` brand type. Бескомпромиссная Enterprise-защита от человеческого 'Typo' (опечатка пропсов) или рефакторингов на уровне фазы Компиляции."*

### ❓ 50. Что такое Dependency Inversion Principle (буква DIP из S.O.L.I.D) в контексте современной Frontend (React) Архитектуры?
*   **Официальный ответ:**
    > Компоненты высокого уровня (UI модули, React-Кнопки/Формы) **не должны** жестко зависеть от модулей низкого уровня (Конкретные библиотеки `Axios`/`API`/`LocalStorage`). И те, и другие должны зависеть от Абстракций (Interfaces Types). Абстракции не должны зависеть от деталей, детали должны зависеть от абстракций.
*   **🗣️ Pitch (Senior):**
    > *"Senior-Архитектор не привязывает бизнес-продукт к хайповым Фреймворкам ('Vendor Lock-in' защита). Мы завтра можем выкинуть 'Zustand' и взять Redux, выкинуть 'Axios' и взять 'Fetch Native'. 
    Я строю DIP (Dependency Inversion Layer): везде в продуктовом UI коде (React.tsx) мы ВООБЩЕ не используем `import { axios } from 'axios'`. UI слой использует исключительно абстрактные хук-мосты: `useUserApi()`, который инжектится (поставляется) через React.Context (или кастомный хук типа Repository Pattern, или стейт-локаторы). 
    Сам Интерфейс контракта вызова `IUserRepository.fetch()` находится в 'Domain Layer' (папке логики FSD), а конкретная 'Грязная' имплементация (на базе сырого Fetch, Axios API выкликов, или MSW моков/LocalStorage баз) инжектится(вставляется/биндится) на самом верхнем уровне корня дерева приложения `App.tsx` (или `DI-Container`-а). 
    Это позволяет 1 кнопкой подменять(Substitute) весь тяжелый Bank API на Mock-данные (MockRepository) в Jest/Playwright-тестах или Storybook(isolated environment), не касаясь жуткого хака файлов через `jest.mock('axios')` и не ломая production-конфиги компонентов!"*
