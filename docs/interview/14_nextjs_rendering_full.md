# 🪐 Модуль 14: Все виды Рендеринга в Next.js 15+ (PPR, Streaming, Dynamic IO)

Для прохождения на позицию **Senior** по Next.js вы должны знать не только базовые SSR/SSG, но и **новейшие паттерны**, которые появились в версиях 14 и 15 (React 19).

Вот полный атлас рендеринга современного Next.js.

---

## 🏗️ 1. Статический Рендеринг (Static Rendering)

*   **Суть:** Страница генерируется **один раз на сервере во время билда** (`npm run build`). Она сохраняется как статичный HTML файл.
*   **Где использовать:** Главная страница, Оферта, FAQ — то, что одинаково для всех пользователей.
*   **Next 15+ Деталь:** По умолчанию все маршруты в Next.js статические, пока вы не используете *динамические функции* (например, `cookies()`, `headers()`, `searchParams`).

---

## ⚡ 2. Динамический Рендеринг (Dynamic Rendering / SSR)

*   **Суть:** HTML генерируется сервером (Node.js) для **каждого запроса**.
*   **Где использовать:** Личные кабинеты, корзины покупок, баланс карты.
*   **Как включить за секунду:** Достаточно вызвать `await cookies()` или поставить `export const dynamic = 'force-dynamic'`.

---

## 🌊 3. Потоковый Рендеринг (Streaming)

*   **Суть:** Страница больше не генерируется «целиком». Сервер отправляет HTML-код **кусками (Chunks)** по мере готовности данных.
*   **Плюс:** Юзер видит сайт за сотые доли секунды.
*   **Как работает:** Кусок страницы (например, список товаров) оборачивается в `<Suspense fallback={<Loader />}>`. Бэкенд Next.js сначала отдает готовую статику (шапку), а тяжелый компонент «досылает» по тому же соединению позже.

---

## 👑 4. Частичный Рендеринг (PPR - Partial Prerendering) — *Вершина Next 15*

**Этот ответ заставит интервьюера «ахнуть». Это экспериментальная фича Next 14/15.**

*   **Суть:** Это комбинация Статического и Динамического рендеринга **внутри одной страницы без компромиссов**.
*   **Как это работает под капотом:**
    1.  Во время сборки Next.js берет страницу и компилирует всю её статическую часть (Каркас, Шапка, Футер) в готовый статический HTML.
    2.  Места, где есть динамический контент (например, `<Suspense>`), Next.js оставляет как **пустые сокеты (Holes)**.
    3.  Когда пользователь кликает на ссылку, Next.js **моментально** отдает статический каркас страницы (0мс).
    4.  Параллельно на сервере запускается динамический рендеринг для сокетов (например, баланса), который «дотекает» в страницу асинхронно.
*   **🗣️ Pitch:**
    > *"PPR решает дилемму: «Использовать ли SSG для скорости или SSR для актуальности?». С PPR мы отдаем каркас со скоростью статики (CDN), а персональные данные догружаем параллельно через Streaming. Это будущее веб-рендеринга."*

---

## 🧠 5. Dynamic IO (Эксперимент в Next 15)

*   **Суть:** Раньше в Next.js всё кэшировалось по умолчанию (что часто ломало динамику). В Next 15 вводится флаг **Dynamic IO**.
*   **В чем разница:**
    *   *Раньше:* Вызовы `fetch` кэшировались автоматически, пока не скажешь `no-store`.
    *   *Теперь (с Dynamic IO):* Любой вызов `fetch` или операции рендеринга **динамические по умолчанию**. Чтобы закэшировать данные, нужно явно использовать API `unstable_cache` или кэшировать маршрут. Это делает рендеринг более безопасным и предсказуемым.

---

### 📊 Итоговая шпаргалка по терминам на интервью:

1.  **Hydration (Гидратация):** Процесс, когда скачанный HTML оживает в браузере за счет навешивания JS-обработчиков React.
2.  **RSC (React Server Components):** Код выполняется **только на сервере**. Не качает JS в браузер.
3.  **Client Components (`"use client"`):** Код выполняется и на сервере (для первого рендеринга), и в браузере (для интерактивности).
