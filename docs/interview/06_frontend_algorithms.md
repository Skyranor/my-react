# ⚙️ Алгоритмы и Структуры Данных (Frontend / Senior Engineer)

На Senior/Product собеседованиях во Frontend (кроме FAANG) редко просят "инвертировать бинарное дерево на доске". Вас проверяют на "Фронтендерские алгоритмы": понимание того, как работает **браузерный Event Loop**, как **React сверяет Virtual DOM**, как оптимизировать рендер 100 000 элементов и управлять кэшом памяти.

От вас ждут не просто решения задачки, а способности оценить **Time & Space Complexity (О-большое)** и привязать абстрактный алгоритм к реальной бизнес-боли (Pitch).

Здесь собраны **15 фундаментальных алгоритмических концепций**, которые ежедневно применяются в проектировании сложных UI архитектур (Next.js, Redux, WebGL, AI Dashboards).

## 🟢 РАЗДЕЛ 1: Графы, Деревья и DOM (Trees & Graphs)

### ❓ 1. Обход дерева (Tree Traversal): BFS (в ширину) vs DFS (в глубину). Что когда применять на UI?
*   **Официальный ответ:**
    > *   **BFS (Breadth-First Search):** Использует Очередь (Queue). Сканирует дерево по слоям сверху вниз. Потребляет больше памяти $\mathcal{O}(W)$ на широких деревьях.
    > *   **DFS (Depth-First Search):** Использует Стек (Stack) или Рекурсию. Ныряет до самого нижнего листа ветки, затем поднимается. Требует мало памяти $\mathcal{O}(H)$.
*   **🗣️ Pitch (Senior):** 
    > *"DOM-дерево и вложенный JSON от бэкенда — это классическая древовидная структура. В 90% случаев (Deep Clone формы, сбор валидации, сериализация стейта) я применяю Рекурсивный DFS. Причина: глубина UI-компонентов редко превышает 20-30 уровней, так что нагрузка на Call Stack минимальна ($\mathcal{O}(H)$ Memory). НО! Если мне нужно найти ближайшего родителя по классу (Event Delegation Target) или вычислить кратчайший фокус ввода (Accessibility Keyboard Traversal), я использую исключительно BFS. Искаемый инпут скорее всего находится 'рядом' по DOM-дереву (соседний `div`), и мне не нужно рекурсивно нырять в глубину футера."*

### ❓ 2. Эвристика React Virtual DOM (Diffing Algorithm): Сведение $\mathcal{O}(n^3)$ к $\mathcal{O}(n)$.
*   **Официальный ответ:**
    > Классический алгоритм сравнения двух графов (Minimum Edit Distance) работает за $\mathcal{O}(n^3)$. Для дерева из 1000 элементов браузеру потребовался бы 1 миллиард операций. Команда React внедрила 2 эвристических (допущения) правила, чтобы свести время рендера к линейному $\mathcal{O}(n)$.
*   **🗣️ Pitch (Senior):**
    > *"Гениальность React Fiber кроется в двух допущениях. 1) Элементы разного типа (например `<div>` сменился на `<section>`) всегда порождают абсолютно разные поддеревья. React даже не будет их сравнивать: он экстренно 'убьет' старую ветку (Unmount) и создаст новую. 2) Допущение атрибута `key`. При вставке элемента в начало огромного списка, React без ключей мутировал бы каждую ноду ($\mathcal{O}(n)$ DOM апдейтов). Когда мы добавляем хэш-ключи, React внутри себя создает HashMap (Словарь). Он сверяет элементы за $\mathcal{O}(1)$ Lookup Time, понимая, что элемент нужно просто 'переместить' без дорогостоящего пересоздания DOM-узла. Без понимания этой эвристики невозможно писать производительные списки транзакций."*

### ❓ 3. Data Normalization: Сплющивание глубокого объекта (Flattening). Зачем это стейт-менеджерам?
*   **Официальный ответ:**
    > Normalization — алгоритмический процесс превращения дерева вложенных объектов (матрешки JSON) в плоские словари (Hash Maps), где данные связаны только по `ID`.
*   **🗣️ Pitch (Senior):**
    > *"Хранить глубоко вложенный JSON от бэкенда (`User -> Posts -> Comments -> Likes`) напрямую в React Context или Redux — это архитектурное самоубийство. Изменение количества лайков у комментария заставит перерендериться ВЕСЬ глобальный стор пользователя ($\mathcal{O}(N)$ глубокий апдейт). Мой паттерн Data Access Layer: на входе от API я пропускаю JSON через рекурсивный Flatten-алгоритм (или либу `normalizr`), превращая дерево в плоские словари: `{ users: {1: {..}}, comments: {32: {..}} }`. Это Реляционная БД на фронтенде. Теперь нажатие лайка — это микро-апдейт по ключу `comments[32].likes++` со сложностью $\mathcal{O}(1)$ Access Time. Ререндерится только одна крошечная кнопка, а не вся страница."*

### ❓ 4. Борьба с "Заморозкой" UI (Time Slicing / Chunking): Как отрендерить 100 000 элементов?
*   **Официальный ответ:**
    > В JavaScript один Main Thread. Если цикл `for` парсит 100к объектов 2 секунды, браузер замерзнет (белый экран, кнопки не нажимаются). Алгоритм 'Chunking' дробит выполнение тяжелой задачи на куски (chunks).
*   **🗣️ Pitch (Senior):**
    > *"Когда мне нужно загрузить и нарисовать гигантский график биржи (100 Mb WebGL Data), я никогда не паршу JSON в одном синхронном цикле. Я использую алгоритм кооперативной многозадачности (Yielding to Main Thread). Я разбиваю массив на 'чанки' по 50 миллисекунд (магически безопасный лимит времени (RAIL Model) для плавного UI). Высчитываю 50ms порции данных, затем делаю паузу — оборачиваю следующую итерацию в `setTimeout({}, 0)` или современный `requestIdleCallback`. В эту миллисекундную щель браузер успевает отрисовать (Paint) крутящийся лоадер, обработать клики юзера и запустить сборщик мусора. UI остается на 100% Responsive."*

## 🟢 РАЗДЕЛ 2: Patterns & Memory / Cache Management (Паттерны и Память)

### ❓ 5. Структура Данных `Trie` (Prefix Tree / Префиксное дерево) в Autocomplete поиске.
*   **Официальный ответ:**
    > Структура дерева, где каждый узел (Node) — это буква словарного слова. Путь от корня до узла формирует строку. Идеально для быстрого поиска строк по приставке (prefix). Время поиска $\mathcal{O}(L)$, где L - длина искомого слова, а не количество слов в словаре.
*   **🗣️ Pitch (Senior):**
    > *"Если у вас есть список из 100 000 банковских контрагентов, поиск подсказок (Autocomplete) через клиентский `array.filter(name.includes(query))` убьет процессор телефона, так как это займёт $\mathcal{O}(N \times M)$ времени. В высоконагруженных UI я преобразую массив сырых строк в `Trie` при старте приложения (или запускаю это внутри Web Worker). В префиксном дереве мне не нужно сканировать 99 000 лишних слов. Поиск слова "ГАЗ" займет всего 3 операции ($\mathcal{O}(3)$ узлов дерева) + небольшой DFS обход дочерних листьев, чтобы вернуть 5 подсказок. Молниеносный поиск из 90-х, воскресший для фронтенд-перфоманса."*

### ❓ 6. Алгоритм LRU Cache (Least Recently Used). Как ограничить память на клиенте?
*   **Официальный ответ:**
    > Кэш, который автоматически удаляет "наименее недавно использованные" элементы, как только превышает свой размер (Capacity). Внутри реализуется как комбинация `Hash Map` + `Doubly Linked List` (Двусвязный список). Чтение и Запись за $\mathcal{O}(1)$.
*   **🗣️ Pitch (Senior):**
    > *"Архитектору (Next.js/React) нельзя бесконечно запихивать результаты `fetch` или `useMemo` в один глобальный словарь Object (Memory Leak). Устройство пользователя лопнет по RAM. Для тяжелых вычислений (например математика нейронной сети в браузере) я обязательно пишу кастомный LRU Cache Wrapper с лимитом `capacity = 50`. HashMap дает мгновенный доступ к результату ($\mathcal{O}(1)$), а Двусвязный Список хранит хронологию 'Свежести'. Когда юзер открывает 51-й график, алгоритм отрезает 'Хвост' списка (Tail Node), выбрасывая самый старый график сборщику мусора, не сдвигая тысячу индексов массива."*

### ❓ 7. Observer vs Pub/Sub Pattern: Под капотом стейт-менеджеров (Redux, EventTarget).
*   **Официальный ответ:**
    > Паттерны для создания 1-to-Many подписок на события.
    > *   **Observer:** Субъект `Subject` (Стор) напрямую хранит в себе массив Обозревателей (Компонентов) и вызывает их методы апдейта.
    > *   **Pub/Sub:** Содержит прослойку `EventBus / Broker` (канал). Издатель и Подписчик не знают о существовании друг друга. 
*   **🗣️ Pitch (Senior):**
    > *"Иногда внедрять Redux/Zustand для связи двух разрозненных виджетов в легаси-коде — оверхед. React не может общаться с Vanilla-JS таблицей через Context. Я строю мост (Bridge) на базе паттерна Event Emitter (PubSub). Мой кастомный хук `useSubscribe(eventName, callback)` работает как Подписчик, а обычный TS-модуль бросает `bus.emit()`. Главная алгоритмическая опасность — Memory Leaks. Подписчик сохранил замыкание (Closure) с хуками реакта в памяти EventBus. Если компонент 'Модалка' уничтожается (Unmount), я ОБЯЗАН в `useEffect cleanup` вызвать `bus.unsubscribe()`. Иначе массив EventBus разрастется до тысяч мертвых функций-зомби, пожирающих RAM."*

### ❓ 8. Garbage Collection (Сборщик Мусора): `Mark-and-Sweep` и WeakRef API.
*   **Официальный ответ:**
    > В V8/JS мусор собирается алгоритмом 'Пометь и Вымети' (Mark-and-Sweep). Движок стартует от глобального объекта (Root / window) и рекурсивно 'красит' все достижимые по ссылкам объекты. Все "неокрашенные" сироты жестко удаляются из `heap` (Кучи).
*   **🗣️ Pitch (Senior):**
    > *"Архитектор обязан управлять 'Сильными' и 'Слабыми' ссылками. Если мы написали кастомный кэш DOM-элементов: `const cache = { [element.id]: expensiveData }` — мы создали сильную ссылку на DOM-узел. Даже если юзер закрыл страницу магазина, этот `div` останется висеть в памяти браузера навечно, так как Объект словаря 'держит' его (Mark & Sweep его найдет). Мой паттерн: я использую структуру данных `WeakMap`. Ключом выступает сам Объект `weakCache.set(element, data)`. Главная магия: как только `element` удалится из UI, `WeakMap` не воспрепятствует GC, он 'отпустит' память тихо и автоматически."*

## 🟢 РАЗДЕЛ 3: Rate Limiting & Concurrency (Параллелизм и Асинхронность)

### ❓ 9. Глубокое копирование (Deep Clone). Circular Reference (Циклические ссылки).
*   **Официальный ответ:**
    > `JSON.parse(JSON.stringify(obj))` — медленный хак, теряет `Date`, `Regex`, Классы, `undefined` и взрывается(Crash) на циклических ссылках (когда `A.parent = B; B.child = A`). Современный браузерный API — `structuredClone(obj)`.
*   **🗣️ Pitch (Senior):**
    > *"Крайне редкая задача переписать Deep Clone на собесе, но она вскрывает уровень синьора. Главный алгоритмический вызов: 'Recursion Call Stack Exceeded' на циклических ссылках. Если бы мне пришлось писать его с нуля, я бы передавал вторым параметром рекурсии структуру данных `WeakSet` или `Set (Visited Nodes)`. Как только рекурсия входит в Объект, она проверяет, был ли этот адрес памяти (reference) уже добавлен в `Set`. Если да — мы обрываем рекурсию и не проваливаемся в бесконечную черную дыру. Но в 2024 на проде я запрещаю писать кастомные клоны: нативный C++ `structuredClone()` превосходит по скорости любые JS полифиллы в 5 раз."*

### ❓ 10. Понимание Debounce & Throttle (Under the Hood). Оценка Замыканий (Closures).
*   **Официальный ответ:**
    > *   **Debounce:** Пауза (игнор) функции, пока не пройдет Z времени ТИШИНЫ (без единого нового вызова). Используем для Search Input.
    > *   **Throttle:** Гарантированное выполнение функции с 'Окошком' (Rate limit). Не чаще чем раз в Z миллисекунд, сколько бы раз ее не вызывали. Используем для Scroll/Resize. Реализация через Closure (Замыкание счетчика `lastRan`)
*   **🗣️ Pitch (Senior):**
    > *"Написать Throttle через `setTimeout` или `Date.now()` — база. Но для сверхчувствительного UI-Scroll'а или Canvas 'Рисовалки' таймауты дают рваную картинку (Jank), так как `setTimeout` болтается в MacroTask Queue и не привязан к частоте кадров монитора (60 Hz). Мой Senior-Throttle всегда базируется на `requestAnimationFrame`. Хук `useRafThrottle(fn)` вызывает функцию ровно 1 раз перед Paint/Composite фазой браузера (откидывая лишние вызовы в кадре). Это синхронизирует математику JS с видеокартой, давая идеальную 'стеклянную' плавность Drag-n-Drop интерфейсов."*

### ❓ 11. Concurrency Limiter (Пул Промисов / Semaphore). Загрузка 1 000 фото в браузере.
*   **Официальный ответ:**
    > `Promise.all` запускает все `fetch` параллельно одновременно. Скачивание 1000 картинок разом убьет Network Tab (ограничение Chrome 6 коннекшенов на домен) или повалит ваш Backend 500 ошибкой 'Слишком много подключений'. Алгоритм "Бассейна пулов" ограничивает "Параллелизм" до X штук зараз.
*   **🗣️ Pitch (Senior):**
    > *"Алгоритм Concurrency Queue — обязательная архитектура для пакетной загрузки (Bulk Uploads) в Банкинге. Я проектирую функцию-итератор: `async function asyncPool(limit, array, promiseFn)`. Внутри я создаю 'Сет' исполняющихся в данный момент промисов (`executing = new Set()`). В цикле `for...of` я запускаю `promiseFn`, ложу его в `executing`, а затем делаю `if (executing.size >= limit) { await Promise.race(executing) }`. Как только СКОРЕЙШИЙ из промисов выполнился, `Promise.race` разблокирует очередь, и мы изящно докидываем в топку следующий файл. Браузер всегда держит строго 5 HTTP подключений, не обрывая Socket."*

## 🟢 РАЗДЕЛ 4: Arrays & Algorithms Tricks (Решение проблем Списков)

### ❓ 12. Two Pointers / Sliding Window Pattern: (Алгоритм Скользящего окна). 
*   **Официальный ответ:**
    > Использование двух индексов (Left, Right) для итерации списка без создания копий массивов ($\mathcal{O}(1)$ Space).
*   **🗣️ Pitch (Senior):**
    > *"Позиционирую этот алгоритм для задачи Virtual Scroll (Виртуализация). У нас массив из 10 000 строк в UI (Redux). Рендерить их в DOM — смерть браузера. Мой кастомный 'Virtual List' работает по принципу 'Скользящего окна'. Window - это видимая часть монитора (Viewport). `PointerLeft` - это `scrollTop / itemHeight` (Индекс начала окна: 50). `PointerRight` - это `PointerLeft + (viewportH / itemHeight)`. Я делаю математический срез `array.slice(left, right)` + Отрисовываю только 20 нод на экране + Абсолютно позиционирую их внутри `div min-height: 100000px`. Это классический LeetCode алгоритм, спасший миллионы мегабайт оперативной памяти."*

### ❓ 13. Map / Reduce Under the Hood: Реализация своих Array Methods с учетом `this`.
*   **Официальный ответ:**
    > Кастомная реализация `Array.prototype.reduce` показывает понимание функционального программирования. Цикл `for(let i=0)`. Ключевой нюанс: проверка `initialValue`, если он не передан, первый элемент массива становится им.
*   **🗣️ Pitch (Senior):**
    > *"Полифиллы почти никто не пишет, но в Enterprise 'Reduce' ломает мозг джунам, когда они пытаются писать на нем конвейеры асинхронных тасок. `acc = await fn(acc)`. Архитектурный паттерн Chain of Responsibility (Цепочка обязанностей) в Next.js Middleware или Express роутере — это тот же самый `Array.reduceRight` под капотом. Я оборачиваю глубокие математические расчеты в Pipeline Composer (`const pipe = (...fns) => val => fns.reduce((a,c)=>c(a), val)`). Код очищается от промежуточных мусорных переменных, становясь чистой, самодокументируемой формулой (Pure Functions data-flow)."*

### ❓ 14. Event Delegation (Делегирование Событий): Алгоритм Bubbling vs Capturing.
*   **Официальный ответ:**
    > Вместо навешивания хука `onClick` (Event Listener) на 10 000 строк таблицы (Убьет память), мы вешаем ОДИН `onClick` на корневой тег `<table />`. Клик (Event) всегда всплывает вверх (Всплытие - Bubbling). Сверяем `event.target`.
*   **🗣️ Pitch (Senior):**
    > *"В React 17+ делегирование спрятали: React сам вешает один 'Слухач' на корень `#root` (раньше на документ) и симулирует Synthetic Events для нас в памяти. НО! Если вы интегрируете сторонний огромный D3.js SVG-график (или Canvas Data Grid) внутрь React дерева, и вам нужно ловить клики по микро-точкам нативного DOM, вешать `addEventListener` на каждую ветку SVG нельзя. Паттерн 'Event Delegation' спасет вашу жизнь. Мы вешаем 1 слушатель на React-враппер, прогоняем алгоритм проверки DOM-дерева через `event.target.closest('[data-id]')`, вытягиваем ID кликнутой точки за $\mathcal{O}(H)$ и инжектим стейт обратно в React State."*

### ❓ 15. Hashing Basics ( Consistent Hashing ): Как балансить WebSocket стейт в микросервисах?
*   **Официальный ответ:**
    > Классический хэш (IP % count_servers) ломается, если сервер упал: весь трафик перемешивается, юзеры теряют чат-сессии. Консистентное хэширование размещает Сервера и IP юзеров в кольце ($\pi / 360^\circ$). При падении 1 машины, страдает только 1 её соседний кусок.
*   **🗣️ Pitch (Senior):**
    > *"Фронтенд-Архитектор обязан понимать инфраструктуру (Backend-Driven FE), когда пишет Реалтайм-фичи (Socket.io). В банке миллионы коннекшенов (например Notification Bell). Обычные HTTP REST Load Balancers (Round Robin) тут рухнут: они не держат долгие сокеты, а юзер, отправивший сообщение в чат на 'Сервер А', может получить ответку от 'Сервера Б' и не увидеть свое сообщение. На уровне Nginx или Node.js Redis Adapter мы проектируем 'Sticky Sessions' через Consistent Hashing. ID юзера хэшируется, математически закрепляя его браузер-сокет строго к одному и тому же инстансу микросервису на долгие часы. Фронт работает как швейцарские часы."*

## 🟢 РАЗДЕЛ 5: Advanced Browser & DOM APIs (Продвинутый DOM)

### ❓ 16. `DocumentFragment` API: Как отрендерить 10 000 элементов без React за 1 Layout?
*   **Официальный ответ:**
    > `document.createDocumentFragment()` создает виртуальную (невидимую) "коробку" для DOM-узлов в оперативной памяти. Когда мы добавляем элементы в фрагмент (`appendChild`), это не вызывает Reflow/Repaint всей страницы.
*   **🗣️ Pitch (Senior):**
    > *"Иногда внедрять React для рендера виджета в легаси-проекте — это overhead. Если мне нужно вставить массив из 10к комментариев на чистом JS, использование `document.body.innerHTML += '...'` или `ul.appendChild(li)` в цикле убьет процессор каскадным Layout Thrashing (10к перерисовок экрана). Я создаю `const fragment = document.createDocumentFragment()`. В цикле я 'набиваю' этот виртуальный фрагмент узлами в памяти. И только в самом финале я делаю `ul.appendChild(fragment)`. Браузер перенесет все 10к элементов в реальный DOM за одну единственную операцию микро-Reflow."*

### ❓ 17. `MutationObserver` API: Отслеживание инжектов (3rd party scripts) в обход React.
*   **Официальный ответ:**
    > Интерфейс, который асинхронно реагирует на ЛЮБЫЕ изменения в DOM-дереве (добавление/удаление нод, изменения атрибутов классов).
*   **🗣️ Pitch (Senior):**
    > *"React полностью контролирует свой Virtual DOM, но он бессилен против внешних расширений Chrome (AdBlock, Grammarly, Google Translate) и баннерных сетей. Grammarly часто 'инжектит' случайные `<span>` прямо внутрь наших контролируемых инпутов, что намертво крашит React (знаменитая ошибка 'Node not found' при отмонтировании родителя). Архитектурный фикс: я вешаю `new MutationObserver()` на критические контейнеры (редакторы текста). Как только обзервер видит, что `addedNodes` содержит чужеродный спан от левого расширения Chrome, наблюдатель тихо вырезает его (sanitize) или подавляет ошибку до того, как React попытается его сравнить."*

### ❓ 18. `ResizeObserver` API: Смерть события `window.addEventListener("resize")`.
*   **Официальный ответ:**
    > Позволяет наблюдать за изменениями физических габаритов конкретного `<div/>`. Не привязан к размеру окна браузера.
*   **🗣️ Pitch (Senior):**
    > *"Хук `useWindowSize` на основе `window.resize` для Canvas/WebGL графиков — это антипаттерн в сложных Layouts. Если у пользователя открывается 'шторка' сайдбара — размер самого браузера не меняется, окно `resize` не выстрелит, но график сожмется и его Canvas 'покрякозябрится'. `ResizeObserver` решает эту фундаментальную архитектурную боль. Я вешаю его прямо на референс (Ref) родительского контейнера графика. Если ширина контейнера изменилась даже на 1 пиксель (из-за флексбоксов или Grid), `ResizeObserver` стреляет событием, и я перерисовываю WebGL Canvas идеально под новые размеры контейнера. Плюс, он работает внутри `requestAnimationFrame`, защищая нас от Layout Thrashing."*

### ❓ 19. Web Workers API: Настоящая многопоточность на фронтенде против Event Loop.
*   **Официальный ответ:**
    > JavaScript — однопоточный язык, но Web Workers позволяют спавнить 'фоновые' системные Потоки (OS Threads). Worker не имеет доступа к DOM, общаясь с Main Thread через Сообщения (`postMessage` / `onmessage`). Внутри него свой собственный Event Loop.
*   **🗣️ Pitch (Senior):**
    > *"В финтехе мы генерировали для B2B-клиентов Excel-отчеты CSV на 50 мегабайт (маппинг тысяч строк данных) прямо в браузере. Выполнение двойного цикла через `map`/`reduce` занимало до 3-х секунд непрерывной работы CPU. На это время интерфейс Next.js 'умирал' (замораживался Main Thread), юзер не мог даже нажать паузу. Мой архитектурный шаг: я вынес всю тяжелую математику (Data Processing, парсинг Excel/PDF) в изолированный `new Worker('excel.js')`. Main Thread просто шлет туда JSON сырых данных `worker.postMessage(data)` и рисует плавный лоадер на 60 FPS. Через 3 секунды Worker возвращает готовый бинарник, и UI 'оживает'."*

### ❓ 20. File, Blob API & `URL.createObjectURL`: Генерация скачиваний "на лету" ($\mathcal{O}(1)$ Network).
*   **Официальный ответ:**
    > Нативный Броузерный API файлов. Blob (Binary Large Object) — это бинарный кусок данных. `createObjectURL` превращает кусок памяти внутри RAM браузера во временную ссылку `blob:http://site.com/123`.
*   **🗣️ Pitch (Senior):**
    > *"Когда юзер жмет кнопку 'Скачать PDF квитанцию', джуниор делает GET запрос на бэкенд, бэкенд формирует файл на сервере AWS, шлет ссылку, и юзер качает. Это жжет наши ресурсы сервера впустую (Data Outbound costs). Если данные для квитанции УЖЕ есть у нас в React State (имя, сумма, дата), я генерирую PDF прямо в браузере пользователя через `jspdf`. Я получаю из библиотеки бинарный `Blob`, и делаю `URL.createObjectURL(blob)`. Затем я прокидываю этот локальный линк скрытому тегу `<a href="..." download>` и эмулирую клик `a.click()`. Скачивание файла происходит моментально, без единого HTTP-звонка на сервера банка."*

### ❓ 21. `queueMicrotask` API: Ручное управление Microtask Queue (Обход `setTimeout`).
*   **Официальный ответ:**
    > Событийный цикл (Event Loop) состоит из Макрозадач (`setTimeout`, `click`) и Микрозадач (Promises, `Observer`). Микрозадачи имеют абсолютный приоритет. `queueMicrotask(fn)` позволяет вручную поставить функцию в конец очереди Микрозадач.
*   **🗣️ Pitch (Senior):**
    > *"Использование `setTimeout(fn, 0)` для 'откладывания кода на потом' — это ловушка производительности. Если в этот момент браузер должен был отрисовать кадр (Render phase) или пользователь кликнул (Task), браузер сначала сделает их, и ваш таймер отложится на непредсказуемые 10-15 мс. Если я пишу критическую библиотеку State Management'a (React Query/Zustand), и мне нужно 'схлопнуть' 50 синхронных мутаций (Batch Updates) стейта в один групповой апдейт 'ПОСЛЕ' того, как текущий синхронный стек отработает, но СТРОГО 'ДО' того, как браузер перерисует Paint, я использую `queueMicrotask(batchFn)`. Эта задача гарантированно проскользнет до отрисовки."*

## 🟢 РАЗДЕЛ 6: События, Сеть и Безопасность (Events & Security)

### ❓ 22. Жизненный цикл события: Capturing vs Target vs Bubbling. `stopImmediatePropagation`.
*   **Официальный ответ:**
    > Любое событие (`click`) имеет 3 фазы: 1) Capturing (Погружение) — спускается от `window` сверху-вниз к элементу. 2) Target — достигает самого элемента. 3) Bubbling (Всплытие) — поднимается от элемента наверх (на чем основан Event Delegation).
    > `e.stopPropagation()` убивает Всплытие наверх (родитель не узнает о клике). `e.stopImmediatePropagation()` убивает Всплытие И БЛОКИРУЕТ все остальные слушатели клика НА ТОМ ЖЕ элементе (соседние хуки).
*   **🗣️ Pitch (Senior):**
    > *"Джуны часто спамят `stopPropagation`, ломая глобальную аналитику или сторонние виджеты, которые слушают клики на `document.body`. В сложном UI (например, Канбан-доска JIRA), когда я перетаскиваю карточку над колонкой, мне нужно, чтобы Drop-зона поймала событие 'НА ПОГРУЖЕНИИ' (Capturing phase: `addEventListener('drop', fn, true)`). Это позволяет Родителю (Колонке) перехватить событие РАНЬШЕ, чем до него доберется Ребенок (само текстовое поле внутри карточки), и заблокировать дефолтное поведение с помощью `preventDefault`."*

### ❓ 23. Оптимизация скролла: Зачем нужен `passive: true` в Event Listeners?
*   **Официальный ответ:**
    > Опция конфигурации события: `{ passive: true }`. Указывает браузеру: "Обещаю, внутри этого слушателя я НИКОГДА не вызову `e.preventDefault()`".
*   **🗣️ Pitch (Senior):**
    > *"Раньше мобильные браузеры (особенно Safari) безбожно 'тормозили' при скролле страницы. Причина — они ждали доли секунды, пока отработает весь JS в обработчиках `touchstart` или `wheel`, чтобы узнать, не вызовет ли джаваскрипт `preventDefault()` (отмену скролла браузера). Как только мы архитектурно начали добавлять `window.addEventListener('scroll', handler, { passive: true })`, браузер понял, что ждать отмены от JS больше не нужно. Он моментально запускает анимацию свайпа на GPU-треде операционной системы, не дожидаясь JavaScript Main Thread'а. Скролл становится идеально плавным (native like)."*

### ❓ 24. Взаимодействие Микрофронтендов: `CustomEvent` API.
*   **Официальный ответ:**
    > Механизм генерации пользовательских событий прямо в DOM. Позволяет компонентам "общаться", не зная друг о друге (PubSub).
*   **🗣️ Pitch (Senior):**
    > *"В корпоративных продуктах (Микрофронтенды) на одной странице может одновременно крутиться React (Хедер), Vue (Сайдбар) и Legacy jQuery-калькулятор. React Context тут бессилен. Наш архитектурный клей — глобальный браузерный `CustomEvent`. Когда в Vue-корзину добавляется товар, Vue делает `window.dispatchEvent(new CustomEvent('CART:ADD', { detail: { id: 1 } }))`. А наш React-виджет корзины просто в `useEffect` слушает это событие на `window`. Нулевая связность (Zero Coupling), полная изоляция фреймворков и идеальный Data Flow поверх зоопарка технологий."*

### ❓ 25. Shadow DOM Event Retargeting: Выход событий из Изоляции.
*   **Официальный ответ:**
    > При использовании Shadow DOM компонента (`mode: 'open'`) внутренний клик по кнопке 'всплывает' наружу в основной DOM. НО браузер подменяет отправителя (`event.target`). Реальным таргетом становится сам кастомный тег `<my-widget>`, а не `<button>` внутри него.
*   **🗣️ Pitch (Senior):**
    > *"Интегрируя B2B виджеты (Web Components) на сайты партнеров, мы столкнулись с болью аналитики (Google Tag Manager). Маркетолог ставит триггер в ГТМ: 'Лови клик по `button.buy`'. Но поскольку наша кнопка спрятана внутри инкапсулированного Shadow DOM, GTM видит только клик по нашему `<body>` (Retargeting скрыл физические 'кишки' нашего компонента). Чтобы починить отслеживание, нам приходилось прокидывать события 'наверх' руками через `CustomEvent`, дублируя каждый клик из тени в свет. Это цена абсолютной инкапсуляции."*

### ❓ 26. Security Basics: Почему JWT токены в `localStorage` — это крах безопасности (XSS)?
*   **Официальный ответ:**
    > Любой JavaScript код на странице может прочитать `localStorage` синхронно. XSS (Cross Site Scripting) — это уязвимость, когда хакер внедряет свой JS-скрипт на ваш сайт (через комментарий форума или уязвимую NPM-библиотеку). Этот скрипт крадет JWT-токен из Storage и шлет на хакерский сервер.
*   **🗣️ Pitch (Senior):**
    > *"При проектировании безопасности(Security) я жестко блокирую хранение Access/Refresh токенов в LocalStorage. Внедрение даже одного уязвимого NPM-пакета выкачает токены миллионов юзеров. Мой стандарт: Бэкенд возвращает JWT строго в форме **`HttpOnly; Secure; SameSite=Strict` Cookie**. Фронтенд (JS код) ФИЗИЧЕСКИ не имеет доступа к ней (`document.cookie` пуст). Сервер браузера сам (на C++) абсолютно безопасно прикрепляет её под капотом к каждому запросу. Это 100% иммунитет к XSS-кражам токенов."*

### ❓ 27. CSRF (Подделка запросов) и роль атрибута Cookie `SameSite`.
*   **Официальный ответ:**
    > CSRF: Хакер присылает вам ссылку на свой сайт (malware.com). На нем скрытая форма `<form action="bank.com/transfer" method="POST">` мгновенно отправляет деньги. Браузер сам прилепит к форме ваши активные Cookie сессии, и платеж пройдет.
*   **🗣️ Pitch (Senior):**
    > *"До 2020 года мы строили защиту через сложные Anti-CSRF-токены, гоняя их туда-сюда. Сегодня проблема пресекается на уровне браузера одной строчкой Cookie — `SameSite=Strict`. Это говорит браузеру: 'Клей куку к запросу ТОЛЬКО если запрос исходит из адресной строки `bank.com`'. В сценарии хакера запрос полетел с `malware.com`, браузер заблокировал передачу сессии, и банк отклонил транзакцию ответом `401 Unauthorized`."*

### ❓ 28. CORS (Cross-Origin Resource Sharing) и оптимизация Preflight `OPTIONS`.
*   **Официальный ответ:**
    > CORS — защита именно браузера. JS код с сайта A не может читать ответы API сайта Б, пока Б не пришлет заголовок `Access-Control-Allow-Origin: сайт А`.
*   **🗣️ Pitch (Senior):**
    > *"Основная перфоманс-угроза CORS в микросервисной архитектуре — это `Preflight OPTIONS` запросы. Когда мы шлем POST запрос с JWT в заголовке `Authorization`, браузер СПЕРВА делает проверочный пустой 'OPTIONS' запрос к API, удваивая время Network Roundtrip. На плохом 3G интернете мы получаем задержку в 500мс просто из-за настроек сетевой безопасности. Мой архитектурный фикс: на API Gateway (NGINX) нужно выставить заголовок `Access-Control-Max-Age: 86400`. Браузер один раз узнает, что домен безопасен, и закэширует это знание в память на сутки(24h). Мы моментально ускоряем все POST запросы юзера в 2 раза!"*

### ❓ 29. Content Security Policy (CSP): Абсолютная защита Фронтенда.
*   **Официальный ответ:**
    > HTTP-заголовок `Content-Security-Policy`. Строгий Allowlist (Белый список) доменов, откуда можно качать скрипты, шрифты и стили. Полностью блокирует In-line код (`<script>alert(1)</script>`).
*   **🗣️ Pitch (Senior):**
    > *"Для банков уровня Сбера CSP — Требование №1 от безопасников. Если XSS хакер смог обмануть React (`dangerouslySetInnerHTML`) и вставил майнер `<script src="hacker.com/miner.js">`, CSP с политикой `default-src 'self' api.sber.ru` жестко убьет этот `script-tag`, потому что домена хакера нет в Whitelist. Трюк при миграции огромных Легаси проектов на CSP: я включаю `Content-Security-Policy-Report-Only` политику. Она не ломает прод, но мы получаем json-репорты о нарушениях на наш сервер. Собираем логи месяц, формируем 'Карту API', и уже потом безболезненно включаем Block Mode."*

## 🟡 РАЗДЕЛ 7: Алгоритмическая База (Middle Level Core)

### ❓ 30. Set vs Array. Как удалить дубликаты из массива? Big O поиск.
*   **Официальный ответ:**
    > Уникальность — `Set` хранит только 1 экземпляр значения.
*   **🗣️ Pitch (Middle):**
    > *"Самый лаконичный способ убрать дубли: `[...new Set(array)]`. Маска 'Set' кардинально выигрывает у методов массива `.includes()` или `.indexOf()` по скорости. Поиск в Массиве — это перебор O(N). Если массив 10,000 элементов, мы делаем 10к итераций процессора. Поиск в `Set.has(element)` работает через хэш-функцию мгновенно за константное время O(1)."*

### ❓ 31. Map vs Object. Чем словарь ключей лучше классического объекта `{}`?
*   **Официальный ответ:**
    > Во-первых, в классическом объекте ключом может быть только Строка или Symbol (под капотом 123 приведется к `'123'`). В `Map` ключом может быть другой Объект или Функция. Во-вторых, Map гарантирует строгий порядок вставки ключей (Итерируемый словарь).
*   **🗣️ Pitch (Middle):**
    > *"Объект плох для частых мутаций (добавлений/удалений) высоконагруженных кэшей. Движок V8 оптимизирует Объекты под статичные Формы (Hidden Classes / C++ Structs), добавление новых ключей ломает эту оптимизацию. Формат `Map` изначально задуман как Хэш-таблица для динамического хранения огромных `Key:Value` объемов данных без де-оптимизации, обеспечивая O(1) и операции размера `.size` под капотом."*

### ❓ 32. Как работает Debounce (Устранение дребезга) под капотом?
*   **Официальный ответ:**
    > Функция, которая откладывает вызов переданного коллбека на 'N' миллисекунд после последнего вызова. Пока в Инпут быстро печатают буквы, функция ждет 'тишины'.
*   **🗣️ Pitch (Middle):**
    > *"Трюк строится на Замыканиях. У функции `debounce` есть замкнутая (локальная) переменная `timer`. Каждое нажатие клавиши вызывает `clearTimeout(timer)`, уничтожая старый таймер до того как он успел сработать, и запускает новый `timer = setTimeout(apiCall, 300)`. Запрос на Бэкенд (Live Search) улетит ровно 1 раз, только когда юзер сделает паузу в печатании на 300мс."*

### ❓ 33. Рекурсия и Stack Overflow Error. Хвостовая рекурсия.
*   **Официальный ответ:**
    > Рекурсия — это когда функция вызывает сама себя. Каждому такому кругу (Вызову) нужно выделить память (Frame) в Call Stack'е процессора для переменных функции. Когда слоев вызова накапливается 10,000+ — оперативная память стека переполняется: `Maximum call stack size exceeded`.
*   **🗣️ Pitch (Middle):**
    > *"Рекурсия незаменима для парсинга Деревьев (Обход Категорий Меню или Комментариев ФБ). Главное правило — вы обязаны иметь 'Base Case' (Условие выхода вроде `if(!node.children) return`). Глубокие обходы лечатся 'Итеративным подходом' с использованием глобального массива Валидаторов Stack[] или циклом `while(queue.length)` вместо вызова функции из самой функции."*
