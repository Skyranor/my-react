# 🧬 Модуль 11: Глубокое погружение в Микрофронтенды (Для Senior)

Этот документ — ваш технологический щит. Он объясняет **все три подхода** к микрофронтендам, их плюсы/минусы и устройство обмена данными. Прочитав его, вы сможете доказать, что ваш подход (NPM) был осознанным решением, а не случайностью.

---

## 🏗️ 1. Три типа композиции (Как собрать монолит из кусков)

Существует три фундаментальных способа склеить разные приложения в одну страницу.

### 🟡 А. Композиция на этапе сборки (Build-time) — *Ваш Опыт в Сбере*
*   **Как работает:** Модули пишутся отдельно, паблишатся как `npm-пакеты`. Хост-приложение делает `npm install @sber/deposits` и собирает единый бандл.
*   **Плюсы (Что продавать банку):**
    *   ✅ **Сверхнадежность:** Ошибки типов (`TypeScript`) ловятся до деплоя.
    *   ✅ **Tree Shaking:** Сборщик вырежет неиспользуемый код из пакета.
    *   ✅ **Единый бандл:** Нет оверхеда на рантайм-подкачку файлов, быстрый старт для юзера.
*   **Минусы:**
    *   ❌ Медленный релизный цикл: Нужно обновить пакет -> обновить Хост -> собрать Хост -> катить на прод.

### 🟢 Б. Композиция в рантайме (Run-time / Module Federation)
*   **Как работает:** Хост и модули деплоятся на **разные** домены. Хост при рендеринге подтягивает чужой JS файл (например, `remoteEntry.js`) через интернет и рендерит компонент.
*   **Плюсы:**
    *   ✅ Независимый деплой: обновили модуль — он обновился у всех за минуту.
*   **Минусы:**
    *   ❌ Сложный дебаг гонок данных (Race Conditions).
    *   ❌ Если упал чужой домен с модулем — у юзера упадет часть экрана.

### 🔴 В. Изоляция через iframe (Старый мир)
*   **Как работает:** Модуль рендерится внутри тега `<iframe>`.
*   **Плюсы:** Абсолютная изоляция CSS и JS.
*   **Минусы:** Отвратительный UX (нельзя всплывающие модалки вынести за край фрейма), оверхед на память браузера.

## 📡 2. Как микрофронтендам БЕЗОПАСНО общаться?

Даже если модули изолированы, им часто нужно синхронизировать данные (например, после оплаты в блоке «Платежи» обновить блок «Баланс» в Шапке).

### 🛠️ Паттерн 1: Шина событий (Event Bus / Custom Events)
*   **Суть:** Использование стандартного `window.addEventListener`.
*   **Пример в коде:**
```javascript
// В модуле Платежи при клике "Оплатить"
window.dispatchEvent(
  new CustomEvent('BALANCE_UPDATE', { detail: { value: 100000 } })
);

// В модуле Шапка
window.addEventListener('BALANCE_UPDATE', (e) => {
  setBalance(e.detail.value);
});
```
*   **Для кого:** Идеально для архитектур build-time, предотвращает утечки памяти (если не забывать `removeEventListener` в `cleanup`).

### 🛠️ Паттерн 2: BroadcastChannel (Межвкладочный)
*   Если пользователь открыл банк в двух вкладках браузера и оплатил в одной — баланс должен обновиться во второй вкладке автоматически. `BroadcastChannel` решает это на уровне браузера.

## 📦 3. Разделяемые зависимости (Shared Dependencies)

Один из главных вопросов на собеседовании: *"Если у вас 5 микрофронтендов, у пользователя скачается 5 копий React?"*.

*   **При Build-time (Ваш подход):** **Нет.** Сборщик (Webpack/Vite) видит все зависимости при сборке единого бандла. Если 5 пакетов просят `react@18`, сборщик оптимизирует их и оставит ровно **одну** копию React.
*   **При Run-time (Module Federation):** Зависимости дедуплицируются через настройки `Federation`:
```javascript
shared: {
  react: { 
    singleton: true, // Гарантирует только одну копию в рантайме
    eager: false     // Позволяет скачивать React лениво
  }
}
```

## 🎨 4. Изоляция Стилей (CSS Clashes)

Что если Модуль А задал `.button { color: red }`, а Модуль Б — `.button { color: blue }`?

*   **Как бороться:**
    1.  **CSS Modules (Стандарт):** Сборщик хеширует имена классов на выходе (`.button_d8sa2`).
    2.  **CSS-in-JS (Styled-components):** Стили генерируются динамически и изолированы на уровне компонента.
    3.  **Shadow DOM:** Тег `<div id="shadow">` с полной изоляцией стилей браузера. Нативные стили страницы внутрь Shadow DOM не проникают вообще.
