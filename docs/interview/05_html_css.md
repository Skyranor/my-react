Здесь собраны **25 продвинутых архитектурных вопросов** с ответами и форматом "Senior Pitch", демонстрирующим глубокую системную эрудицию.

## 🟢 РАЗДЕЛ 1: Browser Rendering & Performance (Сборка пикселей и Перфоманс)

### ❓ 1. Critical Rendering Path: Как браузер парсит HTML/CSS до появления первых пикселей (FCP)?
*   **Официальный ответ:**
    > 1. Браузер качает HTML, парсит его построчно в **DOM**-дерево.
    > 2. Встречая тег `<link>`, качает CSS, блокируя рендер, парсит в **CSSOM**-дерево.
    > 3. DOM и CSSOM сливаются в **Render Tree** (в него попадают только видимые ноды, без `display: none` и `<head>`).
    > 4. **Layout (Reflow):** Расчет геометрии (кто где стоит, ширина/высота).
    > 5. **Paint:** Отрисовка пикселей на слоях (цвета, тени).
    > 6. **Composite:** Видеокарта (GPU) склеивает слои воедино и выводит на матрицу монитора.
*   **🗣️ Pitch (Senior):** 
    > *"CRP для меня — это не теория из учебника, а ежедневная панель Performance в DevTools. Главный враг быстрого First Contentful Paint — блокирующий рендер (Render-blocking resources). Браузер физически не нарисует интерфейс, пока не скачает и не распарсит всю огромную CSSOM. Поэтому для Enterprise-приложений я использую паттерн **Critical CSS**: мы инлайним микро-кусок CSS (стили первого экрана) прямо в тег `<head type="style">`, а оставшийся тяжелый бандл стилей грузим асинхронно браузером в фоне. Это радикально сдвигает метрику LCP влево на графике."*

### ❓ 2. Reflow (Layout) vs Repaint vs Composite: Что "тяжелее" и как это анимировать?
*   **Официальный ответ:**
    > *   **Reflow:** Самое тяжелое. Изменение геометрии (`height`, `margin`, `top`). Вызывает цепную реакцию перерасчета соседей.
    > *   **Repaint:** Отрисовка визуала без изменения геометрии (`color`, `background-color`).
    > *   **Composite:** Склейка графических слоев на уровне Видеокарты (GPU).
*   **🗣️ Pitch (Senior):**
    > *"Классическая джуновская ошибка: писать анимацию слайдера через `left: -100px` или `width: 0 -> 100%`. Это вызывает каскадный Layout Thrashing: браузер вынужден 60 раз в секунду заставлять CPU математически пересчитывать дерево DOM. Это убивает батарейку телефона и дает дикие тормоза. Мой железный стандарт анимаций (Mobile-first): анимировать можно ИСКЛЮЧИТЕЛЬНО свойства `transform` (например `translate3d`, `scale`) и `opacity`. Браузер делегирует эти 3 свойства на стадию Composite (аппаратное GPU-ускорение). Они создают свой собственный изолированный Graphic Layer, не триггеря Layout и давая шелковые 60 FPS на старых Android-девайсах."*

### ❓ 3. Layout Thrashing (FSL - Forced Synchronous Layout): Как JavaScript тормозит CSS?
*   **Официальный ответ:**
    > Сценарий, когда JavaScript сначала изменяет DOM/CSS ноды, а затем в той же функции читает ее геометрические свойства (`element.offsetHeight`, `getBoundingClientRect`). Браузер ВЫНУЖДЕН экстренно синхронно провести дорогой Reflow прямо посреди JS-цикла.
*   **🗣️ Pitch (Senior):**
    > *"Layout Thrashing — тихий убийца перфоманса списков и таблиц. Представьте, что в React `useEffect` цикле мы добавляем элементу класс, а потом сразу читаем его `.scrollHeight`. Браузер паникует: чтобы вернуть точную высоту, ему нужно экстренно рассчитать всю математику страницы прямо сейчас. Если это внутри цикла `for(100)`, FPS упадет до нуля. Я лечу это архитектурно через батчинг (Batching): мы сначала СЧИТЫВАЕМ все мерки со всех элементов, сохраняем в память, и только потом массовым куском ПИШЕМ (изменяем стили) через `requestAnimationFrame`. FastDom паттерн."*

### ❓ 4. Свойство `content-visibility: auto`: В чем его революционность?
*   **Официальный ответ:**
    > Это CSS-свойство пришло на смену JS-фреймворкам для виртуализации. Оно говорит браузеру: "Отложи рендеринг (Layout и Paint) внутренностей этого DOM-узла, пока он физически не приблизится к видимой зоне экрана (Viewport)".
*   **🗣️ Pitch (Senior):**
    > *"В финтехе мы рендерим гигантские ленты транзакций клиента (бесконечный скролл). В старом React-мире мы тащили `react-window` или `react-virtualized`, чтобы руками вырезать DOM-ноды. `content-visibility: auto` (с `contain-intrinsic-size`) отдает эту задачу нативный движок C++ браузера. При загрузке HTML из 10 000 элементов транзакций браузер отрендерит только первые 20. Остальные 9980 нод будут просто висеть в DOM с нулевым Paint/Layout-весом. Рендер гигантской страницы падает с секунд до 20мс. Progressive Enhancement в чистом виде!"*

### ❓ 5. Оптимизация Шрифтов (FOUT vs FOIT). Зачем нужен `font-display: swap`?
*   **Официальный ответ:**
    > *   **FOIT (Flash of Invisible Text):** Браузер не показывает текст 3 секунды, пока кастомный шрифт скачивается (юзер видит белый экран).
    > *   **FOUT (Flash of Unstyled Text):** Браузер мгновенно показывает системный Arial, а когда шрифт скачался — "прыжком" подменяет его.
*   **🗣️ Pitch (Senior):**
    > *"Удержание текста 'в заложниках' в ожидании красивого шрифта недопустимо для Retention. Дефолтное поведение Safari — FOIT (белые буквы). Это смерть конверсии. Мой архитектурный паттерн: 1) В `@font-face` обязательно добавляем `font-display: swap` — приказывая браузеру 'показывай системный fallback немедленно, и подмени потом'. 2) Но при подмене шрифт 'прыгает' (Cumulative Layout Shift - CLS). Чтобы убить этот прыжок, я использую новые CSS API `size-adjust` и `ascent-override`: я подгоняю размеры системного Arial миллиметр в миллиметр под габариты кастомного футуристичного шрифта. При смене шрифта пиксели текста стоят как влитые."*

## 🟢 РАЗДЕЛ 2: CSS Architecture & Frameworks (Архитектура)

### ❓ 6. CSS-in-JS (Styled-Components, Emotion) vs CSS Modules/Tailwind. В чем трагедия CSS-in-JS в эпоху React Server Components?
*   **Официальный ответ:**
    > Styled-Components генерируют стили в Runtime (в браузере юзера) при помощи JavaScript. Server Components (Next.js 13+) рендерятся на Node.js без доступа к браузерному DOM или React-контекстам.
*   **🗣️ Pitch (Senior):**
    > *"Классический CSS-in-JS (Runtime) мертв в Enterprise RSC-модели. Если вы попытаетесь засунуть `<StyledDiv>` в серверный компонент Next.js, он выбросит вас в Client Boundary (`use client`), уничтожив всю прелесть Server Components, так как Emotion должен воткнуть `<style>` тег в клиентский Head через JS. Мы масштабно мигрируем фронтенд Сбера на 'Zero-Runtime' технологии: Tailwind CSS, CSS Modules илиVanilla-Extract. Они компилируются в нативные `.css` файлы на этапе Webpack/Turbopack сборки (Build-time). Итог: 0 KB JavaScript парсинга стилей в браузере, мгновенный FCP, чистый HTML."*

### ❓ 7. Почему хвост Enterprise-трендов уходит в Tailwind CSS (Utility-First)?
*   **Официальный ответ:**
    > В отличие от BEM (`.card__title`), где стили растут пропорционально новым компонентам, Tailwind использует атомарные классы (`flex pt-4 bg-red-500`). CSS файл приложения перестает расти после достижения определенных размеров (обычно 15кб gzip).
*   **🗣️ Pitch (Senior):**
    > *"При масштабировании BEM/SCSS на 100 фронтендеров возникает феномен 'Append-Only CSS'. Никто не рискует удалить класс `.old-button`, боясь сломать верстку в соседнем роуте. CSS пухнет до 1 мегабайта. Tailwind — это дизайн-токены (System Design) на уровне синтаксиса. Он решает две главных боли Энтерпрайза: 1) Безопасное удаление: удалив кусок JSX, вы 100% удалили весь связанный с ним CSS; 2) Ограниченный размер бандла: все 100 страниц банка переиспользуют один и тот же предсказуемый набор токенов `p-4 flex`, и PurgeCSS/jit-компилятор выстригает мертвые стили, отдавая гарантированные 15kb CSS-бандла навсегда."*

### ❓ 8. Блочный контекст форматирования (BFC - Block Formatting Context): Зачем он нужен?
*   **Официальный ответ:**
    > BFC — это изолированная область на странице, внутри которой элементы раскладываются по правилам блоков, и их геометрия НЕ влияет на внешние элементы. Создается (триггерится) часто через `overflow: hidden`, `display: flex/grid` или `float`.
*   **🗣️ Pitch (Senior):**
    > *"BFC — это CSS-инкапсуляция до эры Shadow DOM. Если у вас классический баг 'соприкосновение марджинов' (Margin Collapsing), когда margin-top `<h1>` вываливается из родительского `<div>` и толкает саму обертку вниз — значит, у родителя не создан BFC. Самый легальный хак создать BFC без сайд-эффектов сегодня — это `display: flow-root`. Это изолирует внутреннюю математику отступов компонента от всей внешней страницы."*

### ❓ 9. Stacking Context (Контекст наложения) и "Z-index война" (z-index: 9999).
*   **Официальный ответ:**
    > `z-index` работает не глобально для всей страницы. Он работает только внутри ВАШЕГО локального Stacking Context. Если предку назначен `z-index: 1`, его ребенок с `z-index: 9999` ВСЕ РАВНО окажется ПОД элементом соседа, у которого `z-index: 2`.
*   **🗣️ Pitch (Senior):**
    > *"Джуны решают проблемы модалок через добавление девяток `z-index: 99999999`. Это архитектурный мусор. Stacking Context создается невидимо: свойством `opacity: 0.99`, или `transform: translateZ(0)`, или `position: relative + z-index`. В дизайн-системе (UI Kit) Сбера мы жестко запретили хардкодить `z-indices` в компонентах. У нас есть централизованный `z-index.scss` маппинг (Dropdown=100, Modal=200, Tooltip=300). А сами React-Модалки мы физически выносим из иерархии в самый корень `<body>` при помощи **React Portals**, гарантируя, что никакие случайные Stacking Contexts родителей (с `transform` анимациями) не смогут 'перекрыть' нашу модалку сверху."*

## 🟢 РАЗДЕЛ 3: Modern CSS Features (Новые парадигмы CSS 2023-2024)

### ❓ 10. Container Queries (`@container`) vs Media Queries (`@media`).
*   **Официальный ответ:**
    > Media Queries меняют стили карточки в зависимости от ширины всего viewport'a дисплея (`@media (max-width: 768px)`).
    > Container Queries меняют стили карточки в зависимости от физической ширины её РОДИТЕЛЬСКОГО БЛОКА в DOM-дереве (`@container (max-width: 400px)`). 
*   **🗣️ Pitch (Senior):**
    > *"Container Queries — это святой грааль Component-Driven Development (React/Vue). До их появления компонент `<ProductCard>` был глупым — если я вставлял его в узкий Сайдбар на FullDesktop 4k мониторе, он ломал верстку, т.к. ширина экрана была 4k (`@media` молчал), а ширина самого сайдбара — всего 300px. Теперь я оборачиваю родителя в `container-type: inline-size`. И моя React-карточка сама понимает: 'Ого, меня засунули в блок 300px, я включу мобильную вертикальную раскладку'. Компонент становится на 100% модульным и отвязанным от Layout'a страницы."*

### ❓ 11. Реляционный Псевдокласс `:has()`. Что он ломает и что строит?
*   **Официальный ответ:**
    > Функция "Родительского селектора". Вы можете обратиться к Родителю, если у него есть определенный Ребенок. Пример: `form:has(input:invalid)` (сделать всю форму красной, если внутри есть хоть один инвалидный инпут).
*   **🗣️ Pitch (Senior):**
    > *":has() навсегда убил огромный пласт JS-кода в React. Раньше, чтобы изменить цвет шапки сайта, когда внизу экрана открыто меню, нам приходилось пробрасывать React State (isMenuOpen) через Redux/Zustand на самый верх DOM-дерева до Header-a. Сейчас я пишу `body:has(#mobile-menu.open) .header { color: black }`. Это декларативно, выполняется движком C++ и очищает бизнес-логику фронтенда от визуальных костылей. Поддержка браузеров (Baseline 2023) 93%+, он готов к энтерпрайз продакшену."*

### ❓ 12. CSS Grid Layer `subgrid` — Зачем оно нужно, если есть flexbox?
*   **Официальный ответ:**
    > Позволяет вложенному грид-ребенку перенять (наследовать) разметку колонок своего родительского грида.
*   **🗣️ Pitch (Senior):**
    > *"Главная боль E-commerce карточек (Тарифов/Продуктов): когда вы ставите 3 карточки товара в ряд, у одной карточки Название в 3 строчки, а у другой в 1. Кнопки 'Купить' скачут по высоте вразнобой. На флексбоксах это фиксится джаваскриптом (высчитывая самую высокую карточку) или хардкодом `min-height`. С `subgrid` я просто делаю родительскую сетку рядов, и каждая карточка выравнивает свои внутренние Title/Price/Button по ЕДИНОЙ глобальной 'колее' всего списка товаров. Идеальная визуальная гармония."*

### ❓ 13. Новые единицы Viewport'a для мобилок: `dvh`, `lvh`, `svh` вместо `100vh`.
*   **Официальный ответ:**
    > В iOS Safari и Android Chrome `100vh` учитывает полное окно, "скрывая" кусок дизайна под выезжающей снизу адресной строкой (URL Bar). Новые единицы решают эту боль браузеров: `svh` (Short - без бара), `lvh` (Large - полный экран), `dvh` (Dynamic - адаптируется при скролле).
*   **🗣️ Pitch (Senior):**
    > *"Любой WebApp в полноэкранной модалке (шторка снизу в финтех-приложении) должен быть идеальным по высоте. Если мы используем старый `height: 100vh; overflow: auto;` на айфоне, наша главная кнопка оплаты 'Спрячется' под панелью закладок Safari. Пользователь физически не сможет нажать 'Купить'. Мой стандарт в Global CSS Reset сегодня: повсеместно менять `100vh` на адаптивный `100dvh` (Dynamic Viewport Height). Браузер сам 'сдвинет' наш UI-слой вверх, когда юзер тапает пальцем и панель Safari выезжает снизу."*

### ❓ 14. CSS Variables (Custom Properties) vs SCSS Variables. Производительность в JS.
*   **Официальный ответ:**
    > SCSS переменные (`$color: red`) компилируются в статику на сервере. CSS Variables (`--color: red`) живут в браузере (Runtime), к ним есть нативный доступ через JS, и они учитывают каскад DOM-узлов (можно переопределить только для одного div).
*   **🗣️ Pitch (Senior):**
    > *"SCSS мертв. Создание сложной UI/Дизайн Системы (UI Kit) для банка с поддержкой Dark Mode / Light Mode раньше делали через переключение CSS-классов и дублирование тем в бандле. Сейчас мы определяем глобальный рут: `:root { --bg: #fff; }` и `:root[data-theme='dark'] { --bg: #000; }`. Магия CSS переменных в том, что мы можем менять Токены Дизайна "на лету" прямо из JavaScript (`element.style.setProperty('--bg', newColor)`), например, когда юзер загрузил кастомную обоину премиум-канала. Браузер сам перекрасит весь фронтенд без перезагрузки JS-дерева React Components."*

## 🟢 РАЗДЕЛ 4: HTML API, Semantic & Accessibility (a11y)

### ❓ 15. Семантика HTML5 и SEO: Насколько это реально важно? (Mythbusting)
*   **Официальный ответ:**
    > `<header>`, `<main>`, `<article>`, `<nav>` вместо бесчисленных `<div>`. Гугл любит семантику.
*   **🗣️ Pitch (Senior):**
    > *"Открою SEO-правду: самому алгоритму Google Search (Crawler) глубоко плевать, лежит ваш текст в `<article>` или в `<div class="article">`. Он парсит текст. Но кому НЕ плевать? Apple Safari Reader Mode и Screen Readers (технологии для слепых). Навигация скрин-ридера жестко застрянет, если кнопка оплаты сделана через `<div onClick>` вместо нативного `<button>`. И главное преимущество семантики — Developer Experience (DX). Читать JSX код из кучи вложенных дивов — больно. `<main>` сразу маркирует смысловое ядро."*

### ❓ 16. Accessibility (a11y) & WAI-ARIA. Жизненный цикл `aria-live` в React/Next.js SPA.
*   **Официальный ответ:**
    > За атрибутами ARIA скрывается управление поведением скрин-ридеров (VoiceOver, NVDA) для незрячих пользователей. `aria-live` заставляет ридер ПРОПИЩАТЬ текст, даже если фокус не на нем (например, "Оплата успешна").
*   **🗣️ Pitch (Senior):**
    > *"В мире SPA (Single Page Applications, включая App Router) возникает феномен 'Безмолвного перехода'. Юзер нажимает 'Отправить перевод'. На экране крутится лоадер, а потом появляется Зеленая галочка. Зрячий юзер всё видит. Незрячий (на скрин-ридере) — слышит АБСОЛЮТНУЮ ТИШИНУ. В фокусном DOM-е ничего не изменилось. По законам ЕС и США (ADA Compliance) — это судебный иск. Я всегда проектирую 'Aria Announcer' — скрытый `<div aria-live="polite" class="sr-only">`. Когда React State переходит в 'success', мы пушим туда строку 'Перевод 100 рублей отправлен'. Скрин-ридер перехватывает это изменение DOM и зачитывает глухому юзеру в наушники."*

### ❓ 17. Скрипты: Разница `<script async>`, `<script defer>` и `type="module"`.
*   **Официальный ответ:**
    > Если просто `<script>`: Браузер бросает парсить HTML, качает скрипт, выполняет его, и только потом продолжает парсить.
    > `async`: Качает скрипт параллельно в фоне. Как только скачал — мгновенно бросает парсить HTML и выполняет JS. (Порядок не гарантирован).
    > `defer`: Качает параллельно в фоне. Но ВЫПОЛНЯЕТ скрипты только после того, как весь HTML полностью распарсен (Strict order).
*   **🗣️ Pitch (Senior):**
    > *"Для сторонних виджетов-паразитов (Google Tag Manager, Яндекс.Метрика) — если они не требуют строгой последовательности, я использую `async` (или Next.js `next/script` с `strategy="lazyOnload"`). А вот наш главный, толстый Frontend React-бандл (Chunks) мы ВСЕГДА кладем в `head` с атрибутом `defer`. Это гарантирует, что браузер начнет скачивать 2 мегабайта JS как можно раньше, и при этом ни на секунду не заблокирует первичный парсинг DOM, формируя сверхбыстрый FCP (First Contentful Paint)."*

### ❓ 18. Shadow DOM vs Light DOM: Зачем React'у понадобились Web Components?
*   **Официальный ответ:**
    > Технология инкапсуляции. Shadow DOM позволяет создать изолированное дерево, чьи CSS стили и JS события НИКАК не протекают (и не подвергаются влиянию) глобальной страницы наружу.
*   **🗣️ Pitch (Senior):**
    > *"Представьте, что мы разрабатываем микрофронтенд — Финансовый виджет (кнопку 'Оплатить частями Сплит'), который интегрируется в сайты партнеров (МВидео, Ozon). Если партнер использует старый Bootstrap или кривой глобальный `reset.css (* { box-sizing: content-box })`, наш нежный React-UI мгновенно 'разъедется'. Единственный 100% пуленепробиваемый паттерн архитектуры в B2B — обернуть наш React-рендер в `<custom-element>` с режимом Shadow DOM `mode: 'closed'`. Внутри мы кладем инкапсулированный Tailwind CSS. Это создает 'железный купол'. Стили Озона никогда не покрасят нашу кнопку."*

### ❓ 19. Intersection Observer API: Почему `window.addEventListener('scroll')` это преступление?
*   **Официальный ответ:**
    > Событие `scroll` стреляет тысячи раз в секунду. Проверка `div.getBoundingClientRect()` внутри этого хука вызывает жесточайший Layout Thrashing на CPU. Intersection Observer — это нативный (C++) API браузера, который сам אсинхронно(в фоне) высчитывает перекрытия элементов друг с другом.
*   **🗣️ Pitch (Senior):**
    > *"Любой Infinite Scroll (Бесконечная лента транзакций) или Ленивую загрузку(Lazy Image Load) я строго базирую на `IntersectionObserver`. В React это оборачивается в хук `useIntersection()`. Я ставлю невидимый DOM-триггер `<div id="load-more">` в самом низу списка. Когда этот div пересекает границу экрана (threshold: 0.1), Browser GPU посылает один четкий асинхронный event. Моя бизнес логика делает fetch() следующей страницы. Принцип 'Zero Main-Thread overhead' (Ни байта нагрузки на основной поток JavaScript)."*

### ❓ 20. Логические CSS-свойства (`padding-inline`, `margin-block`). Конец LTR/RTL эпохи.
*   **Официальный ответ:**
    > `margin-left` всегда отталкивает слева. `margin-inline-start` отталкивает со стороны 'Начала чтения'. Для нас — слева. Для арабского сайта (RTL) — справа.
*   **🗣️ Pitch (Senior):**
    > *"Если ваш банк решит выходить на рынок ОАЭ (Дубай), и ваш UI-Kit построен на `padding-left: 20px` — вам придется переписывать физически тысячи строк CSS, добавляя инверсионные классы `body[dir='rtl'] .card { padding-left: 0; padding-right: 20px }`. Это архитектурный мусор. Мой стандарт для любых глобальных проектов: я отказываюсь от физических координат (left, right, top, bottom). Я использую только логические (inline-start, inline-end, block-start). Браузер 'перевернет' и отзеркалит дизайн-систему на 180 градусов бесплатно одной строчкой `dir="rtl"` в теге `<html>`."*

## 🟢 РАЗДЕЛ 5: UX & Продвинутые Трюки (Safari, UX Optimization)

### ❓ 21. Сенсорная боль (Mobile UX): 300ms Click Delay и `touch-action`
*   **Официальный ответ:**
    > Мобильные браузеры (раньше) ждали 300мс после тапа пользователя для проверки, не делает ли он 'Double Tap to Zoom'.
*   **🗣️ Pitch (Senior):**
    > *"Сегодня задержка автоматически отключена при внедрении мета-тега `<meta name="viewport" content="width=device-width, initial-scale=1">`. Но я иду глубже в UX 'Нативного приложения' (PWA). Если у меня есть карта (Bank ATM Map) или слайдер Карусели Stories (как в Инстаграм), где пользователь активно водит пальцами, я жестко вешаю CSS `touch-action: pan-x`. Это отключает дефолтное поведение браузера (попытки скроллить вниз и 'bounce' эффект резиновой тяги на iOS), заставляя свайпер ощущаться 1:1 как нативное iOS Swift приложение."*

### ❓ 22. Overscroll Behavior: "Прокрутка за край" (Scroll Chaining / Bounce)
*   **Официальный ответ:**
    > Когда вы скроллите до конца модального окна внутри, браузер начинает скроллить саму веб-страницу (под модалкой). `overscroll-behavior: contain` блокирует эту передачу события "назад" (Scroll Chaining).
*   **🗣️ Pitch (Senior):**
    > *"Проблема модалок (Bottom Sheet / Drawer 'Шторки' мобильника) — это физическая боль веб-разработки. Когда список дропдауна кончился, и юзер продолжает тянуть палец вниз, на айфоне начинает тянуться задний фон сайта! Ощущение 'дешевизны' фейк-мобильного веба гарантировано. Я исправляю это, оборачивая скроллируемый слой в `overscroll-behavior: none`. Скролл "глотается" и мертвеет ровно в границах шторки. Это микро-Interaction-Architecture, которая делает разницу между Junior и Staff уровнем UX."*

### ❓ 23. Z-Index Conflict & `isolation: isolate`
*   **Официальный ответ:**
    > Моментально форсирует создание нового Stacking Context (Контекста наложения).
*   **🗣️ Pitch (Senior):**
    > *"Свойство-шедевр, о котором почти никто не знает. Если у вас есть компонент 'Блок Тарифов', внутри которого сложные анимации и внутренние карточки с кучей `z-index: 10`. И вдруг эти `10` начинают 'наползать' и пробивать шапку сайта (у которой `z-index: 5`) — вам не нужно повышать индексы до `9999`. Достаточно повесить на саму 'Обертку' тарифа свойство `isolation: isolate`. Браузер запакует весь их внутренний Z-мир в черный ящик. Максимум, на что они будут наслаиваться — это внутри собственной коробки."*

### ❓ 24. `user-select` и `-webkit-tap-highlight-color` (Mobile UI Polish)
*   **Официальный ответ:**
    > Свойства для полировки UI на мобильных телефонах. Убирают серые квадраты клика и 'Выделение текста синим' при долгом тапе.
*   **🗣️ Pitch (Senior):**
    > *"Добавьте банковское приложение на домашний экран iOS (через PWA / Add to Home Screen). Когда клиент нажимает кнопку банкомата, он видит вспышку мерзкого серого квадрата (`tap-highlight-color`). Если он случайно задержал палец на иконке - поплыло синее выделение текста курсорами с лупой (Text Selection). Это мгновенно убивает 100% иллюзию 'Настоящего нативного продукта'. На корневом `body` мы глобально делаем `user-select: none; -webkit-tap-highlight-color: transparent`, возвращая выделение текста только `user-select: text` для инпутов форм и длинных статей FAQ лонридов."*

### ❓ 25. Modern Color Spaces (`oklch()`) vs HEX / RGB.
*   **Официальный ответ:**
    > Новые цветовые пространства, которые работают не в машинных кривых (RGB пиксели монитора), а в координатах восприятия Человеческого Глаза.
*   **🗣️ Pitch (Senior):**
    > *"Организация Темной Темы (Dark Mode) и Дизайн Системы (UI Kit) — это ад, если вы используете HEX (`#FF0000`) или RGB. Представьте: дизайнер говорит сделать Hover эффект кнопки 'чуть светлее или чуть холоднее'. В HEX вы не можете это описать формулой. `oklch(Lightness Chroma Hue)` меняет всё. Я могу динамически в JavaScript и CSS генерировать градиенты 100 оттенков: `oklch(calc(var(--base-light) + 10%) var(--c) var(--h))`. Это позволяет создавать предсказуемые 'пастельные' палитры или яркие Error-стэйты просто манипулируя одной переменной 'h' (цветовой тон), без грязно-серых промежуточных пикселей (Color Dead Zones)."*

## 🟡 РАЗДЕЛ 3: База HTML/CSS (Middle Level Core)

### ❓ 26. В чем разница Flexbox и CSS Grid? Когда что применять?
*   **Официальный ответ:**
    > Flexbox — одномерная система (выстраивает элементы в 1 строку или 1 столбец). CSS Grid — двумерная система (строит элементы в пересечении 'Ширина x Высота' клеточек).
*   **🗣️ Pitch (Middle):**
    > *"Если мне нужно отцентрировать иконку внутри кнопки или раскидать ссылки в Header (лево/право), `display: flex` — идеальный инструмент выравнивания по главной оси. Но если я строю Сетку Карточек Товара (Галерею), где элементы переносятся на новую строку — Flexbox ломается (последняя строка 'разъезжается'). Здесь Grid (`grid-template-columns: repeat(auto-fill, minmax(200px, 1fr))`) выдаст математически ровную 2D-сетку вообще без Media Queries."*

### ❓ 27. CSS Box Model и свойство `box-sizing: border-box`. Зачем?
*   **Официальный ответ:**
    > Box Model состоит из Content, Padding, Border и Margin. По умолчанию ширина элемента (width) учитывает только Content. `border-box` заставляет ширину 'впитывать' Padding и Border внутрь себя.
*   **🗣️ Pitch (Middle):**
    > *"Это фундамент любой верстки (CSS-Reset). Без `box-sizing: border-box`, инпут с 100% шириной и `padding: 10px` просто вывалится за пределы родительского контейнера, сломав мобильную верстку и вызвав горизонтальный скролл. Это директива, которая спасает расчеты Layout."*

### ❓ 28. CSS Specificity (Специфичность селекторов) и `!important`.
*   **Официальный ответ:**
    > Вес селектора: ID (1,0,0) перебивает Классы (0,1,0), Класс перебивает Тег (0,0,1). `!important` ломает каскад, перебивая любой вес (кроме инлайн CSS).
*   **🗣️ Pitch (Middle):**
    > *"Использование `!important` или `#id` для стилизации — это 100% красный флаг. В больших проектах мы обязаны держать специфичность плоской (весом 0,1,0) через классы (BEM / Tailwind / Modules). Если вы вставили `!important`, следующему разработчику придется написать еще один, чтобы перебить ваш стиль, и так до бесконечности."*

### ❓ 29. Зачем нужны Семантические теги `<article>`, `<section>`, `<nav>` вместо `<div>`?
*   **Официальный ответ:**
    > Для Доступности (Accessibility a11y) Screen Reader-ов скрытого текста и для SEO оптимизации ПС (Поисковиками).
*   **🗣️ Pitch (Middle):**
    > *"Браузеру без разницы, `div` это или `header` - они оба блочные (display: block). Но Слепой пользователь (Screen Reader VoiceOver) не может найти 'Меню', если оно завернуто в `div.navbar-flex`. Семантика `<nav>` создает ARIA Landmark ('Ориентир Автоматом'), и юзер может мгновенно перепрыгнуть к навигации клавишами. Также Crawler Google понимает, что внутри `<main>` лежит мясо статьи, а `<aside>` можно проигнорировать."*

### ❓ 30. Разница между `pseudo-classes` (:hover) и `pseudo-elements` (::before). Зачем `content: ""`?
*   **Официальный ответ:**
    > Псевдоклассы описывают **состояние** элемента (навели мышку, активен). Псевдоэлементы создают **новый невидимый (виртуальный) элемент** в DOM'е (до или после контента).
*   **🗣️ Pitch (Middle):**
    > *"Псевдоэлементы невероятно спасают от загрязнения HTML. Если мне нужна декоративная полоска (Border) или кавычки у Заголовка, я не буду писать лишний `<span>` в React, дергая V-DOM. Я рисую это в CSS через `::before`. Главное правило — псевдоэлемент никогда не отрисуется на экране без свойства `content: ""`."*
