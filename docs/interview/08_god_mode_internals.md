## 🚀 1. Низкоуровневые оптимизации V8 (JIT-компиляция)

### ❓ В: "Что такое Hidden Classes (Скрытые классы/Shapes) в V8 и как они влияют на производительность?"
*   **Официальный ответ:**
    > V8 компилирует JS в машинный код. Поскольку JS динамический, V8 создает **Скрытые классы** (Shapes) под капотом для каждого объекта. Они содержат смещения (`offset`) свойств в памяти.
*   **🗣️ Pitch:**
    > *"Если мы создаем объекты с одинаковыми свойствами, но в **разном порядке** (например, `{a:1, b:2}` и `{b:2, a:1}`), V8 создаст **разные** скрытые классы. Это ломает **Inline Caching** (кэширование доступа). 
    > **Вывод:** Всегда инициализируйте свойства объекта в одном порядке и в конструкторе. Это позволяет V8 читать свойства по фиксированному адресу памяти за один такт процессора."*

## 🧠 2. WebAssembly (WASM) в фронтенде

### ❓ В: "Когда оправдано использование WebAssembly в React приложении?"
*   **Официальный ответ:**
    > WASM работает на бинарном уровне, без сборщика мусора (GC). Он НЕ имеет прямого доступа к DOM, поэтому для рендеринга бесполезен.
*   **🗣️ Pitch:**
    > *"WASM нужен там, где JS захлебывается в вычислениях. Это **Парсинг тяжелых файлов** (например, Figma парсит свои файлы до старта), **Криптография** (шифрование ключей в браузере), **Обработка видео/аудио** в реальном времени. В банке это может быть валидация гигантских реестров платежей на клиенте до отправки на сервер."*

## 🌊 3. Гравитация Concurrent React (React 18)

### ❓ В: "Что такое Selective Hydration (Селективная гидратация) и как её готовить?"
*   **Официальный ответ:**
    > Раньше гидратация была монолитной (всё или ничего). В React 18 благодаря `<Suspense>` мы можем гидратировать приложение кусками.
*   **🗣️ Pitch:**
    > *"Если у нас страница состоит из Шапки и тяжелых Комментариев (подгружаемых через `lazy`), `<Suspense>` позволяет пользователю **уже кликать** по шапке, пока комментарии еще качают JS-бандл. Если пользователь кликнет по комментариям раньше времени, React приоритезирует (Lanes) гидратацию именно этого куска. Это концепт **User-driven Priority**."*

## 🔄 4. Реалтайм без конфликтов (Offline-first)

### ❓ В: "Как спроектировать совместное редактирование данных (как в Google Docs/Figma) без конфликтов слияния?"
*   **Официальный подход:** Использование структуры данных **CRDT** (Conflict-Free Replicated Data Types).
*   **🗣️ Pitch:**
    > *"Standard REST/Websocket API сталкивается с перезаписью данных, если два пользователя вводят текст одновременно. Я бы использовал алгоритмы **CRDT** (например, библиотеки `Yjs` или `Automerge`). Они математически гарантируют, что состояние локальных реплик сойдется к одному результату без центрального сервера авторизации конфликтов. Это мастхэв для сложных Offline-first дашбордов."*

## ⚡ 5. Resumability вместо Hydration (Граница будущего)

### ❓ В: "Чем подход Qwik (Resumability) отличается от классического React Next.js?"
*   **Официальный ответ:**
    > Гидратация скачивает JS и запускает все функции компонентов заново, чтобы построить Virtual DOM.
*   **🗣️ Pitch:**
    > *"Qwik делает **Resumability** (возобновление). Приложение на сервере замирает в памяти в определенном состоянии и сериализует все реактивные связи прямо в HTML. Браузер получает страницу, и её вес в JS равен **0 КБ** при старте. JS подгружается лениво **только в момент клика** по конкретной функции. Это решает проблему гидратации на мобильных устройствах."*

## 📡 6. Протоколы: HTTP/3 (QUIC) и gRPC

### ❓ В: "Зачем современному фронтенд-инженеру знать про HTTP/3?"
*   **Официальный ответ:**
    > HTTP/3 работает поверх **UDP** (протокол QUIC), избавляясь от проблемы блокировки начала очереди (Head-of-Line Blocking) в TCP.
*   **🗣️ Pitch:**
    > *"В банковском софте на плохом 3G-соединении TCP часто теряет пакеты и вешает весь канал. С HTTP/3 потеря одного пакета (например, картинки профиля) больше НЕ блокирует загрузку других пакетов (например, баланса). Это резко повышает отзывчивость мобильных веб-интерфейсов."*
