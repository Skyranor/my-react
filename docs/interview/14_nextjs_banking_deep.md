## 🔐 Раздел 1: Безопасность и Авторизация (1-10)

### ❓ 1. Как защитить Server Actions от CSRF-атак?
**Детальный ответ:** Next.js имеет встроенную защиту для Server Actions (Host/Origin check). Однако в банке мы обязаны делать **Runtime Validation** всех входных данных через Zod, чтобы исключить попытки инъекций и манипуляций.
**Пример:**
```ts
'use server';
import { z } from 'zod'; // Стандарт 2026 для валидации типов

const TransferSchema = z.object({
  amount: z.number().positive().max(1000000), // Лимит одной операции
  toAccount: z.string().regex(/^[0-9]{20}$/), // Формат банковского счета
});

export async function secureTransfer(formData: FormData) {
  // 1. Парсим и валидируем данные на сервере
  const result = TransferSchema.safeParse(Object.fromEntries(formData));
  if (!result.success) return { error: 'Некорректные реквизиты' };

  // 2. Обязательная проверка сессии внутри Action
  const session = await getSession(); 
  if (!session || !session.isMfaVerified) throw new Error('MFA required');
}
```
**🗣️ Pitch:** *"Мы внедряем строгую валидацию Zod для каждого действия. Это гарантирует, что невалидные данные не попадут в бизнес-логику, а двойная проверка сессии (MFA) внутри экшена защищает транзакции даже при компрометации клиентского стейта."*

### ❓ 2. Как хранить JWT в Next.js максимально безопасно?
**Детальный ответ:** В финтехе JWT-токены должны жить только в **httpOnly, Secure, SameSite=Strict** куках. Это делает их недоступными для `document.cookie`, что нейтрализует XSS-атаки на кражу сессии.
**Пример:**
```ts
// В Server Action при логине или обновлении токена:
cookies().set('session_token', token, {
  httpOnly: true, // Запрет доступа из JS
  secure: true,   // Передача только по HTTPS
  sameSite: 'strict', // Защита от CSRF на уровне браузера
  maxAge: 3600,   // Время жизни сессии - 1 час
  path: '/',
});
```
**🗣️ Pitch:** *"Для Альфа-Банка мы полностью исключили хранение токенов в localStorage. Использование httpOnly кук — это стандарт де-факто для защиты от кражи сессий. Мы также используем SameSite=Strict, чтобы кука не улетала при переходах со сторонних ресурсов."*

### ❓ 3. Настройка CSP (Content Security Policy) через Middleware.
**Детальный ответ:** CSP ограничивает источники скриптов и данных. В банке мы используем `nonce` для каждого запроса, чтобы разрешить только доверенные скрипты.
**Пример:**
```ts
export function middleware(request: NextRequest) {
  const nonce = Buffer.from(crypto.randomUUID()).toString('base64');
  // Разрешаем скрипты только с нашего домена и с правильным nonce
  const cspHeader = `script-src 'self' 'nonce-${nonce}' 'strict-dynamic'; object-src 'none';`;
  
  const response = NextResponse.next();
  response.headers.set('x-nonce', nonce);
  response.headers.set('Content-Security-Policy', cspHeader);
  return response;
}
```
**🗣️ Pitch:** *"CSP с механизмом 'nonce' — это наш 'бронежилет' против XSS. Даже если злоумышленник внедрит `<script>`, браузер его заблокирует, так как у него не будет динамического ключа, сгенерированного сервером для этой конкретной сессии."*

### ❓ 4. Logout во всех вкладках сразу. Как реализовать?
**Детальный ответ:** При логауте мы удаляем куку и вызываем `revalidatePath('/')`. Next.js 16 автоматически обновляет кэш клиентского роутера, и при любом переходе юзер мгновенно увидит экран логина.
**Пример:**
```ts
export async function logout() {
  cookies().delete('session_token'); // Удаляем сессию
  revalidatePath('/'); // Сбрасываем кэш всех страниц
  redirect('/login');  // Перенаправляем
}
```
**🗣️ Pitch:** *"В банковском приложении логаут должен быть мгновенным и тотальным. Мы используем `revalidatePath`, чтобы гарантировать: ни один байт закешированных данных пользователя не останется доступным после завершения сессии."*

### ❓ 5. Скрытие чувствительных данных из RSC Payload.
**Детальный ответ:** RSC Payload летит в браузер как текст. Если передать туда весь объект из базы, юзер увидит скрытые поля в Network tab. Мы используем **DTO (Data Transfer Objects)**.
**Пример:**
```ts
// Серверный компонент
export default async function AccountInfo() {
  const fullAccount = await db.accounts.findUnique({ id: 1 });
  
  // НЕЛЬЗЯ: return <ClientAccount account={fullAccount} />
  
  const safeAccount = { // DTO
    id: fullAccount.id,
    balance: fullAccount.balance, // Только то, что нужно UI
    maskedNumber: maskCard(fullAccount.number)
  };
  
  return <ClientAccount account={safeAccount} />;
}
```
**🗣️ Pitch:** *"Мы строго соблюдаем принцип минимизации данных. Паттерн DTO в Server Components гарантирует, что в браузер попадает только публичная информация, исключая утечку метаданных или внутренних ID базы данных."*

### ❓ 6. Шифрование стейта внутри Cookies.
**Детальный ответ:** Иногда нужно хранить временные данные (например, шаги оформления кредита) в куках. В банке такие данные должны быть зашифрованы на стороне сервера (JWE).
**Пример:**
```ts
import { encrypt, decrypt } from '@/lib/crypto'; // Наша обертка над Web Crypto API

export async function saveStep(data: object) {
  const encrypted = await encrypt(JSON.stringify(data));
  cookies().set('draft_loan', encrypted, { httpOnly: true });
}
```
**🗣️ Pitch:** *"Если нам необходимо хранить временный стейт на клиенте, мы используем симметричное шифрование AES-256. Юзер видит в куках шум, а сервер может восстановить контекст без обращения к БД."*

### ❓ 7. Почему нельзя делать прямые запросы к БД из клиента?
**Детальный ответ:** Это дыра в безопасности (утечка креденшалов БД) и невозможность контролировать права доступа на уровне строк.
**🗣️ Pitch:** *"Это архитектурная аксиома. Весь доступ к данным — только через Server Components или API-слой. Это позволяет нам централизованно аудировать запросы и применять политики RBAC (Role-Based Access Control) на стороне сервера."*

### ❓ 8. Enumeration Attack и динамические роуты `[id]`.
**Детальный ответ:** Если у нас есть `/tx/[id]`, атакующий может перебирать ID.
**Решение:** Использовать UUID + проверку владения (Ownership check).
**Пример:**
```ts
export default async function Transaction({ params }) {
  const tx = await getTx(params.id);
  const session = await getSession();
  
  if (!tx || tx.userId !== session.userId) {
    notFound(); // Возвращаем 404 вместо 403, чтобы не подтверждать существование чужого ID
  }
}
```
**🗣️ Pitch:** *"Для защиты от перебора мы используем UUID v4 и всегда отдаем `notFound()` при попытке доступа к чужому ресурсу. Атакующий не должен знать, существует ли ID в нашей системе или нет."*

### ❓ 9. Роль `headers()` и `cookies()` в динамике.
**Детальный ответ:** Эти функции делают рендеринг динамическим (нельзя собрать в статику при билде).
**🗣️ Pitch:** *"Мы осознанно управляем динамическим рендерингом. Там, где нужны заголовки авторизации, мы используем `headers()`, но оборачиваем такие блоки в `<Suspense>`, чтобы сохранить преимущества Partial Prerendering для остальной части страницы."*

### ❓ 10. Безопасная работа с API ключами (LLM/Провайдеры).
**Детальный ответ:** Использование переменных окружения БЕЗ префикса `NEXT_PUBLIC_`.
**🗣️ Pitch:** *"В Альфа-Банке мы используем строгую конвенцию: ключи сторонних API (например, OpenAI или Скоринг-систем) доступны только Node.js среде. Это исключает их попадание в клиентский JS-бандл даже случайно."*

## ⚡ Раздел 2: Производительность и UX (11-20)

### ❓ 11. Оптимизация LCP для банковских дашбордов.
**Детальный ответ:** LCP — главная метрика скорости. Мы используем **PPR (Partial Prerendering)** для мгновенного показа "шапки" и меню.
**Пример:**
```tsx
// layout.tsx
export default function DashboardLayout({ children }) {
  return (
    <>
      <StaticSidebar /> {/* Рендерится мгновенно из кэша CDN */}
      <main>
        <Suspense fallback={<DashboardSkeleton />}>
          {children} {/* Динамический контент со списком счетов */}
        </Suspense>
      </main>
    </>
  );
}
```
**🗣️ Pitch:** *"Наша цель — LCP менее 1.2с. Мы разделяем интерфейс на статическую оболочку и динамические слоты. Пользователь видит структуру приложения мгновенно, а данные подгружаются стримингом по мере готовности бэкенда."*

---

### ❓ 12. `useOptimistic` в банковских операциях.
**Детальный ответ:** Позволяет мгновенно обновить UI, пока идет запрос на перевод.
**Пример:**
```tsx
const [optimisticBalance, setOptimistic] = useOptimistic(
  currentBalance,
  (state, amount) => state - amount
);

const handleTransfer = async (formData) => {
  setOptimistic(1000); // UI сразу вычитает 1000 руб
  await performTransfer(formData); // Реальный запрос
};
```
**🗣️ Pitch:** *"Мы используем Optimistic UI для критических сценариев, вроде оплаты услуг или перевода. Это снижает 'физическую' когнитивную нагрузку на пользователя, делая интерфейс отзывчивым даже при медленном 3G."*

### ❓ 13. Борьба с CLS (Layout Shift) при загрузке курсов валют.
**Детальный ответ:** CLS — это прыжки контента. В банках это недопустимо (юзер может нажать не ту кнопку).
**🗣️ Pitch:** *"Мы минимизируем CLS за счет использования Aspect Ratio Boxes и скелетонов с фиксированной высотой. Каждая динамическая область имеет зарезервированное место в DOM еще до получения данных."*

### ❓ 14. Prefetching: когда его ОТКЛЮЧАТЬ?
**Детальный ответ:** По умолчанию Next.js предзагружает все ссылки в вьюпорте. В банке, где много данных, это может перегрузить API.
**Пример:** `<Link href="/history" prefetch={false}>История</Link>`
**🗣️ Pitch:** *"Для тяжелых разделов, вроде 'История транзакций за 5 лет', мы отключаем автоматический prefetch. Мы экономим трафик пользователя и ресурсы сервера, загружая данные только при явном намерении перехода."*

---

### ❓ 15. PDF-генераторы/Тяжелые либы и бандл.
**Детальный ответ:** Использование `next/dynamic` для ленивой загрузки тяжелых модулей.
**Пример:**
```ts
const StatementPdf = dynamic(() => import('@/components/PdfViewer'), {
  ssr: false, // Грузим только на клиенте
  loading: () => <Spinner />
});
```
**🗣️ Pitch:** *"Мы в Альфа-Банке боремся за каждый килобайт JS. Тяжелые компоненты, вроде систем генерации выписок или сложных графиков, загружаются асинхронно только в момент их реальной востребованности пользователем."*

### ❓ 16. `priority` в `<Image>` для банковских карт.
**🗣️ Pitch:** *"Для ключевых визуальных элементов, таких как изображение карты или главный баннер акции, мы используем проп `priority`. Это заставляет браузер начать загрузку этого ресурса еще до того, как основной JS будет исполнен."*

### ❓ 17. ISR (`revalidate`) для публичных страниц банка.

**🗣️ Pitch:** *"ISR позволяет нам иметь производительность статики при актуальности данных в 60 секунд. Это снижает нагрузку на API в периоды волатильности рынка, когда тысячи людей обновляют страницу курса одновременно."*

### ❓ 18. Interaction to Next Paint (INP) и сложные таблицы.

### ❓ 19. Разница `loading.tsx` и точечного `<Suspense>`.
**🗣️ Pitch:** *"Глобальный `loading.tsx` хорош для первого перехода, но для банковских дашбордов мы предпочитаем гранулярные `<Suspense>`. Пользователь должен видеть 'скелет' конкретных виджетов, а не один общий лоадер на весь экран."*

## 🏗️ Раздел 3: Архитектура и Надежность (21-30)

### ❓ 21. Реализация Feature Flags в банке.
**Детальный ответ:** Хранение флагов в Edge Config (Vercel) или в куках, проверка в Server Components.
**Пример:**
```ts
const isNewLoanFlow = cookies().get('ff_loan_v2')?.value === 'active';
return isNewLoanFlow ? <LoanFlowV2 /> : <LoanFlowV1 />;
```
**🗣️ Pitch:** *"Feature Flags — основа нашего CI/CD. Мы выкатываем фичи постепенно (Canary releases) и можем мгновенно отключить проблемный функционал на уровне сервера без пересборки приложения."*

### ❓ 22. Обработка 500 ошибок через `error.tsx`.
**Детальный ответ:** Гарантирует, что падение одного виджета не 'убьет' все приложение.
**🗣️ Pitch:** *"У нас реализована система 'отсеков'. Если падает модуль 'Страхование', остальное банковское приложение продолжает работать благодаря локальным границам ошибок `error.tsx`."*
