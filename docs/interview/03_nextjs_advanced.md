# 👑 Next.js 14+ (App Router, RSC) — Senior/Product Engineer

Этот документ содержит глубокие архитектурные вопросы по экосистеме Next.js (с фокусом на App Router, React Server Components и Performance).

## 🟢 РАЗДЕЛ 1: React Server Components (RSC) & Rendering

### ❓ 1. В чем фундаментальная разница между Server Components (RSC) и Client Components?
*   **Официальный ответ:**
    > RSC выполняются ТОЛЬКО на сервере и не отправляют физический JS-код в браузер (0kb bundle). Они могут напрямую ходить в базу данных (`async/await`). Client Components (`"use client"`) рендерятся как на сервере (SSR, HTML-разметка), так и гидратируются на клиенте, поддерживая стейт (`useState`) и браузерные API (`window`).
*   **🗣️ Pitch (Senior):**
    > *"Джуны часто путают SSR и RSC. SSR — это просто предварительная генерация HTML-строки, но код всё равно скачивается в бандле для 'оживления' (Hydration). RSC — это архитектурный сдвиг. Когда я объявляю `<ChartAnalytics />` как серверный компонент, он делает `await db.query()`, пережевывает 5 МБ JSON-данных через `d3-math`, и отдает клиенту лишь легковесную React-структуру (RSC Payload). Это сэкономило нам мегабайты JS-пакета, физически вырезав тяжелые слои из мобильного девайса клиента."*

### ❓ 2. Что такое RSC Payload и как он решает проблему Navigation State?
*   **Официальный ответ:**
    > RSC Payload — это бинарно-текстовое представление дерева React. Оно содержит отрендеренные 'Серверные' элементы и 'Дырки' (Placeholders) для импорта 'Клиентских' модулей.
*   **🗣️ Pitch (Senior):**
    > *"RSC Payload (текст формата `1:I["ClientComponent"...]` в Network) решает главную проблему Router-а. Если бы Next.js возвращал готовый HTML при навигации (NextLink переходах), то клиентский React терял бы весь стейт (например, модалка закрывалась бы, инпут поиска очищался). Под капотом Next.js скачивает RSC-дерево новой страницы и Reconcile-ит (сливает) его с существующим в памяти Client-side деревом, сохраняя состояние React живым. Это фундаментальная магия Next Router-а."*

### ❓ 3. Паттерн 'Interleaving': Можно ли вложить Server Component внутрь Client Component?
*   **Официальный ответ:**
    > Напрямую импортировать RSC в файл `"use client"` **нельзя** (это вызовет ошибку). Но можно передать Server Component как объект `children` (prop) сквозь Client Component.
*   **🗣️ Pitch (Senior):**
    > *"Это самое частое архитектурное заблуждение(Антипаттерн). Вы не можете сделать `import ServerForm` внутри `"use client" Modal`, потому что вебпак утащит серверный SQL код в клиентский бандл. Мой стандартный паттерн дизайна (Composition Inversion): компонент-обертка. Я рендерю это сверху-вниз: `<ClientModal><ServerForm /></ClientModal>`. React на клиенте поймет, что `children` — это 'дырка', которую заполнит RSC Payload с сервера. Граница (Network Boundary) при этом не нарушена."*

### ❓ 4. SSR против SSG. Что такое Partial Prerendering (PPR) в Next.js 14+?
*   **Официальный ответ:**
    > SSR — динамика (рендеринг на запрос). SSG — статика (рендеринг в билдтайме). PPR — экспериментальная смешанная архитектура, объединяющая статический SSG-каркас и динамический SSR-контент на одной странице без компромиссов.
*   **🗣️ Pitch (Senior):**
    > *"До PPR (Partial Prerendering), присутствие функции `cookies()` на странице заставляло весь маршрут откатываться в тяжелый динамический SSR. С PPR, Next.js на этапе билда пререндерит 95% страницы (Хедер, Сайдбар) в статичный быстрый Edge HTML. А оставшиеся 5% (Корзина юзера, завернутая в `<Suspense>`) Next 'добивает' динамическим стримингом в открытое HTTP-соединение сразу после отдачи статической 'оболочки'. Это ультимативное решение для метрики 'Time-To-First-Byte'."*

## 🟢 РАЗДЕЛ 2: Data Fetching, Caching & Mutations

### ❓ 5. Caching Architecture: Разница между Data Cache и Full Route Cache?
*   **Официальный ответ:**
    > `Data Cache` кэширует результаты вызова нативного `fetch` API между серверами по всему миру. `Full Route Cache` кэширует сам отрендеренный RSC Payload и куски HTML всей страницы (аналог SSG).
*   **🗣️ Pitch (Senior):**
    > *"Next.js невероятно агрессивно кэширует всё по умолчанию. Моя задача как архитектора — правильно управлять тегами. Для глобальных словарей я закладываю жесткий `Full Route Cache`. Для микро-дашбордов я использую `Data Cache` с жесткими тэгами: `fetch('/api', { next: { tags: ['balance', 'u123'] } })`. Это позволяет гибко инвалидировать кэш по конкретному webhook-событию, не снося кэш всей остальной страницы."*

### ❓ 6. `revalidatePath` vs `revalidateTag` — как выборочно сбрасывать кеш?
*   **Официальный ответ:**
    > `revalidatePath('/blog')` удаляет из кеша страницу целиком по URL-пути. `revalidateTag('products')` точечно инвалидирует все `fetch`-вызовы в приложении, помеченные тегом 'products', независимо от маршрута.
*   **🗣️ Pitch (Senior):**
    > *"В Enterprise (СберМаркет) `revalidatePath` — это костыль. Представьте, что аватарка юзера рендерится на 15 страницах. Если вы делаете `revalidatePath('/profile')`, аватарка в 'комментариях' останется старой. Я строю Caching Invalidation архитектуру строго поверх `revalidateTag`. Обновив профиль, я дергаю один ТЕГ `user:123`, и Next.js автоматически находит и пере-рендерит (ISR) все закешированные фрагменты на всех маршрутах, где был этот tag."*

### ❓ 7. Дизайн паттерна 'Server Actions': Безопасность и Валидация Мутаций
*   **Официальный ответ:**
    > Server Actions (`"use server"`) позволяют мутировать данные без создания API эндпоинтов, напрямую связывая форму `<form action={Action}>` с серверной функцией.
*   **🗣️ Pitch (Senior):**
    > *"Server Action компилируется в скрытый публичный POST-эндпоинт. Если форма просто делает `await db.users.delete(userId)`, любой хакер снесет базу через Postman! Я всегда требую тройной кордон Security для Actions: 1) Авторизация (сессия куки). 2) Верификация прав (RBAC/Policy) — может ли юзер удалять. 3) Zod/Valibot-схема строгой DTO валидации `FormData`. Только после этого я делаю UPDATE в БД."*

### ❓ 8. Fetch Deduplication (Мемоизация запросов): Как Next решает N+1 в RSC?
*   **Официальный ответ:**
    > React автоматически кэширует инстансы вызовов нативного API `fetch()` с одинаковыми параметрами на протяжении жизни одного Request-цикла.
*   **🗣️ Pitch (Senior):**
    > *"Раньше разработчики пробрасывали `userData` свойствами на 10 уровней вниз дерева компонентов-детей. В Next.js 14 State Management изменился: я пишу функцию `getUser()`, внутри делаю `fetch(db)`. Затем я вызываю её напрямик(без контекста) В КАЖДОМ из 10 глубоких изолированных компонентов. Компилятор React-а на сервере хлопнет эти 10 вызовов в Один-единственный network-запрос к БД(Deduplication). Код становится невероятно чистым."*

## 🟢 РАЗДЕЛ 3: Architecture & Routing

### ❓ 9. Layouts vs Templates — архитектурная разница в маршрутизации
*   **Официальный ответ:**
    > `layout.tsx` обертывает детей и НЕ размонтируется при навигации (не теряет state). `template.tsx` создает НОВЫЙ экземпляр (Re-mount) при переходе между страницами, сбрасывая state и `useEffect`.
*   **🗣️ Pitch (Senior):**
    > *"В 99% случаев я использую `Layout` (для Header и Сбоку-меню), чтобы UI не мерцал как в нативном SPA. Но `Template` незаменим для двух кейсов: 1. CSS-анимации (когда при смене страницы обертка должна красиво выехать снизу заново). 2. Аналитика: если нужно трекать Event-счетчики(Entry-Page), `template.tsx` монтируется заново на каждый клик меню, триггеря DataLayer скрипт."*

### ❓ 10. Intercepting Routes (`(..)` pattern) — Модальные окна-перехватчики
*   **Официальный ответ:**
    > Загружают контент страницы из другой части приложения внутри текущего окна. При клике открывают модалку, но если юзер войдет по URL напрямую — он получит полную страницу.
*   **🗣️ Pitch (Senior):**
    > *"Это Game-changer архитектуры e-commerce (Паттерн: Быстрый просмотр). Если юзер кликает на товар — поверх ленты товаров всплывает Модалка с URL `/product/1`. Но если он скинет эту ссылку в мессенджер, его друг получит Полноценную Standalone страницу (без ленты сзади). Это убивает боль синхронизации URL и Redux стейта модалок. Работает навигация браузера, отлично дружит с SEO краулерами."*

### ❓ 11. Route Handlers (`route.ts`) vs Server Actions — что когда выбрать?
*   **Официальный ответ:**
    > Route Handlers создают RESTful API (GET/POST). Server Actions — выполняют Data Mutations (RPC функции для RSC/Client компонентов).
*   **🗣️ Pitch (Senior):**
    > *"Если логика работает строго для нашего UI (обновить никнейм) — только `Server Actions`. Они бесшовно интегрированы c формами/TS. Но если проекту нужен **Публичный API** (выдать JSON для мобильного iOS-приложения разработчиков), поддержать **Webhooks** (уведомления от системы Stripe с подписью), или вернуть поток `ReadableStream` (LLM OpenAI генерации) — я разворачиваю классический файловый маршрут `route.ts`, контролируя сырой HTTP Request/CORS Headers."*

### ❓ 12. Дизайн Middleware (`middleware.ts`) — Ограничения (Edge)
*   **Официальный ответ:**
    > Middleware выполняется на Edge серверах ДО того, как запрос дойдет до Next.js сервера. Он может редиректить или добавить куки. Не поддерживает Node.js модули (fs, bcrypt, crypto).
*   **🗣️ Pitch (Senior):**
    > *"Edge Runtime — это восьми-битная V8 среда (Cloudflare Workers). Я использую Middleware как Gateway-Filter на самом 'краю интернета'. Паттерны применения: 1) Редирект `next-intl` языков с анализом заголовка Accept-Language, 2) Добавление защитного Security HTTP Headers (Content-Security-Policy nonce) на лету к HTML странице, 3) Легковесная расшифровка JWT токена (jose library) для защиты (ProtectedRoute) дашборд-роутов от гостей. Настоящую 'тяжелую' MySQL логику я всегда спускаю ниже."*

### ❓ 13. Streaming & `<Suspense>` Boundaries (Стриминг UI)
*   **Официальный ответ:**
    > Стриминг — отправка кусков HTML клиенту по мере готовности сервером, заворачивая медленные API части в React `<Suspense fallback={...}>`.
*   **🗣️ Pitch (Senior):**
    > *"Прятать страницу 'Товара' за одним гигантским Loading-экраном (дожидаясь БД отзывов) — преступление против конверсии магазина. Я применяю архитектуру Progressive Rendering: Layout и Title рендерятся сервером мгновенно (100мс). Медленный блок 'Отзывы' декомпозируется и оборачивается в `<Suspense fallback={<Skeletons/>}>`. Сервер отправляет первый HTML(скелеты), а когда MySQL отдает отзывы, сервер в открытое(Keep-Alive/Chunked) HTTP-соединение докидывает `<script>` тег, который мгновенно 'вклеивает' <div> с отзывами вместо скелета. Юзер потребляет контент поэтапно."*

### ❓ 14. Font Optimization in Next 14 (`next/font` модуль)
*   **Официальный ответ:**
    > Загружает файлы Google Fonts (или локальные) на этапе сборки (CI Builder) и кладет их на 1 сервер с вашим приложением, устраняя лишние внешние HTTP-запросы и утечки приватности (IP).
*   **🗣️ Pitch (Senior):**
    > *"Внешние шрифты часто рушат UX прыжками (Flash of Unstyled Text / CLS метрика). Концепция `next/font` на этапе сборки SWC Compiler-а вычисляет шрифт, ставит `size-adjust` CSS свойства и fallback-шрифт. Инжектит сгенерированный preload хэш в `<head>`. В результате: 100% Privacy Privacy Policy compliance (не шлем IP-адреса юзеров серверам Google), и полностью решенная проблема сдвигов верстки (Core Web Vitals спасены)."*

### ❓ 15. Bundle Analysis & Оптимизация пакетов (Tree Shaking)
*   **Официальный ответ:**
    > Осуществляется через `@next/bundle-analyzer` и флаг `optimizePackageImports`.
*   **🗣️ Pitch (Senior):**
    > *"В моем CI пайплайне стоит жесткий Performance Budget — First Load JS не больше 80kb. Чтобы не затащить гигантские UI-иконки целиком в Runtime бандл, я применяю: 1) Флаг `optimizePackageImports` в `next.config.ts`, чтобы Next.js хирургически выкусил только одну импортированную `LucideIcon` из гигантского Barrel-index-файла либы. 2) Динамическая(Lazy) подгрузка. Тяжелый редактор кода (TinyMCE) нужен только при клике на 'Написать комментарий' — я импортирую его через `const Editor = next/dynamic()`, отрезая от критического дерева первой отрисовки."*

## 🟢 РАЗДЕЛ 4: Optimization, Vitals and Security

### ❓ 16. Оптимизация Изображений под капотом: `next/image`
*   **Официальный ответ:**
    > Заменяет нативный тег `<img>`. Автоматически пережимает форматы в легкий WebP/AVIF (на лету), добавляет Lazy-loading (ленивую загрузку) браузера и борется с Cumulative Layout Shift (CLS шлейфом), жестко резервируя размер.
*   **🗣️ Pitch (Senior):**
    > *"Джуны просто кидают фото в `next/image`. Senior разработчик архитектурно таргетит метрику **LCP** (Largest Contentful Paint — главная SEO-метрика Google). Главный banner(картинку) первого экрана я ОБЯЗАН помечать атрибутом `priority` (Next.js вшивает в `<head>` тег `<link rel="preload">`, заставляя Network скачивать картинку задолго до парсинга JS/HTML). Для Remote-картинок (с бэкенда Amazon S3) я внедряю проп `placeholder="blur"` в связке с `blurDataURL` (micro-base64 строчка), чтобы при 3G-интернете интерфейс не дергался(CLS = 0), а изящно блюрился."*

### ❓ 17. SEO & Метаданные (Dynamic Metadata + OpenGraph) в App Router
*   **Официальный ответ:**
    > Важнейшие метаданные (title, description, og:image) для поисковиков и соцсетей. В App Router экспортируется константа `metadata` или асинхронная JS-функция `generateMetadata`.
*   **🗣️ Pitch (Senior):**
    > *"Hardcode-SEO мертв в e-commerce. Мой стандарт — `async generateMetadata`. Но есть подводный камень: чтобы получить Title товара, мы делаем `fetch(product)`. И в самом UI компоненте мы делаем СНОВА ТОТ ЖЕ `fetch(product)`. Разработчики пугаются двойных запросов к БД. Но я знаю секрет лона: Next Fetch Memoization склеит эти 2 вызова в 1 network-вызов. Дополнительно, для 'Вау-шейринга' в Telegram, я поднимаю динамический сервис генератора картинок `@vercel/og` (React-разметка конвертируется в SVG/PNG картинку прямо на Edge Function за доли секунды), позволяя каждой статье иметь красивую OG:Image обложку с живым текстом."*

### ❓ 18. Хуки работы с URL (`next/navigation`): Куда пропал `router.query`?
*   **Официальный ответ:**
    > В Next.js 13+ (App Router) больше нет толстого объекта `router` из `next/router`. Роутер разделили: `useRouter` (только для методов `push`, `replace`), `usePathname` (читать путь) и `useSearchParams` (читать Query/GET параметры).
*   **🗣️ Pitch (Senior):**
    > *"Разделение роута — это гениальный архитектурный ход со стороны команды Vercel ради гранулярных ререндеров. В старом Next.js 12, если вы читали `router.query` внутри тяжелого Layout-а, то изменение ЛЮБОГО query-параметра заставляло жестко перерендериваться всё дерево. В App Router, используя хук `useSearchParams` только глубоко внутри микро-компонента `<Pagination />`, React перерендерит ровно этот микро-дробный кусок UI, не трогая остальную страницу. Это радикально спасает CPU девайса на сложных дашбордах."*

### ❓ 19. Динамические(Dynamic) vs Статические(Static) Route Segments: Как конфигурировать?
*   **Официальный ответ:**
    > Если страница читает `cookies()`, `headers()` или содержит отключенный кеш `fetch(... { cache: 'no-store' })` — она становится динамичной(SSR) во время сборки. 
*   **🗣️ Pitch (Senior):**
    > *"На масштабах Enterprise иногда нужно 'насильно' сказать компилятору поведение. Я использую директивы `export const dynamic = 'force-dynamic'` (чтобы жестко запретить кэш и принудить SSR, например, на странице Оплаты) или `force-static` (игнорируя динамические вызовы, отдавая SSG). Но самое важное: функция `generateStaticParams()` (замена getStaticPaths из Next12). Если у меня в базе 10 000 статей `/[id]`, с помощью этой функции на этапе CI/CD Pipeline я генерирую топ-100 самых популярных статей в статику (Static HTML Cache), оставляя остальные 9900 генерироваться на лету (On-Demand). Это спасает CI-билд от зависания на 2 часа."*

### ❓ 20. Управление состоянием сложных форм в Next.js 14/15: Хук `useActionState` (ex `useFormState`)
*   **Официальный ответ:**
    > `useActionState(action, initialState)` (в React 19) позволяет управлять состоянием ошибок и UI-ответов от бекендного Server Action (например, вернуть на клиент `error: 'Пароль слишком слабый'`). Идеально стыкуется с `useFormStatus()`.
*   **🗣️ Pitch (Senior):**
    > *"Главное преимущество нативных хуков (Progressive Enhancement) — формы будут работать, даже если у юзера выключен JS в Safari (экзотика), или JS бандл завис при скачивании из метрического метро. Но в реальных банковских энтерпрайз интерфейсах DTO-json на 50 полей (Ипотека) невозможно отправить стандартным `<form target>`. Мой архитектурный компромисс: я использую `react-hook-form` в связке с Zod-резолвером-схемы на клиенте для мгновенной UX валидации, а в `onSubmit` передаю типизированный JSON payload `await myServerAction(data)`. Нативный `useFormStatus` я оставляю только для простых 'Кнопок Лайков'."*

## 🟢 РАЗДЕЛ 5: Advanced Engineering (DX, Sentry, Telemetry)

### ❓ 21. Serialization Boundary: Ограничения при передаче Props с Сервера на Клиент
*   **Официальный ответ:**
    > Server Components могут передавать параметры в Client Components (`<ClientUI data={data} />`), но эти данные должны быть "сериализуемыми" (Serializable). Нельзя передавать JS Функции (`onClick={() => {}}`), Классы или объекты Date/Map/Set.
*   **🗣️ Pitch (Senior):**
    > *"Грань сериализации — это физическая сеть (Network Bytes). RSC Payload передается как JSON-подобный стрим. Если вы попытаетесь передать инстанс класса `new Date()` с сервера в Client-компонент, Next.js упадет с ошибкой в консоли. Моя архитектурная задача на слое Data Access Layer (DAL) состоит в том, чтобы СТРУКТУРИРОВАТЬ данные: превращать даты в ISO-строки (`.toISOString()`), мапить классы баз данных (Prisma Model/TypeORM) в Plain JavaScript Objects (POJO). В Next 14 появился Server Actions — единственный способ 'передать функцию из серверного в клиентский компонент' (т.к. Next.js заворачивает ее в RPC-обработчик)."*

### ❓ 22. Интернационализация (i18n) в App Router: Проблема Layout Shared UI
*   **Официальный ответ:**
    > Встроенного i18n-роутинга больше нет (как в Pages router). Мы создаем динамический роут на самом верху `app/[locale]/page.tsx`, и читаем язык из URL параметра.
*   **🗣️ Pitch (Senior):**
    > *"В App Router i18n реализован через Middleware (который матчит URL `/ru/about` и рерайтит, если префикса нет). Архитектурная боль: как перевести SEO-токены (Metadata Title) на сервере, ведь мы не можем использовать `useTranslation` (он использует Client Context). Мой подход (на базе `next-intl`): я вытягиваю переводы словарей асинхронно `const t = await getTranslations({ locale, namespace: 'seo' })` прямо внутри серверного `generateMetadata` или `Server Component`. Это позволяет делать 100% серверный, SEO-идеальный Multi-Language проект, не отгружая в клиентский браузер ни байта словарей `en.json/es.json`."*

### ❓ 23. `instrumentation.ts` (Инструментация): Как мониторить сервер в Next.js?
*   **Официальный ответ:**
    > Специальный файл `instrumentation.ts` выполняется ровно один раз во время физического старта Node.js (или Edge) сервера Next.js.
*   **🗣️ Pitch (Senior):**
    > *"Пытаться инициализировать Sentry или Datadog (APM трекинг) просто в корне `layout.tsx` — ошибка, так как код выполнится N раз или раздует Webpack-бандл. Файл `instrumentation.js` — это архитектурная точка входа для бэкенд телеметрии (OpenTelemetry). В момент загрузки(бутстрапа) процесса Node.js, я стартую коннекшены с пулом базы данных и инициализирую Sentry Node Client. Это единственный гарантированный способ 'наблюдать' за Server Actions и Route Handlers без сайд-эффектов, отлавливая утечки памяти бэкенда Next.js."*

### ❓ 24. Fine-grained Caching `unstable_cache` и `unstable_noStore`
*   **Официальный ответ:**
    > *   `unstable_cache` позволяет закешировать ресурсоемкую асинхронную функцию (например парсинг Markdown), даже если она не делает HTTP `fetch` вызовов.
    > *   `unstable_noStore` — директива, которая жестко запрещает Next.js кешировать весь компонент.
*   **🗣️ Pitch (Senior):**
    > *"Иногда узкое место бэкенда это даже не `fetch(SQL)`, а тяжелый вычислительный алгоритм (например, расчет кредитного скоринг-плана на Node.js CPU). Нативный Web Fetch-кэш нам не поможет (вычислитель выполняется локально). В этом случае я оборачиваю 'чистую' JS-функцию математики в `const calculate = unstable_cache(heavyMath, ['credit-calc'])`. Next.js сам сложит возвращаемый JSON-результат функции в глобальный кэш (Redis/FileSystem). При следующем заходе мы не потратим ни миллисекунды CPU сервера. `noStore` применяю в Auth-хуках, чтобы СТРОГО защитить компонент сессии от кэша."*

### ❓ 25. Route Handlers: Как вернуть Streaming (потоковый) ответ и Webhooks?
*   **Официальный ответ:**
    > `app/api/route.ts` возвращает объект `Response`. Его можно связать с объектом `ReadableStream`, чтобы отдавать данные клиенту частями (например, при генерации нейросетей).
*   **🗣️ Pitch (Senior):**
    > *"Для банковских Webhooks от платежных сервисов я использую классический `route.ts (POST)`. Обязательно проверяю криптографическую подпись (HMAC / Stripe Signature) тела запроса через Node.js Crypto. Но главная сила Route Handlers в 2024 году — это AI Integration Streaming. Когда я стучусь к OpenAI API, я не хочу заставлять юзера смотреть в 'белую страницу' 10 секунд ожидания ответа. Внутри route.ts я инициализирую `ReadableStream`, пробрасываю `reader` из OpenAI, и мой Next.js бэкенд 'проксирует' печатающиеся куски букв к браузеру юзера в реальном времени (Transfer-Encoding: chunked), удерживая TTFB(метрику) на низком уровне <300мс."*

## 🟢 РАЗДЕЛ 6: Next.js в Enterprise (Microfrontends & Scaling)

### ❓ 26. Multi-Zone Next.js: Архитектура микрофронтендов (Base Path Routing)
*   **Официальный ответ:**
    > Zones — это подход, при котором одно доменное имя (например, `bank.com`) обслуживается несколькими РАЗНЫМИ независимыми Next.js приложениями (за них отвечают разные команды разработчиков).
*   **🗣️ Pitch (Senior):**
    > *"Enterprise-монолит Next.js со временем становится неподъемным (билд CI/CD длится 40 минут). Я переводил команду на Архитектуру Multi-Zone (аналог Module Federation, но для HTTP серверов Node.js). Приложение А (Маркетинг) живет на `bank.com/`. Приложение Б (Дашборд юзера) имеет флаг `basePath: '/app'` в `next.config.js` и крутится совершенно на другом Vercel-репозитории. Перед ними стоит Edge Router / Cloudflare, который по URL Request-path рерайтит трафик на нужный деплой. Умные линки (через хэштеги или кастомные Proxy) позволяют навигироваться сквозь зоны. Команды могут деплоить А и Б в любой день недели независимо, спасая TTM (Time to Market)."*

### ❓ 27. Error Boundaries: Вложенные `error.js` vs Глобальный `global-error.js`
*   **Официальный ответ:**
    > Разница заключается в глубине перехвата. Вложенный `error.js` оборачивает отдельный виджет, изолируя ошибку. `global-error.js` оборачивает Root Layout (самый верх) для фатальных сбоев.
*   **🗣️ Pitch (Senior):**
    > *"Идеальная Next.js Архитектура Resilience (Отказоустойчивости): на каждый логический блок (Сайдбар, Чат) есть дочерний `error.tsx`. Если БД падает с Timeout 504 при запросе чата, упадет (отрендерится красным градиентом) только карточка чата к кнопкой '🔁 Try Again' (костыль вызывающий функцию `reset()`). Остальная часть экрана(Навигация) продолжит работать(Zero-downtime UX). Но если падает сам корневой Root Layout (например сломалась библиотека подгрузки CSS/i18n), то `error.js` нас не спасет (т.к. он вложен 'ВНУТРЬ' body). Исключительно для крашей всего HTML мы создаем `global-error.tsx`, который рестартит свой собственный `<html><body>`, изящно оповещая юзера об ошибке 500."*

### ❓ 28. Vercel KV / Redis: Архитектура Rate Limiting (Ограничения запросов) Server Actions
*   **Официальный ответ:**
    > Rate Limiter защищает публичный API от брутфорса (подбор паролей) и DDoS атак, считая кол-во входов по IP адресу за окно в минуту.
*   **🗣️ Pitch (Senior):**
    > *"Server Action авторизации по умолчанию открыт всему интернету (публичный POST URL). Любой бот-нет может стучаться 1000 раз в секунду в форму Логина. Я проектирую Rate Limiter Архитектуру строго на уровне Edge Middleware (или внутри самого Action через легчайший Upstash Redis Vercel KV Store). Читаем IP юзера из запроса: `req.ip || headers()`. Инкремент Redis-счетчика `INCR login_attempts:ip`. Если счетчик больше 5 — я не пускаю HTTP-хит даже до тяжелого CPU-парсинга SQL базы данных, а мгновенно отдаю статус `429 Too Many Requests`. Это спасает Node.js сервера(Connection Pool БД) от смерти под жесткой DDoS атакой."*

### ❓ 29. Context API: Проблема React Context в архитектуре (App Router RSC)
*   **Официальный ответ:**
    > React Context (`createContext`) физически НЕ работает в React Server Components, поскольку серверные компоненты (RSC) не поддерживают рантайм-стейт (Hooks).
*   **🗣️ Pitch (Senior):**
    > *"В старом SPA-React мы любили заворачивать всё приложение в огромный `<ThemeProvider><UserProvider> <App/>`. В App Router делать огромные 'Client' провайдеры в корне RootLayout — это опасная архитектурная пуля в ногу. Это заставит ВСЕ дочерние компоненты стать клиентскими (Deopt Boundary), уничтожив концепцию и перфоманс RSC (они все попадут в бандл). Я применяю паттерн **Leaf Nodes** (Клиентизация 'листьев' дерева). Context-Стейт я создаю локально (изолированно внутри папки компонента `/features/Theme.tsx`). Сервер отдает 95% RSC-дерева, и если только глубоко слева лежит `Dropdown`, я оборачиваю только ЕГО в локальный Context-Провайдер. Контекст пускается по капиллярам, а не по главным артериям."*

### ❓ 30. Что такое "Taint" (Отравление данных) и зачем пакет `server-only` в Next.js?
*   **Официальный ответ:**
    > Пакет `server-only` заставляет Webpack компилятор выдавать Build-time ошибку сборки, если вы случайно импортировали 'приватный' серверный модуль (подключение к БД ORM) внутрь Client Component (файла с директивой `"use client"`).
*   **🗣️ Pitch (Senior):**
    > *"Security Data Leak (Утечки БД API-ключей и паролей юзеров) — самая страшная системная угроза RSC (App Router), потому что Backend и Frontend теперь лежат бок о бок в одних папках (Colocation). Джун может 'неудачно' импортировать утилиту, или напрямую в проп `<ClientComponent user={rootUser} />` бросить объект целиком(с хэшом пароля). Это будет выставлено наружу в HTML-сорс кода браузера навсегда. Моя железобетонная защита: 'Data Access Layer (DAL)'. Файлы ORM БД я жестко помечаю библиотекой `import "server-only"`. Любой Data-объект, текущий из DAL в UI, я пропускаю новым секьюрити-API React'а: `taintObjectReference("Ошибка: нельзя ликать юзера", data)`. Компилятор упадет с красной ошибкой в пайплайне CI/CD, если отравленный хэш(объект) попытается пересечь сетевую границу и уйти в браузер."*

## 🟢 РАЗДЕЛ 6: Next.js Ecosystem Deep Dive

### ❓ 31. Иерархия Кэширования в Next.js (4 Уровня Cache)
*   **Официальный ответ:**
    > 1) **Request Memoization** (запрос склеивается на сервере), 2) **Data Cache** (результат fetch() между серверами), 3) **Full Route Cache** (SSG статика), 4) **Router Cache** (клиентский кэш на 30/300 секунд в браузере при навигации).
*   **🗣️ Pitch (Senior):**
    > *"Джуны теряются в слоях кэширования App Router. Архитектору важно понимать гранулярность. Если мне нужно инвалидировать стейт корзины юзера, я использую теги для 'Data Cache'. Но есть великая проблема 'Router Cache' (в браузере юзера). Если клиент переходит со страницы товара `A` на главную `Home` и обратно на `A`, он мгновенно получит товар `A` (Stale старье из памяти, минуя сервер). Чтобы сбросить этот хардкорный бразуерный Routing-cache при мутациях (добавил в корзину), я гарантированно выполняю `revalidateTag` на сервере (запуская Server Action), что шлет скрытый Signal (RSC Payload Header) в браузер: 'Очисти свой Router Cache для этого URL'."*

### ❓ 32. Optimistic UI (Оптимистичные обновления) с хуком `useOptimistic` (React 19 / Next 14)
*   **Официальный ответ:**
    > `useOptimistic` позволяет мгновенно (синхронно) обновить UI на клиенте ДО ТОГО, как тяжелый Server Action отправит POST-запрос и вернет ответ от базы данных в США.
*   **🗣️ Pitch (Senior):**
    > *"Банковский 'Like' (добавить транзакцию в Избранное) не имеет права 'думать' полсекунды. Это убивает Mobile UX. Я строю Архитектуру Оптимистичного UI. При клике на звездочку, `startTransition(()=> addOptimistic(id))` мгновенно закрашивает звезду желтым. Параллельно (в фоне) летит Next.js Server Action запрос с Мутацией. Если сервер (Action) возвращает `error` (юзер оффлайн) — React автоматически (без моего кода Rollback-а) откатит звезду обратно в 'Серую' и покажет Toast. Идеальный, 'бесшовно-лживый' контентный опыт."*

### ❓ 33. Интеграция NextAuth.js (Auth.js) в Edge Middleware: JWT vs Database Sessions
*   **Официальный ответ:**
    > Auth.js поддерживает 2 стратегии: `JWT` (Токен в куки) и `Database` (ID Сессии в куки, а данные в БД). Middleware Edge не умеет напрямую читать Database, поэтому там используется JWT.
*   **🗣️ Pitch (Senior):**
    > *"Ахиллесова пята Edge Middleware — отсутствие TCP-драйверов к базам (например PostgreSQL). Если вы используете 'Database Session Strategy', Middleware не сможет проверить, залогинен юзер или нет (он рухнет с ошибкой `module pg not found`). Я строю Гибридную Auth-архитектуру: В куки (Token) лежит миниатюрный JWT с ID юзера на 1 КБ (верифицируется за 1мс на Cloudflare Middleware для редиректа гостей на /login). Но внутри самих защищенных серверных роутов `layout.tsx` (Node.js) я могу позволить себе тяжелую `await getSession()` из базы SQL, чтобы вытянуть массив его 15 ролей и аватарку для рендера."*

### ❓ 34. Разница `fetch()` Cache в Next 13, 14 и 15 версии
*   **Официальный ответ:**
    > В Next.js 13 и 14 нативный `fetch(url)` по умолчанию имел поведение `cache: 'force-cache'` (SSG на всегда). В Next.js 15 это поведение отменили, и дефолтом стал `cache: 'no-store'` (как в старом добром React).
*   **🗣️ Pitch (Senior):**
    > *"Агрессивное кэширование Vercel (Next 13-14) создало огромное количество багов на проде ('данные не обновляются'). Архитектурно, начиная с Next.js 15, я больше не боюсь непредсказуемой 'заморозки' fetch. По дефолту он ведет себя честно и динамично на каждый реквест (Dynamic SSR). Если я хочу закэшировать тяжелый API Каталога, я ЭКСПЛИЦИТНО добавляю `{ cache: "force-cache", next: { tags: ["catalog"] } }`. Explicit is better than implicit (Явное лучше неявного). Это золотое правило системного дизайна."*

### ❓ 35. Hydration Mismatches: Откуда берутся ошибки HTML-дерева в Next.js? 
*   **Официальный ответ:**
    > Ошибка 'Text content did not match. Server: "A" Client: "B"'. Происходит, когда HTML строка, отрендеренная Node.js сервером, не совпадает с тем, что пытается отрендерить React на клиенте при старте.
*   **🗣️ Pitch (Senior):**
    > *"Главный спонсор Hydration багов — клиентские даты `new Date().toLocaleString('us-EN')` (На сервере Vercel часовой пояс лондонский UTC 0, а у юзера московский GMT+3). Второй спонсор — расширения Chrome (например Grammarly), которые вставляют свои `<div id="grammarly">` в наш HTML еще ДО hydration-цикла. Я лечу это архитектурно: 1) Для кастомных дат/рандома я использую флаг `<div suppressHydrationWarning>` или рендерю плейсхолдер до первого `useEffect`. 2) Я жестко запрещаю использовать window.innerWidth для условного рендеринга (Mobile vs Desktop) в корне. Это сломает гидратацию. Вместо этого — CSS Media Queries (is-hidden-mobile)!"*

### ❓ 36. Продвинутый префетчинг: Как работает `<Link prefetch>` под капотом App Router?
*   **Официальный ответ:**
    > Когда `next/link` попадает в Viewport (на экран юзера), Next.js автоматически в фоне скачивает RSC-Payload будущей страницы.
*   **🗣️ Pitch (Senior):**
    > *"Prefetching — это иллюзия мгновенности интернета в Next. НО! Он префетчит только статические куски маршрута. Если страница `/[id]` динамическая, Next.js 'умно' префетчит только дерево Layout.tsx, вплоть до границы самого первого `<Suspense>`. Чтобы не спамить (DDoS) наш сервер микро-запросами от юзеров, быстро скроллящих ленту, я отключаю префетч для тяжелых профилей: `<Link prefetch={false}>`. Даже с `false`, при физическом `hover` (наведении мышки на ссылку) Next.js агрессивно скачает этот роут. Это спасает 60% холостых сетевых API вызовов на проекте."*

### ❓ 37. Docker Standalone Mode: Деплой Next.js On-Premise (Вне Vercel)
*   **Официальный ответ:**
    > Vercel идеален, но дорог. Для деплоя в AWS/свой Kubernetes мы используем Docker с `output: 'standalone'` в `next.config.js`. Компилятор соберет минимальный Node.js сервер без `node_modules`.
*   **🗣️ Pitch (Senior):**
    > *"Деплой Enterprise-банка за 10млн долларов на публичный Vercel невозможен по Policy Seguridad (Безопасности). В `next.config.ts` мы включаем режим `standalone`. Next.js анализирует AST дерево импортов, и создает микро-папку `.next/standalone`, копируя туда ТОЛЬКО нужные файлы из `node_modules/` (размер Node-сервера падает с гигантских 800MB до 50MB). Далее мы заворачиваем это в Alpine Docker Image. Единственный нюанс Out-of-Vercel Architecture: мы теряем Edge Network и Next.js Image Optimization API (поэтому библиотеку sharp внутри докера мы ставим вручную, а кэш картинок маунтим на S3-Volume)."*

### ❓ 38. `draftMode()`: Режим предварительного просмотра (CMS) в App Router
*   **Официальный ответ:**
    > Позволяет CMS (Sanity, Strapi, WordPress) показывать разработчику или редактору изменения на сайте В РЕАЛЬНОМ ВРЕМЕНИ, обходя и отключая жесткий SSG/Edge кэш Next.js.
*   **🗣️ Pitch (Senior):**
    > *"Контент-редакторы банка ненавидят ждать CI-Билда 15 минут, чтобы увидеть опечатку в статье. Я интегрирую `draftMode`. Редактор жмет кнопку 'Preview' в CMS. Имея эту куку, КАЖДАЯ страница (перехватываемая флагом `const { isEnabled } = draftMode()`) мгновенно игнорирует любой `Data Cache` и переходит в Dynamic SSR O(1). ### ❓ 39. Третий код: Аналитика и Оптимизация `next/script`
*   **Официальный ответ:**
    > Подключение Я.Метрики, GTM, Pixel. Модуль гарантирует, что сторонние жирные JS файлы не будут блокировать парсинг HTML (убивать FCP страницы). Стратегии: `beforeInteractive`, `afterInteractive`, `lazyOnload`.
*   **🗣️ Pitch (Senior):**
    > *"Сторонние трекеры (Yandex.Metrika, Chat Widgets) на 1МБ кода убивают перфоманс любого крутого банковского SPA. Я жестко ограничиваю их загрузку через стратегию `strategy="lazyOnload"`. Next.js физически 'заморозит' (отложит) скачивание этих скриптов в Network панеле ДО ТОГО МОМЕНТА, пока браузер полностью не отрендерит нашу реальную страницу и не перейдет в состояние `Idle` (свободный поток). Это 'читы' для метрики Lighthouse TTI (Time to Interactive). Баннеры аналитики не должны конкурировать за CPU и RAM с рендерингом React/CSS!"*

### ❓ 40. Multi-tenant Architecture (Multi-domain & Subdomains) поверх Next.js
*   **Официальный ответ:**
    > Мульти-аренда: один репозиторий обслуживает 100 разных доменов (`apple.com` и `store.apple.com`), подставляя разный контент и брендинг, но используя одну кодовую базу компонентов.
*   **🗣️ Pitch (Senior):**
    > *"Архитектура SaaS платформ (напр. Shopify). У нас 1 Next.js на Node.js. Клиент регистрируется, и ему генерируется поддомен `my-shop.bank.com`. Как это разрулить в 1 кодовой базе? Магия Cloudflare Middleware: он ловит запрос `my-shop.bank.com/catalog`. Middleware извлекает стрингу `'my-shop'` из Host-хедера, и делает внутренний(скрытый) Next.js-rewrite на URL: `/catalog?tenant=my-shop`. Внутри `page.tsx` я читаю `tenant` из SearchParams, делаю `db.getShop('my-shop')` и отдаю персонализированный CSS-стиль и Базу Данных товаров! При этом в адресной строке у браузера остается красивый `my-shop.bank.com`. Это высшая форма SaaS System Design."*

## 🟢 РАЗДЕЛ 7: Advanced Deep Dive. Security & Observability

### ❓ 41. RSC Serialization Boundaries: Как прокинуть Promise с Сервера на Клиент?
*   **Официальный ответ:**
    > Сервер может вернуть `Promise` как проп в клиентский компонент. Клиент может 'распаковать' ответ с помощью нового React 19 хука `use(promise)`.
*   **🗣️ Pitch (Senior):**
    > *"Это революция в стриминге(Streaming). Мой Server Component `Dashboard` делает `const statsPromise = fetchStats()`. Я **не делаю** `await`, а прокидываю сырой Promise вниз: `<ClientWidget stats={statsPromise} />`. Внутри Клиентского графика `ClientWidget` я вызываю `<Suspense fallback={..}> <Stats data={use(statsPromise)} /> </Suspense>`. Next.js в этот момент стримингом (по сети) льет резолв промиса прямо в бразуер! Клиент 'размораживает' UI в момент 'await' серверного вычисления. Тотальное размытие границ между Бэкендом и Фронтом (Network-agnostic code)."*

### ❓ 42. Server Actions: Как работает CSRF-Protection под капотом (Next 14)?
*   **Официальный ответ:**
    > Если Server Action физически это POST эндпоинт, почему хакеру не отправить CSRF-линк вашей бабушке `actionForm.submit()`? В Next 14 защита автоматизирована ядром.
*   **🗣️ Pitch (Senior):**
    > *"Раньше мы генерировали CSRF Tokens и клали их в скрытый `<input />`. Next.js Team гениально упростили безопасность. При вызове любого Server Action, Next.js Server автоматически проверяет HTTP заголовки `Origin` и `Host` с браузера. Если запрос прилетел с портала `evil-hacker.com`, а ожидался `my-bank.ru`, Action умрет с Security Exception (Host mismatch error) до того, как код прочтет 1 строчку базы данных. Никаких токенов больше не нужно. Но для кастомных REST API Route (`route.ts`) вы обязаны внедрять CORS/Origin проверку вручную."*

### ❓ 43. Sentry & Observability Server Components: Как отлавливать 'Тихие Утечки памяти' (OOM)?
*   **Официальный ответ:**
    > Установить `@sentry/nextjs`. Использовать Vercel Web Vitals и логировать Node.js Environment.
*   **🗣️ Pitch (Senior):**
    > *"Node.js сервер (пока он рендерит Server Components) подвержен классическим бэкенд утечкам памяти (Memory Leaks, OOM - Out of Memory). Sentry из коробки отлично трекает клиентские браузерные Runtime-error (клик в null), но на бэке (Server Actions) он может не поймать CPU Spike (100% зависание). На Enterprise уровне (Сбер) я оборачиваю весь сервер через OpenTelemetry Client (`instrumentation.js`). Все SQL-мутации я трекаю как Sentry Transaction: `Sentry.startSpan({ name: 'UpdateBalance' })`. Это позволяет найти Bottleneck — например, что `await db.table.count()` в Server Action занимает 8 секунд, вешая Node-сервер целиком!"*

### ❓ 44. Почему `localStorage` или `window` иногда ломает билд (Next.js Hydration Error)?
*   **Официальный ответ:**
    > Файл `"use client"` все равно сначала отрендерится на сервере (SSR) для формирования SEO-разметки HTML, где объекта `window` или `localStorage` физически не существует в Node.js.
*   **🗣️ Pitch (Senior):**
    > *"Классика. Джун пишет `const theme = localStorage.getItem('theme')` на корневом уровне Client-компонента (или в кастомном хуке `window.innerWidth`). Билд Next.js ломается ('Window is not defined'). Архитектурно мы решаем это 3-мя способами: 1. Безопасность через React Lifecycle (чтение localStorage ТОЛЬКО внутри <body> `useEffect`, который срабатывает исключительно в браузере). 2. Использование безопасной обертки `typeof window !== 'undefined'`. 3. Для тяжелых либ (canvas редактор), которые даже не имеют SSR имплементации, я жестко блокирую их рендер на сервере через `const DynamicEditor = dynamic(()=>import('./Map'), { ssr: false })`."*

### ❓ 45. Next.js Routing: Soft Navigation vs Hard Navigation (Почему страница "не моргает" белым экраном?)
*   **Официальный ответ:**
    > Компонент `<Link>` и метод `router.push` используют 'Soft Navigation' — они делают фоновый XHR (Fetch) за RSC-Payload-ом новой страницы и вставляют DOM, не перезагружая тяжелый HTML-каркас и CSS, сохраняя JS Memory (SPA-опыт).
*   **🗣️ Pitch (Senior):**
    > *"Но есть ловушка (Bust!). Если между страницей '/home' и '/catalog' нет общего 'Layout.tsx' узла в файловой системе (они лежат в разных Route Groups), то Soft Navigation 'сорвется'. Next.js попытается склеить 2 абсолютно разных DOM-каркаса, поймет, что это тяжело, и сделает Hard Reload (Мигание белого экрана, полная перезагрузка CSS/JS ресурсов браузера). Я тщательно проектирую иерархию папок `app/` так, чтобы общие секции(Header/Menu) ВЕЗДЕ имели единый (Singleton) Shared `layout.tsx`. Это гарант молниеносной(SPA) клиентской навигации без потери стейта."*

### ❓ 46. Как `unstable_cache` обходит ограничения Fetch-Caching'a (Расчеты по CPU)?
*   **Официальный ответ:**
    > Нативный кэш Next.js (Cache-Control) работает только для сетевых URL запросов `fetch()`. Чтобы закэшировать тяжелую JS-математику (или вызовы ORM БД вроде `Prisma.user.findMany()`), нужен специальный модуль `unstable_cache`.
*   **🗣️ Pitch (Senior):**
    > *"В мире Prisma ORM (или Drizzle) вы не делаете HTTP `fetch`. Вы вызываете JS-Библиотеку. Значит, Next.js не сможет закэшировать результат БД, и будет насиловать базу на каждый заход юзера(SSR). Я решаю проблему кэша 'Инстансов кода'. Оборачиваем функцию БД в `const cachedGetUsers = unstable_cache(async () => db.users.find(), ['users-key'], { tags: ['auth-tag'] })`. Vercel/Next под капотом положит возвращенный массив-JSON юзеров в свой глобальный Redis-меморис (Edge Data Cache). Скорость вырастает в сотни раз (0 ms hit), а инвалидируется это так же элегантно через `revalidateTag('auth-tag')`."*

### ❓ 47. Секьюрити Риски (Security Disclosures): Проблема 'React Server Components Source Code Leak'
*   **Официальный ответ:**
    > При 'Soft Navigation' к странице с RSC, браузер получает `.rsc` Payload, который представляет собой сериализованный текст. Если в этом тексте есть пароли — они видны юзеру. Нельзя отдавать лишние поля из базы!
*   **🗣️ Pitch (Senior):**
    > *"Частая уязвимость джунов 'Mass Assignment'. Они делают `const data = await SQL('SELECT * FROM Users')` и передают ВЕСЬ массив в `props` Client-компонента `<UserTable data={data}>`. Они забывают, что колонка `user.hashed_password` и `user.private_email` попадет в RSC стрим (Network Payload) и любой школьник, открыв DevTools, прочитает их (без рендеринга на самом UI!). Золотое архитектурное правило Data-Access-Layer(DAL) в App Router — используем **Data Transfer Objects (DTO)**. Мы фильтруем `user` через `Zod / Object.pick` до 2-х полей `[{id, name}]` ЕЩЁ НА СТОРОНЕ Серверного компонента(Node.js), и только обрезку отправляем клиенту. Ни байта лишнего(секретного) не пересекает Network."*

### ❓ 48. Как Next.js 14 Partial Prerendering решает проблему медленного First Contentful Paint (FCP / LCP)?
*   **Официальный ответ:**
    > Рендерит всю страницу как SSG статику на этапе сборки (`next build`), но оставляет вырезанные «дырки» там, где есть `<Suspense>`.
*   **🗣️ Pitch (Senior):**
    > *"Архитектура до PPR: либо вся страница 100% статика (Быстро, но не подходит для E-com корзины), либо вся страница Dynamic SSR (Учитывает Корзину Куки, но тормозит TTFB на 400ms). PPR ломает парадигму. На этапе CI Билда, компилятор натыкается на `<Suspense>` с динамической функцией `Cart()`. Он не падает в SSR. Он генерирует HTML код Layout-а (Static Shell Header + Footer), а в месте корзины просто вшивает отрендеренный Fallback(Скелетончик). При запросе этого URL — HTML Каркас отдается из кэша мгновенно(10ms CDN Hit / LCP Top 1). И следом за ним (по открытому HTTP потоку) сервер 'догоняет' динамический HTML Корзины."*

### ❓ 49. Архитектура Edge API: Limits & Boundaries Vercel (Timeout Errors)
*   **Официальный ответ:**
    > У бессерверных инфраструктур (Serverless Functions) жесткие лимиты: 10/15 секунд на вычисление Node.js процесса у Free/Pro планов Vercel. Тяжелый парсинг видео 100% упадет по таймауту 504.
*   **🗣️ Pitch (Senior):**
    > *"Если Action (или API) делает генерацию большого PDF-отчета (30 секунд), и мы деплоим это на Vercel/Netlify Serverless — система 'убьет' процесс на 10-й секунде (Gateway Timeout 504), и клиент получит Crash. Архитектурное решение в Serverless/Edge эре — 'Asynchronous Jobs'. Если процесс долгий (LLM Generation, PDF), мы НЕ заставляем HTTP-Fetch висеть в Await-е. Я использую Cloud-сервисы вроде Upstash/Trigger.dev/Inngest. Action просто ставит Event в фоновую очередь (Message Queue) 'Сгенерируй PDF' (занимает 10ms) и отвечает клиенту 200 OK (`status: processing`). А клиент поверх React(SWR/Zustand) начинает поллинг(Polling) или слушает SSE Webhooks для получения финального результата."*

### ❓ 50. Что такое Server Actions "Closure" утечки (Lexical Scope Leaks)?
*   **Официальный ответ:**
    > Server Action, написанный прямо внутри серверного компонента `export default function Page(){ async function myAction() {'use server'} }`, "захватывает" локальные переменные (Closures) текущего дерева рендера. 
*   **🗣️ Pitch (Senior):**
    > *"Это крутой хак и одновременно ловушка. Если вы объявляете 'use server' функцию внутри замыкания RSC Компонента, Next.js под капотом магически сериализует все внешние переменные (из замыкания scope-а) и шифрует их в скрытое Hidden-input поле формы. Это может 'распухнуть' до 5 МБ (если в замыкании большой массив SQL) и убить POST-запрос! Моя жесткая корпоративная рекомендация: Выносите все Server Actions в абсолютно отдельные изолированные файлы `user.actions.ts`. Избегайте Inline(вложенных) Server Actions, чтобы не засорять клиентский Payload лишними инкапсулированными бинарниками и скрытыми ID базы данных."*

## 🟡 РАЗДЕЛ 7: База Next.js (Middle Level Core)

### ❓ 51. SSR vs SSG vs ISR. Что это за аббревиатуры?
*   **Официальный ответ:**
    > *   **SSR** (Server-Side Rendering): HTML генерируется каждый раз "заново" по запросу юзера. Нагружает Node.js, дает 100% свежие данные (`getServerSideProps`).
    > *   **SSG** (Static Site Generation): HTML генерируется только один раз при Билде (на CI/CD Pipeline). Отдает статику мгновенно через CDN (`getStaticProps`).
    > *   **ISR** (Incremental Static Regeneration): Смесь 👆. Страницы генерируются на этапе сборки как статика, но могут 'фоново' перегенерироваться при первом запросе юзера раз в сутки (`revalidate: 60`).
*   **🗣️ Pitch (Middle):**
    > *"Всю систему рендеринга определяет Контент. Блог-посты и Страницы Контаков — это железно SSG статика за 0 миллисекунд. Индивидуальная корзина магазина eCommerce — это SSR (либо вообще CSR в браузере), потому что цены и корзина меняются ежесекундно по юзерам. Карточки миллионов товаров мы делаем через ISR — кэшируем, но перебиваем раз в сутки без пере-сборки всего бандла (Incremental Build)."*

### ❓ 52. App Router (v13+): В чем разница файлов `page.tsx` и `layout.tsx`?
*   **Официальный ответ:**
    > `layout` — это "обертка" над `page`. Страница `page` переписывается при навигации на другой URL. А `layout` — логически не уничтожается и сохраняет внутренний Стейт Реакт-хуков.
*   **🗣️ Pitch (Middle):**
    > *"Если у вас есть общие Sidebar или Header навигации, и вы положите их в глобальный 'page.tsx', при клике на дочернюю ссылку Sidebar жестко перерендерится и моргнет. Умный Layout.tsx решает эту боль. Next.js не перерисовывает Лейаут-Обертки, он подменяет только `{children}` прокинутый внутрь. Стейты и плееры, вызванные в `layout`, будут жить бесконечно."*

### ❓ 53. Изображения: Почему используется `<Image />` компонент, а не `<img>`?
*   **Официальный ответ:**
    > Он предотвращает сдвиг верстки из-за картинок (CLS - Cumulative Layout Shift) и автоматически оптимизирует формат и вес файла налету. 
*   **🗣️ Pitch (Middle):**
    > *"Обычный `<img>` — враг Web Vitals. Он тянет PNG в 3 МБ на мобильный телефон. Компонент `<Image />` обращается к Image Optimization Node API. Next.js понимает, что с мобильного зашел Safari, конвертирует PNG в сверхлегкий WebP/AVIF в 30КБ и жестко фиксирует Ширину и Высоту квадрата до загрузки файла, чтобы сберечь верстку от прыжка FCP. Плюс он подключает нативное отложенное скачивание (Lazy Loading) для Картинок вне экрана."*

### ❓ 54. Что такое Гидратация (Hydration)?
*   **Официальный ответ:**
    > Процесс присоединения слушателей событий JavaScript (`onClick`) и Virtual DOM'а к статичной HTML разметке, которая приехала с сервера.
*   **🗣️ Pitch (Middle):**
    > *"Когда вы скачиваете SSR страницу, HTML парсится мгновенно, юзер видит интерфейс и кнопки. Но они еще 'мертвые'. Скачивается JS Bundle React'а. React пробегается во вторую фазу: сверяет текущий DOM со своей 'виртуальной' инструкцией, и 'оживляет' (Hydrate) кнопки, вешая на них Listeners. Ошибка Mismatch случается, когда HTML Сервиса (`new Date.now()`) и Код Клиента случайно не совпали лбами."*
