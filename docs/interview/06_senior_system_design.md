# 👑 Модуль 6: Senior System Design, Soft Skills & Diagnostics

На уровне **Senior/Lead** вопросы меняются. Техническая база (Fiber, JS) считается по умолчанию известной. Интервьюеры проверяют **мышление**, **способность решать проблемы** и **бизнес-подход**.

---

## 🏗️ 1. Архитектурный System Design (Проектирование)

### ❓ В: "Спроектируйте архитектуру крупного приложения (например, Trello или YouTube) с нуля"
*   **Официальный подход:**
    Интервьюер ждет системности. Не начинайте писать код. Идите по шагам:
    1.  **Сбор требований (Requirements):** Какая нагрузка? Онлайн-обновления? Мобильные клиенты?
    2.  **Выбор стейт-менеджера:** (например, для Trello нужен реалтайм — значит, WebSocket + клиентская нормализация типа Redux/Zustand для оффлайна).
    3.  **Сегментация кода (Микрофронтенды vs Монолит):** Для масштаба команд.
    4.  **Стратегия рендеринга:** SSR для SEO (Главная страница), CSR для Личного Кабинета.
*   **🗣️ Pitch:**
    > *"Первым делом я соберу функциональные требования и оценю лимиты. Если это оффлайн-приложение вроде Trello, я выберу IndexedDB/Zustand для кэширования локальных изменений с последующей синхронизацией по вещающим шинам (Websockets). Я разделю домены на Microfrontends (Module Federation), чтобы команды релизились независимо."*

## 🩺 2. Диагностика (Troubleshooting)

### ❓ В: "Приложение лагает на проде. Ваши действия по шагам?"
*   **Официальный подход:**
    Не гадать. Сначала метрики, потом локализация.
*   **🗣️ Pitch:**
    > *"1. **Метрики:** Смотрю Grafana/Sentry/Analytics, есть ли скачки ошибок.
    > 2. **Профайл:** В браузере открываю вкладку Performance и записываю лог лагов (ищу длинные Task-и > 50мс).
    > 3. **Memory:** Проверяю вкладку Memory на предмет утечек (Heap Snapshots).
    > 4. **Сеть:** Проверяю вкладку Network (размер JS бандлов, медленные API запросы).
    > Только после этого я локализую компонент, который вызывает каскадный рендеринг или дублирует запросы через батчинг (Batching/Throttle)."*

## 🤝 3. Техническое Лидерство (Soft Skills)

### ❓ В: "Как вы будете внедрять новую технологию (например, Tailwind или CSS-in-JS) в старый проект?"
*   **Официальный подход:**
    Бизнес не прощает рефакторинг ради рефакторинга. Нужна аргументация.
*   **🗣️ Pitch:**
    > *"Я никогда не стану переписывать всё сразу. Шаги такие:
    > 1. **Исследование:** Проведу бенчмарк (Benchmark). Составит ли технология выигрыш в скорости загрузки или скорости поставки фич (Time to market)?
    > 2. **Презентация:** Защищу идею перед командой и бизнесом (покажу профит в цифрах).
    > 3. **Пилотный проект:** Внедрю технологию на одной изолированной странице/модуле.
    > 4. **Правила (Linter/CI):** Создам документацию по соглашениям, чтобы код писался единообразно."*

---

### ❓ В: "Что делать с токсичным или слабым разработчиком в команде?"
*   **🗣️ Pitch:**
    > *"Сначала — приватный диалог (1-on-1). Без наездов, через «Я-сообщения» («Мне кажется, что кодревью затягивается»). Обозначить общую цель команды.
    > Если разработчик слабый: составить план развития (PDP), дать ментора, разбить задачи на более мелкие. Если токсичность вредит команде годами и диалоги не помогают — это вопрос смены команды."*

## 💸 4. Работа с Техническим Долгом (Tech Debt)

### ❓ В: "Как менеджер хочет фичу RIGHT NOW, но архитектура не позволяет без костылей?"
*   **🗣️ Pitch:**
    > *"Я объясню менеджеру риски. «Мы можем сделать костыль за 2 дня, но через месяц фича Х упадет, а поддержка вырастет в 3 раза».
    > **Компромисс:** Мы делаем заглушку сейчас, чтобы успеть в релиз, но я ОБЯЗАТЕЛЬНО завожу тикет в бэклог на техдолг и закладываю 10-20% спринта на его закрытие. Без договоренностей "временное" становится вечным."*

### 🔥 Вопросы-Ловушки (Сюрпризы на Senior):

1.  **"Расскажите про ваш самый большой факап в карьере"**
    *   *Интервьюер проверяет честность и способность делать выводы.*
    *   *Совет:* Расскажите про реальную ошибку (уронил прод, удалил базу), но сделайте упор на то, **как вы её спасли** и **какой регламент внедрили** (тесты, бэкапы), чтобы это никогда не повторилось.
2.  **"Оцените время выполнения задачи Х, о которой вы ничего не знаете"**
    *   *Совет:* Никогда не давайте точную оценку (например, "2 дня"). Скажите: *"Мне нужно 2 часа на декомпозицию и анализ кода, после этого я дам вилку (например, 3-5 дней) с учетом рисков."*
3.  **"Зачем вашему бизнесу юнит-тесты?"**
    *   *Совет:* Тесты нужны не для проверки работоспособности сейчас, а для **безопасного рефакторинга завтра**. Чтобы при добавлении новой фичи не отвалился старый модуль на проде. Это экономит миллионы на багах.
